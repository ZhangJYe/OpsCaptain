## 一、核心挑战：诊断报告怎么打分？

### RAG 评测 vs 多智能体评测

| | RAG 评测 | 多智能体评测 |
|---|---|---|
| **输出** | 排序后的文档 ID 列表 | 结构化诊断报告 |
| **正确答案** | 有 groundtruth（相关文档 ID） | 没有唯一正确答案 |
| **评测指标** | Recall@K, Hit@K | 需要多维度 |
| **自动化** | 纯计算，无需 LLM | 需要 LLM-as-Judge |

### 诊断报告的四维质量

```
一份好的诊断报告应该：
┌──────────────┐
│ 正确性  ←─── 结论对吗？
├──────────────┤
│ 完整性  ←─── 有没有漏掉关键信息？
├──────────────┤
│ 逻辑性  ←─── 推理过程通顺吗？
├──────────────┤
│ 可操作性 ←─── 给出的建议能执行吗？
└──────────────┘
```

## 二、总体方案：三层评测金字塔

```
         ┌─────────┐
         │ Layer 3 │  ← 行为指标（延迟、成本、路由准确率）
         │ 自动采集 │     每次请求自动记录，CI 持续监控
         ├─────────┤
         │ Layer 2 │  ← LLM-as-Judge（正确性/完整性/逻辑性/可操作性）

         ├─────────┤
         │ Layer 1 │  ← Golden Case 回归测试
         │ 每次提交 │     已知正确答案的 case，快速验证不退化
         └─────────┘
```

## 三、Layer 1：Golden Case 回归测试

### 3.1 设计思路

```go
// internal/ai/agent/eval/types.go (新增)
type DiagCase struct {
    ID              string   `json:"id"`
    Query           string   `json:"query"`
    ExpectedIntent  string   `json:"expected_intent"`   // 期望的意图分类
    ExpectedDomains []string `json:"expected_domains"`   // 期望调用的领域

}
```

```go
// internal/ai/agent/eval/diag_test.go (新增)
func TestDiagGoldenCases(t *testing.T) {
    cases := loadDiagCases(t, "testdata/diag_golden.jsonl")

    for _, tc := range cases {
        t.Run(tc.ID, func(t *testing.T) {
            // 1. 用真实 Runtime 执行
            rt := buildTestRuntime(t)
            result := runMultiAgent(t, rt, tc.Query)

            // 2. 验证意图分类
            intent := extractIntent(result)
            if intent != tc.ExpectedIntent {
                t.Errorf("意图: 期望 %s, 实际 %s", tc.ExpectedIntent, intent)
            }

            // 3. 验证领域覆盖
            domains := extractDomains(result)
            for _, expected := range tc.ExpectedDomains {
                if !contains(domains, expected) {
                    t.Errorf("领域遗漏: 期望包含 %s, 实际 %v", expected, domains)
                }
            }

            // 4. 验证输出包含关键信息
            for _, must := range tc.MustMention {
                if !strings.Contains(result.Summary, must) {
                    t.Errorf("关键信息遗漏: 期望包含 %q", must)
                }
            }

            // 5. 验证不包含不该有的内容
            for _, forbidden := range tc.MustNotMention {
                if strings.Contains(result.Summary, forbidden) {
                    t.Errorf("不应该包含: %q", forbidden)
                }
            }
        })
    }
}
```

## 四、Layer 2：LLM-as-Judge 质量评分

### 4.1 设计思路

Layer 1 只能测"确定性"的指标（路由对不对、关键词有没有）。但诊断报告的**质量**需要 LLM 来评判。

```
                    ┌─────────────┐
   Golden Case ────▶│ Multi-Agent │───▶ 诊断报告
                    │  (优化前)   │
                    └─────────────┘
                    ┌─────────────┐
   Golden Case ────▶│ Multi-Agent │───▶ 诊断报告
                    │  (优化后)   │
                    └─────────────┘
                           │
                           ▼
                    ┌─────────────┐
                    │ LLM Judge   │ ← 对两份报告打分
                    │ (DeepSeek)  │
                    └─────────────┘
                           │
                           ▼
               ┌─────────────────────┐
               │ 对比分数 → 提升了吗？ │
               └─────────────────────┘
```

### 4.2 评分维度与 Prompt

```
你是一个运维诊断质量评估专家。请对以下诊断报告从四个维度打分（1-5分）：

用户问题：{query}
诊断报告：
{report}

评分维度：
1. 正确性：诊断结论是否与证据一致？有没有明显错误？
   1分=结论与证据矛盾  3分=结论基本合理  5分=结论精确且有证据支撑

2. 完整性：是否覆盖了必要的排查维度？
   1分=遗漏关键信息  3分=覆盖主要维度  5分=全面且无遗漏

3. 逻辑性：推理过程是否清晰、连贯？
   1分=逻辑跳跃/矛盾  3分=基本连贯  5分=推理严密，因果明确

4. 可操作性：建议是否具体可执行？
   1分=没有建议或建议模糊  3分=有建议但不够具体  5分=建议明确、可立即执行

输出 JSON：
{
  "correctness": 4,
  "completeness": 3,
  "coherence": 4,
  "actionability": 5,
  "overall": 4,
  "comments": "一句话评价"
}
```

### 4.3 Judge 评测代码

```go
// internal/ai/agent/eval/judge.go (新增)
type DiagScores struct {
    Correctness   int    `json:"correctness"`
    Completeness  int    `json:"completeness"`
    Coherence     int    `json:"coherence"`
    Actionability int    `json:"actionability"`
    Overall       int    `json:"overall"`
    Comments      string `json:"comments"`
}

type JudgeResult struct {
    CaseID    string
    Query     string
    Baseline  DiagScores  // 优化前
    Candidate DiagScores  // 优化后
    Delta     DiagScores  // 差值
}

func JudgeDiagReport(ctx context.Context, query, report string) (DiagScores, error) {
    prompt := buildJudgePrompt(query, report)
    response := callLLM(ctx, prompt)
    var scores DiagScores
    json.Unmarshal([]byte(response), &scores)
    return scores, nil
}

// 批量对比
func CompareRuns(ctx context.Context, cases []DiagCase, 
    baselineRunner, candidateRunner Runner) ([]JudgeResult, error) {
    
    results := make([]JudgeResult, 0, len(cases))
    for _, tc := range cases {
        // 跑基线
        baselineReport := baselineRunner.Run(ctx, tc.Query)
        // 跑候选
        candidateReport := candidateRunner.Run(ctx, tc.Query)
        // LLM 打分
        baselineScores, _ := JudgeDiagReport(ctx, tc.Query, baselineReport)
        candidateScores, _ := JudgeDiagReport(ctx, tc.Query, candidateReport)
        
        results = append(results, JudgeResult{
            CaseID:    tc.ID,
            Query:     tc.Query,
            Baseline:  baselineScores,
            Candidate: candidateScores,
            Delta: DiagScores{
                Correctness:   candidateScores.Correctness - baselineScores.Correctness,
                Completeness:  candidateScores.Completeness - baselineScores.Completeness,
                Coherence:     candidateScores.Coherence - baselineScores.Coherence,
                Actionability: candidateScores.Actionability - baselineScores.Actionability,
                Overall:       candidateScores.Overall - baselineScores.Overall,
            },
        })
    }
    return results, nil
}
```

### 4.4 Judge 的可靠性

**问题：** LLM 打分也可能不准。怎么知道 Judge 自己靠不靠谱？

**对策：**

1. **人工标注校准集**：挑 10-20 个 case，人工打分，用于校准 Judge
2. **多 Judge 投票**：用两个不同模型打分（如 DeepSeek + Qwen），取平均
3. **Judge 一致性检查**：同一个报告打两次分，看一致性
4. **反馈循环**：如果发现 Judge 打分和人工判断明显不符，把 case 加入校准集

```
// 校准流程
func CalibrateJudge(t *testing.T) {
    // 1. 加载人工标注的校准集
    calibrationCases := loadCalibrationCases()
    
    // 2. Judge 打分
    var totalDiff float64
    for _, cc := range calibrationCases {
        scores, _ := JudgeDiagReport(ctx, cc.Query, cc.Report)
        diff := abs(scores.Overall - cc.HumanScore)
        totalDiff += diff
    }
    
    // 3. 计算误差
    avgDiff := totalDiff / float64(len(calibrationCases))
    t.Logf("Judge 平均误差: %.2f 分", avgDiff)
    
    // 4. 如果误差 > 1 分，需要调整 Judge prompt 或增加校准样本
    if avgDiff > 1.0 {
        t.Errorf("Judge 误差过大: %.2f", avgDiff)
    }
}
```

## 五、Layer 3：行为指标监测

这些指标自动采集，不需要人工标注。每次 CI 跑完自动对比。

### 5.1 延迟

| 指标 | 说明 | 采集方式 |
|------|------|---------|
| `p50_latency` | 中位延迟 | Runtime.Dispatch 中记录 |
| `p95_latency` | 95 分位延迟 | Prometheus histogram |
| `p99_latency` | 99 分位延迟 | Prometheus histogram |

**优化前后对比：**
```
优化前: p50=2.3s  p95=5.1s  p99=8.7s
优化后: p50=1.8s  p95=4.2s  p99=7.1s   ← 下降了 18-22%
```

### 5.2 路由准确率

| 指标 | 说明 | 采集方式 |
|------|------|---------|
| `triage_hit_rate` | Triage 命中关键词的比例 | 对比 LLM Triage 的结果 |
| `triage_fallback_rate` | Fallback 到默认规则的比例 | `intent == "alert_analysis" && 未命中任何规则` |
| `domain_precision` | 调用的 domain 是否必要 | LLM Judge 评估（见 Layer 2） |

**P0 Triage 优化的核心指标：**
```
优化前: fallback_rate = 40%（大量未命中）
优化后: fallback_rate = 10%（LLM 兜底）
```

### 5.3 LLM 调用成本

| 指标 | 说明 | 采集方式 |
|------|------|---------|
| `tokens_per_request` | 每次请求消耗的 token 数 | TokenAudit 已有 |
| `llm_calls_per_request` | 每次请求调用了多少次 LLM | 计数 |
| `cost_per_request` | 每次请求的费用 | token × 单价 |

**P1 LLM Reporter 的成本影响：**
```
优化前: 4 次 LLM 调用 (Triage×0 + 3个Specialist + Reporter×0) = 3 次
优化后: 5 次 LLM 调用 (Triage×1 + 3个Specialist + Reporter×1) = 5 次
→ 成本 +67%，但输出质量可能 +100%
→ 需要 trade-off 分析
```

### 5.4 降级率

| 指标 | 说明 |
|------|------|
| `degradation_rate` | 任意 Specialist 返回 Degraded 的请求比例 |
| `avg_confidence` | 最终报告的平均置信度 |

## 六、A/B 对比基础设施

### 6.1 核心思路

复用现有的 `runtime.Runtime`，跑两次，对比输出。

```go
// internal/ai/agent/eval/ab.go (新增)
type Runner interface {
    Run(ctx context.Context, query string) (*Result, error)
}

// 基线 Runner（当前代码）
type BaselineRunner struct {
    rt *runtime.Runtime
}

// 候选 Runner（优化后的代码）
type CandidateRunner struct {
    rt *runtime.Runtime
}

func RunAB(ctx context.Context, cases []DiagCase, 
    baseline, candidate Runner) (*ABReport, error) {
    
    results := make([]ABCaseResult, 0, len(cases))
    
    for _, tc := range cases {
        // 跑基线
        baseResult, baseErr := baseline.Run(ctx, tc.Query)
        baseLatency := baseResult.Latency
        
        // 跑候选
        candResult, candErr := candidate.Run(ctx, tc.Query)
        candLatency := candResult.Latency
        
        // LLM Judge 评分
        baseScores, _ := JudgeDiagReport(ctx, tc.Query, baseResult.Summary)
        candScores, _ := JudgeDiagReport(ctx, tc.Query, candResult.Summary)
        
        results = append(results, ABCaseResult{
            CaseID:      tc.ID,
            Query:       tc.Query,
            BaselineScore:  baseScores,
            CandidateScore: candScores,
            BaselineLatency: baseLatency,
            CandidateLatency: candLatency,
        })
    }
    
    return summarize(results), nil
}
```

## 七、各优化的具体评测方案

### 7.1 P0：Triage 升级评测

**核心问题：** 新的 Triage 真的比关键词匹配准吗？

**评测方法：**

```
1. 准备 50 个真实运维 query（从日志/告警中采集）
2. 人工标注正确答案（intent + domains）
3. 分别用关键词 Triage 和 LLM Triage 分类
4. 对比准确率
```

**指标：**

| 指标 | 关键词 Triage | LLM Triage | 目标 |
|------|-------------|-----------|------|
| Intent Accuracy | ? | ? | > 90% |
| Domain Precision | ? | ? | > 85% |
| Domain Recall | ? | ? | > 90% |
| Fallback Rate | ? | ? | < 10% |

**测试代码结构：**

```go
func TestTriageAccuracy(t *testing.T) {

        // 关键词 Triage
```

### 7.2 P1：Staged Execution 评测

**核心问题：** 串行化会让 Specialist 更聚焦吗？延迟增加了多少？

**评测方法：**

```
1. 用同一批 Golden Case
2. 并行模式跑一遍（基线）
3. Staged 模式跑一遍（候选）
4. 对比：Reporter 输出的质量 + 端到端延迟
```

**特殊指标：**

| 指标 | 说明 |
|------|------|
| `focus_hint_utilization` | 第二阶段 Specialist 是否真的用了 focus_hints |
| `result_relevance` | 第二阶段的结果是否更聚焦（LLM Judge 评估） |
| `stage2_speedup` | 第二阶段因为聚焦而加速了多少 |

### 7.3 P1：LLM Reporter 评测

**核心问题：** LLM Reporter 比模板拼接强在哪？

**评测方法：** 标准 A/B 对比（见第六节），重点关注：

- **可操作性** — 这是模板 Reporter 最弱的维度
- **矛盾检测** — 故意构造矛盾 case（Metrics 说正常，Logs 说异常）
- **置信度校准** — LLM Reporter 给的置信度合理吗？

**矛盾检测专项测试：**

```jsonl
{"id":"contra-001","query":"paymentservice 状态异常","metrics_result":"CPU 正常，内存正常","logs_result":"大量 500 错误，connection refused","knowledge_result":"无相关记录","expected":"应该检测到指标和日志的矛盾，标注需要人工介入"}
```

### 7.4 P2：Self-Reflect 评测

**核心问题：** 自省循环真的能改进不够好的回答吗？

**评测方法：**

```
1. 构造"信息不足"的 case（query 模糊、知识库无记录）
2. 无 Self-Reflect 跑一遍 → 记录质量和延迟
3. 有 Self-Reflect 跑一遍 → 记录质量、延迟、重试次数
4. 对比
```

**特殊指标：**

| 指标 | 说明 |
|------|------|
| `reflect_trigger_rate` | 触发自省的比例（不应 > 30%，否则说明初次质量太差） |
| `reflect_improvement_rate` | 自省后有改善的比例 |
| `reflect_avg_rounds` | 平均重试轮数（应接近 1） |
