# 多智能体优化基线评测记录

> 日期：2026-04-26
> 目的：在实施 `多智能体架构优化分析.md` 中的 P0/P1/P2 优化前，先建立可复现 baseline，避免后续只能凭主观感受判断“优化有没有效果”。

---

## 一、为什么先测基线

多智能体优化会同时影响路由准确率、执行成本、延迟和报告质量。如果不先测当前版本，就无法回答三个关键问题：

1. 优化前到底差在哪里？
2. 优化后是质量提升，还是只是成本和延迟增加？
3. 如果指标变差，具体是哪一层退化了？

本次先测 P0 Triage baseline，因为 Triage 是 Multi-Agent 的入口。入口路由不准时，下游 Specialist 和 Reporter 再强也会被错误任务分配拖累。

---

## 二、已落地的评测能力

新增多智能体评测入口：

```powershell
go run ./internal/ai/cmd/agent_eval_cmd -mode routing -runner triage -format markdown -name current-triage-baseline
```

当前支持两类模式：

| 模式 | 用途 | 默认是否依赖外部服务 |
|---|---|---|
| `routing` | 评测 intent / domains / fallback rate | 否 |
| `golden` | 评测完整输出是否满足 Golden Case 断言 | 取决于 runner |

当前支持两类 runner：

| Runner | 用途 |
|---|---|
| `triage` | 只测 Triage，快速、确定性、适合提交级 baseline |
| `multi_agent` | 跑完整 Supervisor 链路，需要 `agent_eval_integration` build tag，适合夜间或手动评测 |

评测数据：

```text
internal/ai/agent/eval/testdata/diag_golden.jsonl
```

## 三、当前 Triage Baseline

运行命令：

```powershell
go run ./internal/ai/cmd/agent_eval_cmd -mode routing -runner triage -format markdown -name current-triage-baseline
```

结果摘要：

| 指标 | 当前值 | 解释 |
|---|---:|---|
| Case 数 | 20 | Golden Case 路由样本 |
| Intent Accuracy | 0.40 | 只有 40% 的 query 被分到正确 intent |
| Domain Precision | 0.77 | 被调用的 domain 中有一定比例是多余的 |
| Domain Recall | 0.88 | 召回看起来较高，主要因为 fallback 默认全开三类 domain |
| Domain F1 | 0.77 | 综合 domain 质量一般 |
| Fallback Rate | 0.70 | 14/20 query 没命中规则，走默认 alert_analysis |

关键观察：

- 当前 Triage 对 “告警 / Prometheus / 日志 / SOP / MySQL” 这类显式关键词能命中。
- 对 “HPA 怎么配置”“错误码是什么意思”“服务降级和熔断”等知识问答，容易 fallback 到 `alert_analysis`。
- 对 “5xx 升高”“CrashLoopBackOff”“504”“connection refused”等故障语义，当前关键词覆盖不足，也会 fallback。
- Domain Recall 偏高并不代表路由好，因为 fallback 默认调用 `metrics + logs + knowledge`，属于“宁可全查”的粗放策略。

## 四、方案 Review 结论

`多智能体架构优化分析.md` 的主线是合理的，优先级建议保持：

1. P0 Hybrid Triage
2. P1 Staged Execution / LLM Reporter
3. P2 Self-Reflect
4. P2 Unified Path
5. P3 Contract Enforcement / Cost Router

但实施时需要加约束：

- Triage 优化前后必须用同一批 case A/B 对比。
- Triage 不应把 memory 拼进 routing goal，raw query 决定 routing，memory 只进入 specialist 上下文。
- Staged Execution 必须配置化：`parallel | staged`，不能直接替换并行模式。
- LLM Reporter 必须配置化：`template | llm | auto`，先用于复杂 query 和评测集。
- Self-Reflect 应排在 LLM Reporter 后面，否则缺少可靠判断基础。
- Contract Enforcement 先做 audit-only，再决定哪些规则强制降级。

## 五、P0 Hybrid Triage 的验收标准

第一阶段目标：

| 指标 | 当前 baseline | P0 最小目标 | 长期目标 |
|---|---:|---:|---:|
| Intent Accuracy | 0.40 | >= 0.80 | >= 0.90 |
| Domain Recall | 0.88 | 不低于 0.88 | >= 0.90 |
| Domain Precision | 0.77 | >= 0.85 | >= 0.90 |
| Fallback Rate | 0.70 | <= 0.30 | <= 0.10 |

实现策略：

```text
规则快速路径：
  命中高置信关键词 -> 直接返回

LLM 兜底：
  未命中规则 -> 轻量 LLM 分类
  超时或解析失败 -> 回退到现有默认规则

配置保护：
  multi_agent.triage_mode: rule | hybrid | llm
  multi_agent.triage_llm_timeout_ms: 1000
```

## 六、面试讲解版本

可以按 STAR 结构讲：

**Situation：**
项目已经有 Supervisor-Triage-Specialists-Reporter 多智能体链路，但 Triage 还是关键词规则。我们准备做 Hybrid Triage、Staged Execution、LLM Reporter 等优化。

**Task：**
我没有直接改架构，而是先补评测 baseline。因为多智能体优化不是单点功能，必须能量化回答“路由准不准、成本有没有增加、报告质量有没有提升”。

**Action：**
我设计了三层评测：Golden Case 回归、LLM-as-Judge 质量评分、行为指标监测。第一步先落地 P0 routing baseline，用 20 条 Golden Case 测 intent accuracy、domain precision/recall 和 fallback rate，并提供可重复运行的命令。

**Result：**
baseline 显示当前 Intent Accuracy 只有 0.40，Fallback Rate 达到 0.70。这个结果证明优化 Triage 是必要的，而且后续 Hybrid Triage 可以用同一批 case 做 A/B 对比，避免主观判断。

面试官追问时可以补充：

- Domain Recall 高不代表好，因为 fallback 会把三个 domain 全开，成本和噪音都会变高。
- 我把完整多智能体评测和 Triage 评测拆开，是为了让 CI 中的 baseline 不依赖 Prometheus、MCP、Milvus 或 LLM。
- 真实全链路评测保留为 integration runner，适合夜间任务或手动验收。

## 七、下一步

下一步实施 P0 Hybrid Triage：

1. 增加配置项：`triage_mode`、`triage_llm_timeout_ms`。
2. 保留规则快速路径。
3. 未命中规则时调用 LLM 输出结构化 intent/domains。
4. 解析失败或超时时回退现有规则。
5. 用本文 baseline 命令做 A/B 对比并更新结果。
