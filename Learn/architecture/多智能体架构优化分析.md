## 一、当前架构回顾

### 1.1 整体流程

```
用户输入 "checkoutservice CPU 告警"
        │
        ▼
ShouldUseMultiAgentForChat()  ← 关键词匹配，决定走哪条路
        │
   ┌────┴────┐
   │ 关键词命中  │  → Multi-Agent 路径
   │ "告警"等   │
   └─────────┘
        │
        ▼
   Supervisor.Handle()
        │
        ├─ 1️⃣ Dispatch(triage)     → 关键词匹配 → intent + domains
        │                              (纯规则，无 LLM)
        │
        ├─ 2️⃣ 并行 Dispatch 到各 specialist
        │    ├─ metrics  (goroutine)
        │    ├─ logs     (goroutine)
        │    └─ knowledge(goroutine)
        │    WaitGroup.Wait()
        │
        └─ 3️⃣ Dispatch(reporter)    → 模板拼接 → 输出报告
                                       (纯规则，无 LLM)
```

### 1.2 核心组件

| 组件 | 文件 | 行数 | 职责 |
|------|------|------|------|
| **Runtime** | `internal/ai/runtime/runtime.go` | 436 | 任务调度引擎（Registry + Ledger + Bus + Artifacts + Dispatch） |
| **Supervisor** | `internal/ai/agent/supervisor/supervisor.go` | 197 | 编排入口：Triage → Specialists → Reporter |
| **Triage** | `internal/ai/agent/triage/triage.go` | 108 | 意图分类 + 领域路由（关键词匹配） |
| **Specialists** | `skillspecialists/{metrics,logs,knowledge}/` | ~ | 指标 / 日志 / 知识库查询 |
| **Reporter** | `internal/ai/agent/reporter/reporter.go` | 418 | 结果聚合（chat/report 两种模式，纯模板拼接） |
| **Contracts** | `internal/ai/agent/contracts/contracts.go` | ~ | Agent 职责边界定义（Must/MustNot，仅元数据） |
| **路由决策** | `internal/ai/service/chat_multi_agent.go` | 181 | 关键词命中 → Multi-Agent，未命中 → ReAct Agent |

### 1.3 架构亮点

- **单体内 Runtime**：不引入分布式复杂度，优先验证业务模型
- **文件持久化**：Ledger + Artifact Store 用文件存储，零额外依赖
- **Event Bus**：任务事件可追溯，支持全链路追踪
- **超时保护**：每个 Agent Dispatch 30s 超时，可配置
- **OpenTelemetry**：每次 Dispatch 自动创建 Span
- **降级哲学**：Degraded 而非 Fatal，局部失败不阻塞整体
- **两条路径共存**：Single-Agent (ReAct) + Multi-Agent (Supervisor)，保留 legacy fallback

## 二、6 个核心瓶颈

### 瓶颈 1：Triage 是纯关键词匹配，没有语义理解

**代码证据：**

```go
// internal/ai/agent/triage/triage.go
var triageRules = []rule{
    {keywords: []string{"告警", "alert", "prometheus"}, ...},
    {keywords: []string{"文档", "知识库", "runbook", "sop"}, ...},
    {keywords: []string{"日志", "log"}, ...},
}

func matchesRule(query string, keywords []string) bool {
    for _, keyword := range keywords {
        if strings.Contains(query, strings.ToLower(keyword)) {
            return true
        }
    }
    return false
}
```

**问题：**

| 用户输入 | 匹配结果 | 实际意图 |
|---------|---------|---------|
| "服务响应变慢了" | 未命中 → fallback alert_analysis | 性能分析 |
| "用户反馈下单失败" | 未命中 → fallback alert_analysis | 故障排查 |
| "pod 一直重启" | 未命中 → fallback alert_analysis | 故障排查 |
| "HPA 怎么配置" | 未命中 → fallback alert_analysis | 知识问答 |

**后果：** 大量查询 fallback 到默认规则，三个 domain 全开，浪费资源且路由不准。

### 瓶颈 2：Specialist 之间完全隔离，没有信息共享

**代码证据：**

```go
// internal/ai/agent/supervisor/supervisor.go
for _, domain := range rawDomains {
    wg.Add(1)
    go func(name string) {
        defer wg.Done()
        childTask := protocol.NewChildTask(task, name, specialistQuery, ...)
        result, dispatchErr := rt.Dispatch(ctx, childTask)
        // ← 每个 specialist 不知道其他 specialist 发现了什么
        mu.Lock()
        childResults = append(childResults, result)
        mu.Unlock()
    }(agentName)
}
wg.Wait()
```

**问题：** 如果 Metrics 发现 "paymentservice CPU 100%"，Logs 仍在全局扫描，不知道应该聚焦 paymentservice。

**后果：** 检索效率低、噪音多、结果关联性差。

### 瓶颈 3：Reporter 是模板拼接，不是 LLM 合成

**代码证据：**

```go
// internal/ai/agent/reporter/reporter.go
func (a *Agent) Handle(ctx context.Context, task *protocol.TaskEnvelope) {
    // ...
    sections = append(sections, text.reportTitle)
    sections = append(sections, text.userGoalHeading+"\n"+query)
    sections = append(sections, text.intentHeading+"\n"+fallback(intent, "alert_analysis"))
    sections = append(sections, text.executionSummaryHeading)
    // 纯字符串拼接，无推理
    for _, result := range raw {
        sections = append(sections, fmt.Sprintf("### %s\n%s", label, result.Summary))
    }
    // ...
    return strings.Join(sections, "\n\n")
}
```

**问题：**
- 无法交叉验证：Metrics 说正常，Logs 说异常 → Reporter 不会检测矛盾
- 无法综合推理：三个结果指向同一根因 → Reporter 不会归纳
- 无法生成建议：没有 "基于以上分析，建议执行 X"

**后果：** 输出质量天花板低，本质是"信息罗列"而非"智能诊断"。

---

### 瓶颈 4：没有反馈循环（Self-Correction）

```
Supervisor → Triage → Specialists → Reporter → 结束
                                                ↑
                                        没有"不满意就重新来"
```

**问题：** 如果 Reporter 发现证据不足、结果矛盾，没有机制触发重新查询。

**对比：** Plan-Execute-Replan 模式（`plan_execute_replan/`）已有自省循环，但 Multi-Agent 路径没有。

---

### 瓶颈 5：两条路径各走各的，无法统一调度

```
ShouldUseMultiAgentForChat():
  关键词命中 → Multi-Agent（Supervisor → Runtime.Dispatch）
  关键词未命中 → ReAct Agent（chat_pipeline → Eino compose.Graph）
```

**问题：**
- 两套执行引擎，代码维护成本高
- 无法 A/B 对比（接口不同，结果结构不同）
- 无法渐进迁移（all-or-nothing）

---

### 瓶颈 6：Contract 只是元数据，运行时不强制

**代码证据：**

```go
// contracts.go
Agent Contract {
    Must:    [...]    // ← 定义了但 Dispatch 时不检查
    MustNot: [...]    // ← 定义了但 Dispatch 时不检查
}
```

**问题：** 只能事后查 trace，不能事前拦截。一个 Specialist 如果输出了不该输出的格式、调了不该调的工具，运行时不会阻止。

## 三、优化方案

### 🔴 P0：升级 Triage —— 从关键词到语义路由

**目标：** 路由准确率从"关键词覆盖率"提升到语义理解。

**方案：渐进式 Hybrid Triage**

```
阶段1（低风险，立即可做）：
  ├─ 关键词先匹配（保留现有逻辑，快速路径，零延迟）
  └─ 未命中 → LLM 快速分类（轻量 prompt，超时 1s fallback 到关键词）

阶段2（进阶）：
  ├─ 输入扩展：query + memory_context（利用用户历史偏好）
  └─ 动态决定：是否需要多 Agent？还是单 Agent 就够了？
```

**LLM 分类 Prompt 设计：**

```
你是一个运维查询分类器。根据用户输入，输出 JSON：

输入：{query}

输出格式：
{
  "intent": "alert_analysis | incident_analysis | kb_qa | data_query",
  "domains": ["metrics", "logs", "knowledge"],  // 需要的领域
  "priority": "high | medium | low",
  "use_multi_agent": true | false  // 是否需要多 Agent
}

规则：
- alert_analysis: 涉及告警、指标异常、性能问题 → metrics + logs + knowledge
- incident_analysis: 涉及故障排查 → logs + knowledge
- kb_qa: 纯知识查询 → knowledge only
- 简单问候、闲聊 → use_multi_agent: false
```

**收益：**
- 路由准确率质的提升
- 减少不必要的多 Agent 调用（降成本）
- 新增 domain 不需要维护关键词表

**风险：** LLM 调用增加延迟。缓解：超时 1s fallback 到关键词；可考虑用更轻量的模型。

---

### 🟠 P1：Specialist 间的信息传递 —— Staged Execution

**目标：** 让后执行的 Specialist 利用先执行的结果聚焦。

**方案：Staged Execution**

```
  ┌──────────┐
  │  Triage  │ → 确定 primary_domain + execution_order
  └────┬─────┘
       ▼
  ┌──────────┐
  │ Primary  │ → 先跑最高优先级 specialist
  │Specialist│    输出 focus_hints: {service: "paymentservice", anomaly: "CPU"}
  └────┬─────┘
       │
       ▼
  ┌──────────┐  ┌──────────┐
  │  Logs    │  │Knowledge │ → 第二阶段 specialist 收到 focus_hints
  │(带focus) │  │(带focus) │    检索范围聚焦到具体服务
  └────┬─────┘  └────┬─────┘
       │              │
       └──────┬───────┘
              ▼
         ┌──────────┐
         │ Reporter │
         └──────────┘
```

**实现要点：**

1. **Triage 扩展输出**：新增 `primary_domain` 和 `execution_order` 字段
2. **TaskEnvelope 扩展**：新增 `focus_hints` 字段
3. **Supervisor 改造**：从并行 `WaitGroup` 改为分阶段执行
4. **Specialist 适配**：检查 `focus_hints`，有则聚焦检索

```go
// protocol/types.go 新增
type FocusHint struct {
    Service  string `json:"service"`
    Anomaly  string `json:"anomaly"`
    Severity string `json:"severity"`
}

type TaskEnvelope struct {
    // ...existing fields...
    FocusHints []FocusHint `json:"focus_hints,omitempty"`
}
```

**收益：**
- Logs/Knowledge 检索更精准（从"全局扫"变"聚焦查"）
- 减少无关信息噪音
- 可能反而降低总延迟（第二阶段更快）

**风险：** 串行化可能增加总延迟。缓解：如果执行顺序中多个 domain 没有依赖，仍可并行。

### 🟠 P1：Reporter 升级 —— 从模板拼接转为 LLM 合成

**目标：** 输出从"信息罗列"变为"智能诊断"。

**方案：LLM-Powered Reporter**

```
当前 Reporter 输出（模板拼接）：
  # 告警分析报告
  ## 用户目标：...
  ## 指标：CPU 使用率 85%
  ## 日志：paymentService 出现 context canceled
  ## 知识库：建议检查资源限制
  ## 结论：
  - 指标：CPU 使用率 85%
  - 日志：paymentService 出现 context canceled
  - 知识库：建议检查资源限制

优化后 Reporter 输出（LLM 合成）：
  ## 诊断结论
  根据多维度分析，paymentservice 的 CPU 告警很可能是由资源不足引起：
  1. 当前 CPU 使用率 85%（指标），接近 limit
  2. 日志中出现 context canceled × 12（日志），可能是 CPU throttling 导致超时
  3. 知识库建议检查 HPA 配置和 resource limits（知识库），与前述发现一致

  → 建议操作：调整 HPA 阈值或增加 resource limits

  ## 证据链
  | 来源   | 发现                      | 置信度 |
  |--------|---------------------------|--------|
  | 指标   | CPU 85%，接近 limit       | 高     |
  | 日志   | context canceled × 12     | 中     |
  | 知识库 | 资源限制 SOP              | 高     |

  ## 矛盾项
  无

  ## 建议下一步
  1. 检查 HPA 配置
  2. 查看 paymentservice 的 resource limits
  3. 如确认资源不足，调整配置或扩容
```

**LLM Prompt 设计：**

```
你是一个运维诊断专家。根据以下 Specialist 的检索结果，生成一份诊断报告。

用户问题：{query}
意图分类：{intent}

Specialist 结果：
{specialist_results}

要求：
1. 交叉验证各来源的发现，标注一致性
2. 检测矛盾项（如指标正常但日志大量报错）
3. 生成置信度评估
4. 给出可操作的建议步骤
5. 用 Markdown 格式输出
```

**与"Reporter 用 LLM"和"直接让 ReAct Agent 做所有事"的区别：**
- ReAct Agent 做所有事 = 一个 LLM 同时负责检索决策 + 结果评估 + 综合回答（单点）
- Multi-Agent + LLM Reporter = 检索由 Specialist 分工完成（可并行/串行），LLM 只负责最后的综合推理。分工更清晰，每一步可独立优化。

**收益：**
- 输出质量质的飞跃
- 自动交叉验证和矛盾检测
- 生成可操作的建议

**风险：** 增加一次 LLM 调用 → 增加成本 + 延迟。缓解：仅对复杂查询启用 LLM Reporter；简单查询保留模板。

### 🟡 P2：加入自省循环（Self-Reflection）

**目标：** 一次执行不够就自动重试，提高回答完整度。

**方案：Reflect Node**

```
Supervisor
    │
    ├─ Triage → Specialists → Reporter
    │                              │
    │                              ▼
    │                      ┌──────────────┐
    │                      │ Self-Reflect │ ← 新增
    │                      │ (轻量LLM判断) │
    │                      └──────┬───────┘
    │                             │
    │              ┌──────────────┼──────────────┐
    │              ▼              ▼              ▼
    │          "足够了"     "证据不足"    "结果矛盾"
    │              │              │              │
    │              ▼              ▼              ▼
    │           返回结果    重新查询(最多1次)  标记需人工介入
```

**Self-Reflect 判断标准：**

```
根据以下报告，评估是否需要补充信息：

报告：{reporter_output}

回答：
{
  "status": "complete | incomplete | contradiction",
  "reason": "简短说明",
  "missing_domains": [],  // 如果不完整，还需要查什么
  "refined_query": ""     // 如果需要重新查询，精炼后的问题
}

判断标准：
- complete: 证据充分，结论明确
- incomplete: 信息不足，需要补充
- contradiction: 来源间存在矛盾，需标记
```

**实现要点：**
- 在 Supervisor 中加入 `SelfReflect` 节点
- 最多重新查询一轮（防无限循环）
- 通过 Event Bus 记录自省过程（Traceable）
- `max_reflect_rounds` 可配置（默认 1）

**收益：**
- 处理"一次查询证据不足"的场景
- 矛盾自动标记（不静默忽略）
- 可追溯（知道系统为什么重试了）

### 🟡 P2：统一两条执行路径

**目标：** Single-Agent (ReAct) 和 Multi-Agent (Supervisor) 通过统一 Runtime 调度。

**方案：ChatPipeline 实现 runtime.Agent 接口**

```go
// 当前：ChatPipeline 是 Eino compose.Runnable
func BuildChatAgent(ctx) (compose.Runnable[*UserMessage, *schema.Message], error)

// 优化后：ChatPipeline 也实现 runtime.Agent
type ChatPipelineAgent struct {
    runnable compose.Runnable[*UserMessage, *schema.Message]
}

func (a *ChatPipelineAgent) Name() string { return "chat_pipeline" }

func (a *ChatPipelineAgent) Handle(ctx context.Context, task *protocol.TaskEnvelope) (*protocol.TaskResult, error) {
    msg, err := a.runnable.Invoke(ctx, taskToUserMessage(task))
    if err != nil {
        return nil, err
    }
    return &protocol.TaskResult{
        TaskID:  task.TaskID,
        Agent:   a.Name(),
        Status:  protocol.ResultStatusSucceeded,
        Summary: msg.Content,
    }, nil
}
```

**收益：**
- 通过统一 Runtime 调度，共享 Ledger/Bus/Artifacts
- A/B 测试更简单（同一接口，切换 agent name）
- 未来可以动态选择（简单查询 → ChatPipeline，复杂查询 → Supervisor）

### 🟢 P3：Contract 运行时强制执行

**目标：** 在 Dispatch 时检查 Contract，事前拦截而非事后回溯。

**方案：Dispatch 前置拦截器**

```go
// runtime/runtime.go Dispatch 方法改造
func (r *Runtime) Dispatch(ctx context.Context, task *protocol.TaskEnvelope) (*protocol.TaskResult, error) {
    agent := r.registry.Get(task.Assignee)
    contract := contracts.Get(agent.Name())

    // 1. 输入校验
    if violations := contract.ValidateInput(task); len(violations) > 0 {
        return degradedResult(task, "input_violation", violations), nil
    }

    // 2. 执行
    result, err := agent.Handle(ctx, task)

    // 3. 输出校验
    if violations := contract.ValidateOutput(result); len(violations) > 0 {
        result.Status = protocol.ResultStatusDegraded
        result.DegradationReason = strings.Join(violations, "; ")
    }

    return result, err
}
```

**收益：**
- 越权操作在运行时被拦截
- 违规操作被记录为 Degraded（不崩溃）
- Contract 从"文档"变为"守卫"

### 🟢 P3：成本感知路由

**目标：** 简单问题不调用完整的 Multi-Agent 流程。

**方案：在 ShouldUseMultiAgentForChat 中加复杂度判断**

```go
func ShouldUseMultiAgentForChat(ctx context.Context, query string) bool {
    // 第1层：功能开关
    if !chatMultiAgentEnabled(ctx) { return false }

    // 第2层：复杂度判断
    complexity := estimateComplexity(query)
    switch complexity {
    case "simple":   // "什么是 HPA？"
        return false // RAG only
    case "medium":   // "MySQL 连接数异常"
        return false // ReAct Agent 够用
    case "complex":  // "checkoutservice 全链路故障"
        return true  // 需要 Multi-Agent
    }
}
```

**复杂度判断：** 可以用 LLM 快速分类（同 Triage），也可以基于规则（关键词数量、query 长度、是否含多个服务名）。

## 四、优化路线图

```
         現在                        3个月                        6个月
          │                           │                            │
P0: Triage ────→ Hybrid Triage ────→ LLM Triage ──────────────→
          │                           │
P1: Info ────→ Staged Execution ──→ LLM Reporter ──────────────→
  Sharing     │  (先跑主要的，          │  (合成+矛盾检测)
              │   再跑辅助的)          │
              │                       │
P2: Feedback─┼───────────────────→ Self-Reflect ──→ Unified Path
   Loop      │                       (自省+重试)     (统一Runtime)
              │
P3: Contract─┼─────────────────────────────────→ Runtime Enforcement
   & Cost    │                                   + Cost Router
```

### 实施顺序的理由

1. **P0 Triage 先行**：路由是 Multi-Agent 的"大脑"，大脑不准，下游优化都白搭
2. **P1 Staged Execution + LLM Reporter 并行推进**：分别优化执行质量和输出质量，互不阻塞
3. **P2 Self-Reflect**：有了 LLM Reporter 后，自省循环才有判断基础
4. **P2 Unified Path**：当两条路径都稳定后统一
5. **P3**：锦上添花，在核心链路稳定后做

### 决策 2：P1 Staged Execution —— 质量 vs 速度

- [ ] 优先质量：Staged Execution（串行化，后续 Agent 利用前面的结果）
- [ ] 优先速度：保持当前并行模式，在 Reporter 端用 LLM 做交叉验证（速度更快但信息利用率低）
- [ ] 可配置：通过 `multi_agent.execution_mode: parallel | staged` 切换

### 决策 3：P1 LLM Reporter —— 成本 vs 质量

- [ ] 全面启用 LLM Reporter
- [ ] 仅复杂查询启用（通过 Triage 的 `use_multi_agent` 或复杂度判断）
- [ ] 可配置：通过 `multi_agent.reporter_mode: template | llm` 切换

### 决策 4：单 Agent vs 多 Agent 长期定位

- [ ] 多 Agent 做成主路径，逐步淘汰单 Agent ReAct
- [ ] 两条路径长期共存（简单查询走单 Agent，复杂查询走多 Agent）
- [ ] 先统一接口再决定（P2 Unified Path 之后再做判断）

### 决策 5：Skills 系统 vs Specialists 定位

当前 `skillspecialists/` 和 `specialists/` 两套并存。AGENTS.md 说"Skills 改造采用平行迁移策略"。优化多 Agent 时：

- [ ] 以 `skillspecialists/` 为基准，废弃 `specialists/`
- [ ] 合并两套，统一为 Skills-Driven Agent
- [ ] 暂时不管，等优化方案稳定后再处理
