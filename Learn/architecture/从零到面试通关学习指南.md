## 第四步：核心模块理解清单

你需要对每个模块都能回答三个问题：
1. **它做什么**（What）
2. **为什么需要它**（Why）
3. **你做了哪些设计**（How）

### 4.1 ReAct Agent（Chat Pipeline）

**📖 简单理解：** ReAct = Reasoning + Acting。模型先思考"我需要查什么"，然后调用工具去查，查完再思考"这些结果够不够"，不够就再查，循环往复。

**为什么需要它：**
> 普通 LLM 只能凭训练数据回答问题，不知道你的系统当前状态。ReAct Agent 让 LLM 能主动调用工具获取实时信息。

**你做了哪些设计：**
- 用 Eino 的图编排（Graph）构建执行流程：Input → Template → ReAct
- Prompt 分三层：静态规则 + 动态配置 + 运行时上下文
- RAG 文档不进 System Prompt（防注入 + 缓存友好）
- 最多 25 步推理循环，有超时保护

### 4.2 RAG 检索系统

**📖 简单理解：** 把文档切成小段 → 用 Embedding 模型转成向量 → 存到 Milvus → 用户提问时也转成向量 → 找最相似的文档段 → 喂给 LLM 做参考。

**为什么需要它：**
> LLM 不知道你们公司内部的 runbook、SOP 和历史故障处理经验。RAG 让 LLM 能"查资料"再回答。

**你做了哪些设计：**
- 四种检索模式：纯召回 / 改写 / 改写+重排 / 混合检索
- Query Rewrite：用 LLM 把"服务挂了"改写成"服务故障 pod crashloop"
- Rerank：检索出候选文档后，用 LLM 重新打分排序
- Hybrid：向量检索（语义相似）+ BM25（关键词匹配），融合召回
- RetrieverPool：Milvus 连接复用，失败短 TTL 缓存防雪崩

### 4.3 Context Engine（上下文工程）

**📖 简单理解：** LLM 一次最多读几千个 token。你不能把所有历史对话 + 所有记忆 + 所有文档都塞进去。Context Engine 就是"智能剪辑师"——按需、按预算、把最重要的信息选出来。

**为什么需要它：**
> 这是 LLM 应用从 Demo 走向工程化的关键一步。没有 Context Engine 的项目，要么 prompt 爆炸，要么信息遗漏。

**你做了哪些设计：**
- 四类来源：History / Memory / Documents / Tool Results
- 每类有 Token 预算上限（如 History 2000、Memory 500）
- 三种 Profile：chat / aiops / report，不同场景不同策略
- 全链路 Trace：记录选中了什么、丢弃了什么、为什么丢弃

### 4.4 Memory 系统

**📖 简单理解：** 三层记忆。短期记忆 = 当前对话历史；长期记忆 = 跨会话的重要信息（用户偏好、历史结论）；工作记忆 = 当前任务的工具调用结果。

**为什么需要它：**
> 没有记忆的 AI 每次都是"初次见面"。有了长期记忆，系统能记住"上次你说过 checkoutservice 的配置是这样的"，下次对话就能直接参考。

**你做了哪些设计：**
- 异步抽取：对话结束后，LLM 从对话中提取关键信息，存入长期记忆
- MQ + 本地 fallback 双链路：MQ 可用走 MQ，不可用走本地 goroutine
- 记忆过滤：置信度、安全标签、过期时间、作用域（Session/User/Project/Global）
- 全局上限 + per-session 上限：防止无限增长
- 读锁优化：平时用 RLock，只在更新访问计数时短暂拿写锁

### 4.5 MQ 异步任务

**📖 简单理解：** 用户提交问题后立刻拿到一个 task_id，可以关页面。后台 MQ 消费者执行任务，完成后结果写入 Redis。用户回来后用 task_id 查结果。

**为什么需要它：**
> AIOps 分析可能跑几分钟。如果让 HTTP 请求一直等着，用户关页面任务就断了。MQ 解耦"用户请求"和"任务执行"。

**你做了哪些设计：**
- 两条 MQ 链路：Chat Task（聊天异步）+ Memory Extraction（记忆抽取）
- RabbitMQ 拓扑：Exchange → Queue + Retry Queue（带 TTL 死信回流）+ DLQ
- Redis 状态机：queued → running → succeeded/failed
- 重试机制：失败自动进入 retry queue，超阈值进 DLQ
- 启动自愈：启动时自动连接 MQ，失败自动重试
- 优雅关闭：收到 SIGTERM → 标记 unready → 等待 in-flight → 关闭

### 4.6 安全与韧性

**📖 简单理解：** AI 系统比普通系统多一层风险——LLM 可能被 prompt injection 攻击，可能输出敏感信息，可能乱调工具。

**你做了哪些设计：**
- Prompt Guard：输入侧过滤危险 prompt
- Output Filter：输出侧过滤敏感信息
- 工具白名单：MySQL 只允许 SELECT，必须表白名单
- 审批门：高风险操作拦截
- 降级开关：手动关闭 AI 能力
- Token 审计：追踪 LLM 调用成本
- JWT + RateLimit：认证和限流

## 第六步：面试高频问题 & 你的回答

### Q1: "介绍一下你这个项目的架构"

> 整个系统分四层。最上是 Controller 层，用 GoFrame 做 HTTP 路由，支持 Chat、流式、AIOps、异步任务、文件上传五种入口。
>
> 中间是 Service 层，负责记忆管理、上下文装配、审批、降级、Token 审计。
>
> 再往下是 Agent 执行层，核心是用字节的 Eino 框架做的 ReAct Agent。它可以根据用户问题自动决定调用哪个工具——查 Prometheus、查日志、查知识库。复杂故障排查走 Plan-Execute-Replan 模式，先规划再执行再复核。
>
> 底层是四个支撑系统：RAG（知识检索）、Context Engine（上下文装配）、Memory（记忆管理）、MQ（异步任务）。

### Q2: "RAG 你是怎么做的？做了哪些优化？"

> RAG 我做了四个层面的优化。
>
> 第一层是 Query Rewrite：用户口语化的提问（比如"服务挂了"），先用 LLM 改写成更适合搜索的关键词形式。
>
> 第二层是检索：不是只做向量检索，我还加了 BM25 关键词检索，两种结果用 RRF 融合，互补召回。
>
> 第三层是 Rerank：Milvus 返回的候选文档可能排序不准，我再过一次 LLM 重新打分排序。
>
> 第四层是工程优化：RetrieverPool 按地址+top_k 缓存 Milvus 连接，失败也做短 TTL 缓存防止雪崩。整个链路每个环节都有超时保护和降级策略。

### Q3: "Context Engine 是什么？为什么要单独做？"

> Context Engine 解决的是 LLM 上下文管理的问题。LLM 一次只能处理有限的 token，但一个请求可能涉及历史对话、长期记忆、RAG 文档、工具结果——全塞进去 prompt 就炸了。
>
> 我的做法是：定义四类来源，每类分配 token 预算。不同场景（聊天/AIOps/报告）用不同 Profile 控制允许哪些来源。整个装配过程都有 Trace 记录，能看到选中了什么、丢弃了什么、为什么丢弃。
>
> 这比简单拼接字符串要工程化得多——出了问题能回溯，想做 A/B 测试能控制变量。

### Q4: "你的记忆系统怎么设计的？"

> 三层记忆。短期记忆就是当前对话的 history，长期记忆是跨会话的重要信息，工作记忆是当前任务的工具返回结果。
>
> 核心挑战是长期记忆的提取。我的做法是异步提取——对话结束后先写短期记忆，再通过消息队列异步触发 LLM 抽取关键信息。MQ 不可用时还有本地 goroutine fallback。
>
> 记忆写入前有校验：太短、太长、纯代码、assistant 套话都会过滤掉。记忆还有作用域（Session/User/Project/Global）、置信度、安全标签和过期时间，避免长期运行后上下文被污染。

### Q5: "为什么用消息队列？"

> 两个原因。一是解耦请求和任务生命周期——AIOps 分析可能跑几分钟，如果用户关页面任务就断了。用了 MQ，用户提交后拿 task_id 就可以关页面，回来再查结果。
>
> 二是有两条独立的异步链路。一条是 Chat Task：用户请求入队，消费者执行，结果存 Redis。另一条是 Memory Extraction：对话结束后异步抽取长期记忆，不阻塞用户。
>
> 可靠性方面：RabbitMQ 配了 Retry Queue（带 TTL 死信回流）和 DLQ。失败先重试，超阈值进死信。状态机持久化在 Redis。

### Q6: "遇到的最大挑战是什么？"

> 最大的挑战是 RAG 评测的时候，我在云端服务器的部署策略出了问题。
>
> 我一开始想把完整的原始遥测数据（11GB+ 的 parquet 文件）上传到云端做预处理。逻辑上没错，但那台服务器根盘只有 40G，已经跑着 Docker 容器和其他服务。数据传上去后盘直接满了。
>
> 我没有硬撑，先做了资源归因——查清楚每个目录占了多少空间，发现原始遥测数据加上 Docker 层和缓存，把盘吃光了。
>
> 然后我把方案改成了：本地做重预处理（parquet → markdown 证据文档），云端只接收轻量产物、建索引、跑评测。同时把云端脚本改成支持 skip-telemetry 模式，保证这条轻量路径是可执行的。
>
> 这件事让我学到了一点：**"能做"不等于"应该做"**——部署方案要考虑资源约束，不是功能对了就行。

### Q7: "这个项目还有什么可以改进的？"

> 三个方向。第一是 RAG 评测闭环——目前 baseline 还在跑，需要固化 build/holdout split，持续跟踪 Recall@K。
>
> 第二是 Prompt 管理——目前 prompt 写在代码里，后续如果调优频繁，应该抽到外部文件，支持版本管理。
>
> 第三是可接入 Anthropic Prompt Caching——Prompt 分层结构已经为 CacheScope 预留了标记，如果以后切 Claude 模型，可以直接映射。

## 第七步：理解你项目的"底层逻辑"

这部分帮你建立更深层的理解，面试时能说出"为什么"。

### 7.1 这个项目的本质是什么？

本质上是把**信息检索 + 证据整合**自动化。

传统运维流程：人 → 查 Prometheus → 查日志 → 查 Wiki → 脑中整合 → 输出结论。
你的系统：人 → 自然语言 → 系统自动查多个数据源 → LLM 整合 → 输出结论。

### 7.2 为什么 LLM 应用需要这么多"工程化"？

裸调 LLM API 只需要 10 行代码。但真正能用的系统需要：

- **上下文管理**：不然 LLM 不知道你在说什么
- **工具调用**：不然 LLM 不知道实时信息
- **记忆系统**：不然每次都是"初次见面"
- **安全防护**：不然 prompt injection 就搞崩你
- **异步任务**：不然长任务卡住 HTTP 请求
- **可观测性**：不然出了问题不知道哪坏了

你的项目把这些都做了，这就是"工程化"。

### 7.3 你的项目"技术含量"在哪里？

| 层级 | 做了什么 | 技术含量 |
|------|---------|---------|
| Agent | ReAct + Plan-Execute-Replan | Eino 框架的深度使用 |
| RAG | Rewrite + Retrieve + Rerank + Hybrid | 检索链路完整闭环 |
| Context | Assembler + Budget + Profile + Trace | 上下文工程的独立模块 |
| Memory | 三层记忆 + MQ 异步 + 多维过滤 | 记忆系统的长期运行考虑 |
| 韧性 | 超时/降级/fallback/重试覆盖全链路 | 生产级系统的必备能力 |

## 第八步：学习路径建议

按这个顺序学，每天 1-2 小时，一周能讲清楚：

| 天 | 学什么 | 怎么学 |
|----|--------|--------|
| **第1天** | 整体结构 | 读 `main.go` + `AGENTS.md`，画一张自己的架构图 |
| **第2天** | Chat Pipeline | 读 `chat_pipeline/`，理解 ReAct 怎么执行的 |
| **第3天** | RAG | 读 `rag/query.go`，跟踪一次检索的完整链路 |
| **第4天** | Context + Memory | 读 `contextengine/` 和 `memory_service.go`，理解上下文怎么装配 |
| **第5天** | MQ 异步 | 读 `chat_task_queue.go`，画 RabbitMQ 拓扑图 |
| **第6天** | 面试模拟 | 对着镜子把上面 7 个问题各讲一遍 |
| **第7天** | 查漏补缺 | 读 `Learn/面试/` 里的文档，补充故事细节 |

## 附录：快速自检清单

面试前，逐条问自己，能脱口而出才算过关：

- [ ] 能用一句话说清楚这个项目是干什么的
- [ ] 能说出技术栈里每个技术的作用和选型原因
- [ ] 能画出系统架构图（四层）
- [ ] 能讲清楚一次请求从头到尾怎么走的
- [ ] 能讲清楚 RAG 的四个优化点
- [ ] 能讲清楚 Context Engine 为什么单独做
- [ ] 能讲清楚记忆系统的三层设计和异步抽取流程
- [ ] 能讲清楚 MQ 的两条链路和可靠性机制
- [ ] 能讲出部署磁盘爆满的故事（问题→诊断→修正→教训）
- [ ] 能说出三个改进方向
