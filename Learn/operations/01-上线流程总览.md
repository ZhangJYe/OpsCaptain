# GitHub Actions + 阿里云上线流程总览

这套方案是按“先稳定上线，再逐步演进”的思路设计的，适合当前这个仓库。

## 一、先判断项目特点

这个项目不是单纯的静态站点，也不是纯无状态服务。

- 后端是 Go 服务，已经有可直接复用的容器化入口。
- 前端是静态页面，不需要复杂前端构建。
- 系统依赖外部服务，例如大模型 API、Milvus、MySQL、Prometheus、MCP。
- 项目会写本地文件和运行时数据，所以它还不适合一上来就做多副本弹性扩容。

基于这些特点，最稳妥的第一版生产方案是：

1. GitHub Actions 负责 CI/CD。
2. 镜像推送到阿里云 ACR。
3. 用阿里云 ECS 跑 Docker Compose。
4. 前端和后端都容器化，放在同一台 ECS 上。
5. MySQL 使用 RDS，Milvus 独立部署，不和业务服务打包在一起。

## 二、为什么我没有直接让你上 ACK/Kubernetes

因为当前代码里有两个现实约束：

- 上传文件会落到本地目录。
- 运行时数据也会落到本地目录。

这意味着当前版本更像“有状态单体服务”，而不是标准的“可随意横向扩容的无状态服务”。

如果现在直接上 Kubernetes，你会很快遇到这些问题：

- 文件放哪儿。
- 多副本之间的数据如何一致。
- Milvus 和业务容器如何拆分。
- 变更后的排障复杂度明显提高。

所以第一版上线，我选择的是 ECS + Docker Compose。  
这是工程上更稳、更容易学、也更容易运维的路线。

## 三、实际发布链路

完整链路如下：

1. 开发者 push 代码到 GitHub。
2. `ci.yml` 先执行 Go 测试，并验证前后端镜像都能构建。
3. `cd.yml` 在主分支或手动触发时构建正式镜像。
4. GitHub Actions 登录阿里云 ACR，并推送前后端镜像。
5. GitHub Actions 通过 SSH 连接 ECS。
6. ECS 拉取新的镜像，并用 `docker compose up -d` 更新服务。
7. 部署脚本用 `/healthz` 做一次发布后验证。

## 四、这次我额外修掉了什么问题

为了让这条链路真正可用，我顺手把几个上线前必须处理的问题一起补了：

- 增加了 `.dockerignore`，避免把本地 `.env`、测试产物、Milvus 本地数据卷打进镜像。
- 给前端增加了 `config.js` 运行时配置能力，不再把 API 地址写死在 `localhost`。
- 给前端单独加了 Nginx 容器，统一走同域 `/api` 代理，避免 CORS 和部署地址错乱。
- 补了生产环境专用的 `deploy/config.prod.yaml`，把运行参数和部署参数分开。

## 五、你应该怎样理解这套设计

你可以把它分成三层：

- 第 1 层：代码是否可测试、可构建。
- 第 2 层：镜像是否安全、可复制、可部署。
- 第 3 层：线上环境是否能稳定拿到配置、依赖和健康状态。

这三层如果有任意一层没做好，上线都会变成“碰运气”。

这也是我在实现里一直强调的核心工程原则：

- 先保证可重复。
- 再保证可发布。
- 最后再追求更高级的平台形态。
