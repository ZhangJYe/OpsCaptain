# 第 7 章：MQ 异步任务 — "用户关页面，任务继续跑"

> **本章目标**：理解 MQ 异步任务的完整架构，掌握 RabbitMQ 拓扑、Redis 状态机、可靠性机制和双链路设计。能向面试官清晰解释"请求和任务为什么要解耦"以及"重试/死信/自愈/优雅关闭"的工程价值。

## 1. 白话理解：什么是 MQ 异步任务？

### 1.1 一句话解释

**MQ 异步任务 = "下单取餐"模式**。你点完餐拿一张小票就去找位子坐了，后厨异步做菜，做好了叫号，你再去取。你不用一直站在柜台前等，后厨也不会因为你走了就不做你的菜。

### 1.2 一个类比：你去吃麻辣烫

```
同步模式（没有 MQ）:
  你: "老板，来一碗麻辣烫"              ← HTTP 请求
  老板: 开始洗菜、切菜、煮、调味...        ← 任务执行（可能要 3-5 分钟）
  你: 站在柜台前干等...                   ← HTTP 连接保持中
  你: 等不及，走了                        ← 用户关页面/超时
  老板: 做好了出来喊人：人呢？             ← 任务白做了

异步模式（有 MQ）:
  你: "老板，来一碗麻辣烫"
  老板: 给你一张小票（#37）               ← 返回 task_id，立即响应
  你: 找位子坐下，刷手机                  ← 用户关页面也行
  老板: 小票进厨房 → 做菜                 ← MQ 消费执行
  老板: #37 好了！                        ← Redis 写入 "succeeded"
  你: 拿小票取餐                          ← 用 task_id 查结果
```

### 1.3 一张图看懂

```
┌──────────────────────────────────────────────────────────────────┐
│                    MQ 异步任务架构全貌                              │
│                                                                  │
```

## 2. 为什么需要 MQ 异步任务？

### 2.1 AIOps 分析的现实挑战

AIOps 故障诊断不是"秒回"的场景：

```
一个典型的 AIOps 请求可能涉及：
├── Query Rewrite（LLM 改写查询）          ~2s
├── RAG 检索（向量搜索 + Rerank）          ~3s
├── Prometheus 查询告警                    ~2s
├── 日志检索                               ~5s
├── ReAct Agent 多步推理                   ~30-60s
│   ├── Step 1: 理解问题，调用 query_docs   ~5s
│   ├── Step 2: 查指标，调用 query_metrics  ~3s
│   ├── Step 3: 查日志，调用 query_log      ~5s
│   ├── Step 4: 交叉分析                    ~10s
│   └── Step 5: 生成诊断报告               ~15s
├── Plan-Execute-Replan（复杂场景）   最多 3min
└── 记忆异步抽取                           ~1s
────────────────────────────────────────
总计：可能 1-5 分钟
```

### 2.2 没有 MQ 会怎样？

| 问题 | 具体表现 |
|------|---------|
| **HTTP 超时** | 浏览器/Nginx/负载均衡默认超时通常 30-120s，超过就断开 |
| **用户焦虑** | 盯着白屏等 3 分钟，体验极差 |
| **资源浪费** | HTTP 长连接占用 Web Server 连接池 |
| **任务丢失** | 用户关页面或网络断开 → 连接中断 → 服务端 context canceled → 任务被取消 |
| **不可恢复** | 任务一旦被 cancel 就没有重试机制，只能重新提交 |

### 2.3 MQ 带来的核心价值

```
没有 MQ：用户请求生命周期 = 任务执行生命周期（同生共死）

有 MQ：  用户请求生命周期 ≠ 任务执行生命周期（解耦！）

用户只需要两件事：
  1. 提交任务 → 立刻拿到 task_id
  2. 查询结果 → 用 task_id 轮询/WebSocket 拿结果

中间发生什么（关页面、断网、换设备），任务照常执行。
```

## 3. 两条 MQ 链路

OpsCaption 有两条独立的 RabbitMQ 异步链路，分别处理不同类型的耗时任务。两者共享同一个 Exchange (`opscaption.events`)，但使用不同的 RoutingKey、Queue 和 Consumer 实例，互不干扰。

### 3.1 链路一：Chat Task（聊天异步任务）

**场景**：用户提交 AIOps 分析请求，预计耗时 1-5 分钟。

**Controller 层代码**（`internal/controller/chat/chat_v1_chat_task.go`）：

```go
func (c *ControllerV1) ChatSubmit(ctx context.Context, req *v1.ChatSubmitReq) (res *v1.ChatSubmitRes, err error) {
    // ① 上下文装配: SessionID / RequestID / TraceID
    ctx = context.WithValue(ctx, consts.CtxKeySessionID, id)
    ctx = context.WithValue(ctx, consts.CtxKeyRequestID, requestID)

    // ② Prompt 安全检查
    if err := rejectSuspiciousPrompt(ctx, msg); err != nil {
        return nil, err
    }

    // ③ 提交异步任务 → Service 层
    task, err := submitChatTask(ctx, id, msg)  // → SubmitChatTask()

    // ④ 返回 task_id，用户可立即关闭页面
    return &v1.ChatSubmitRes{
        TaskID:    task.ID,
        Status:    string(task.Status),  // "queued"
        CreatedAt: task.CreatedAt,
    }, nil
}

func (c *ControllerV1) ChatTask(ctx context.Context, req *v1.ChatTaskReq) (res *v1.ChatTaskRes, err error) {
    // ⑤ 用 task_id 查询 Redis 获取结果
    record, err := getChatTask(ctx, taskID)  // → GetChatTask() → Redis GET

    // ⑥ 会话隔离校验: 不属于该 session 的 task 返回 403
    if sessionID != "" && sessionID != record.SessionID {
        writeStatus(ctx, http.StatusForbidden)
        return nil, fmt.Errorf("task does not belong to session")
    }
    return &v1.ChatTaskRes{...}, nil
}
```

**Service 层核心逻辑**（`internal/ai/service/chat_task_queue.go`）：

```go
func SubmitChatTask(ctx context.Context, sessionID, query string) (*ChatTaskRecord, error) {
    // ① 生成 UUID + 写入 Redis（status=queued, TTL=24h）
    record := &ChatTaskRecord{
        ID: uuid.NewString(), SessionID: sessionID, Query: query,
        Status: ChatTaskStatusQueued, CreatedAt: now, UpdatedAt: now,
    }
    saveChatTaskRecord(ctx, cfg, record)  // g.Redis().Do("SETEX", key, ttl, json)

    // ② 发布消息到 RabbitMQ Exchange（routingKey: chat.task.request）
    client.publishEvent(ctx, chatTaskEvent{
        TaskID: record.ID, SessionID: record.SessionID,
        Query: record.Query, Attempt: 0,
    }, client.cfg.RoutingKey)

    // ③ 立即返回 task_id
    return record, nil
}
```

**Consumer 执行流程**（`handleDelivery` → `processEvent`）：

```go
func (c *rabbitMQChatTaskClient) processEvent(event chatTaskEvent) error {
    exec := getChatTaskExecutor()  // 异步注入的执行函数

    // ① 幂等去重（在 handleDelivery 中已完成）
    // ② Redis: status=running, StartedAt=now
    record.Status = ChatTaskStatusRunning
    saveChatTaskRecord(context.Background(), c.cfg, record)

    // ③ 超时控制: execute_timeout_ms（默认 120s）
    execCtx, cancel := context.WithTimeout(context.Background(), c.cfg.ExecuteTimeout)
    defer cancel()
    result, err := exec(execCtx, event.SessionID, event.Query)

    // ④ 成功 → Redis: succeeded + answer + detail + trace_id
    record.Status = ChatTaskStatusSucceeded
    record.Answer = result.Answer
    record.Detail = result.Detail
    saveChatTaskRecord(context.Background(), c.cfg, record)

    // ⑤ 失败 → 由 handleDelivery 的上层逻辑进入重试/DLQ
    return err
}
```

**publishRaw 细节**——消息持久化（Persistent + TTL 配合死信回流）：

```go
func (c *rabbitMQChatTaskClient) publishWithCurrentChannel(...) error {
```

### 3.2 链路二：Memory Extraction（记忆异步抽取）

**场景**：每轮对话结束后，LLM 从对话中抽取长期记忆。每次耗时约 1-2s，虽短但频率高（每轮都触发），不适合阻塞 HTTP 响应。

**完整流程**（`internal/ai/service/memory_service.go`）：

```go
func (s *MemoryService) PersistOutcome(ctx context.Context, sessionID, query, summary string) {
    if strings.TrimSpace(summary) == "" {
        return
    }

    // ① 同步写入 SimpleMemory（短期记忆，必须成功）
    sessionMem := mem.GetSimpleMemory(sessionID)
    sessionMem.AddUserAssistantPair(query, summary)

    // ② 尝试入队 MQ → enqueueMemoryExtraction()
    enqueued, err := enqueueMemoryExtraction(ctx, sessionID, query, summary)
    if enqueued {
        return  // MQ 入队成功，链路结束
    }

    // ③ MQ 不可用 → 本地 fallback 链路
    release, err := acquireMemoryExtractionSlot(ctx)  // semaphore 获取槽位（最多 8）
    if err != nil {
        return  // 槽位耗尽，本次跳过抽取（不阻塞主流程）
    }

    // ④ 启动独立 goroutine 异步抽取
    go func(parent context.Context) {
        defer release()
        // context.WithoutCancel: 脱离父请求生命周期
        extractCtx, cancel := boundedMemoryContext(parent)
        defer cancel()
        report := processMemoryEventFunc(extractCtx, ...)
    }(ctx)
}
```

**boundedMemoryContext**——独立 goroutine 的超时与隔离：

```go
func boundedMemoryContext(parent context.Context) (context.Context, context.CancelFunc) {
    base := context.Background()
    if parent != nil {
        base = context.WithoutCancel(parent)  // ← 不受父请求 cancel 影响
    }
    return context.WithTimeout(base, memoryExtractionTimeout(parent))  // 默认 1500ms
}
```

**MQ Consumer 消费逻辑**（`internal/ai/service/memory_queue.go`）：

```go
func (c *rabbitMQMemoryClient) consumeEvent(event memoryExtractionEvent) (err error) {
    defer func() {
        if recovered := recover(); recovered != nil {
            err = fmt.Errorf("memory extract panic: %v", recovered)  // panic 兜底
        }
    }()

    extractCtx, cancel := context.WithTimeout(context.Background(), c.cfg.MemoryExtractTimeout)
    defer cancel()

    report := processMemoryEventFunc(extractCtx, mem.MemoryEvent{
        SessionID: event.SessionID, Query: event.Query, Answer: event.Summary, ...
    })
    // agent=llm → LLMMemoryAgent（LLM 不可用时 fallback 到 RuleMemoryAgent）
    // agent=rule → RuleMemoryAgent（纯规则提取）
    return nil
}
```

**Memory Agent 降级链**（`processMemoryEventWithConfiguredAgent`）：

```go
func processMemoryEventWithConfiguredAgent(ctx context.Context, event mem.MemoryEvent) *mem.MemoryExtractionReport {
    agent := mem.MemoryAgent(mem.NewRuleMemoryAgent())  // 默认用 rule

    if loadMemoryAgentMode(ctx) == "llm" && memoryAgentLLMConfigured(ctx) {
        chatModel, err := newMemoryChatModel(ctx)
        if err == nil {
            agent = mem.NewLLMMemoryAgent(chatModel, mem.NewRuleMemoryAgent())
            // LLM Agent 内嵌 rule fallback
        }
    }
    return mem.ProcessMemoryEventWithAgent(ctx, event, agent)
}
```

**双链路设计全景对比**：

```
                        PersistOutcome()
                             │
                  ① sync     │     ② async
              ┌──────────────┼──────────────┐
              ▼              ▼              ▼
        SimpleMemory    MQ 入队尝试     MQ 不可用时
        (同步写入)          │          本地 goroutine
        ✓ 必须成功      ┌───┴───┐     + semaphore(8)
                        │成功   │失败       │
                        ▼       ▼          ▼
                  RabbitMQ   返回false  context.WithoutCancel
                  Consumer      │       + context.WithTimeout
                  消费抽取      │       processMemoryEventFunc
                        └───────┘
                      (降级到本地链路)

Memory Agent 内部还有第二层降级:
  agent=llm → LLM 可用 → LLMMemoryAgent
            → LLM 不可用 → RuleMemoryAgent (内嵌 fallback)
  agent=rule → RuleMemoryAgent
```

### 4.2 重试机制详解

重试不靠代码里的 `for` 循环 sleep，而是靠 RabbitMQ 的**死信机制**：

```
第 1 次消费失败：
  ├─ attempt=0 < max_retries=3
  ├─ 发布消息到 retry queue（routingKey: xxx.retry）
  ├─ retry queue 设了 x-message-ttl=2s →
  │   2 秒后消息"过期"，被自动 dead-letter 回主队列
  └─ ACK 原消息（不 requeue，避免死循环）

第 2 次消费（消息已自动回到主队列）：
  ├─ attempt=1 < 3
  └─ 同上，再回 retry queue

第 3 次消费：
  ├─ attempt=2 < 3
  └─ 同上

第 4 次消费：
  ├─ attempt=3 >= max_retries=3
  ├─ 不再进入 retry queue
  └─ 直接发布到 DLQ + 标记 completed
```

**为什么用死信而不是代码 sleep？**

| 方式 | 优点 | 缺点 |
|------|------|------|
| 代码 sleep 重试 | 简单直观 | goroutine 阻塞浪费资源；服务重启丢失重试状态 |
| 死信回流 | 不占 goroutine；重启不丢 | 需要理解 RabbitMQ TTL 和 DLX |

## 5. Redis 状态机

### 5.2 Redis 存储设计

## 6. 可靠性机制

### 6.1 At-least-once + 幂等消费

RabbitMQ 保证 **at-least-once** 投递（网络抖动时可能重复投递），消费端必须**幂等**。

```go
// 幂等实现：completed TTL 集合
func (c *rabbitMQChatTaskClient) handleDelivery(delivery amqp.Delivery) {
    event, _ := decodeChatTaskEvent(delivery.Body)

    // ① 去重检查
    if c.completed.Has(event.TaskID) {    // TTL 集合，10 分钟内记住已处理的 task_id
        metrics.ObserveChatTask("deduped")
        c.ack(delivery)                    // 重复消息，直接 ACK 丢弃
        return
    }

    // ② 执行业务逻辑
    if err := c.processEvent(event); err != nil {
        // ③ 失败处理...
    }

    // ④ 标记已处理
    c.completed.Mark(event.TaskID)         // 加入 TTL 集合
    c.ack(delivery)
}
```

**幂等设计要点**：
- `completed` 是一个 **TTL 集合**：每 10 分钟自动清理过期条目，最多 20000 条
- 先查 `completed.Has(taskID)` 再执行，防止重复消费
- 任务完成后 `completed.Mark(taskID)` 标记

### 6.2 重试与死信队列（DLQ）

```
max_retries = 3（可配置）
retry_delay = 2s（消息在 retry queue 的 TTL）

执行失败时的处理逻辑：
  ├─ attempt < max_retries → publish to retry queue → 消息 2s 后回流主队列
  └─ attempt >= max_retries → publish to DLQ → 人工排查
```

DLQ 中的消息**不会自动重试**，需要：
1. 监控告警 → 运维人员发现
2. 排查原因 → 修复问题
3. 手动重新发布 → 或直接丢弃

### 6.3 超时保护

Chat Task 和 Memory Extraction 都有独立的超时保护：

```go
// Chat Task: execute_timeout_ms（默认 120s）
execCtx, cancel := context.WithTimeout(context.Background(), cfg.ExecuteTimeout)
defer cancel()
result, err := exec(execCtx, sessionID, query)

// Memory Extraction: extract_timeout_ms（默认 1500ms）
extractCtx, cancel := context.WithTimeout(context.Background(), cfg.MemoryExtractTimeout)
defer cancel()
report := processMemoryEventFunc(extractCtx, event)
```

| 任务 | 默认超时 | 配置项 |
|------|---------|--------|
| Chat Task 执行 | 120s | `chat_async.execute_timeout_ms` |
| 发布到 MQ | 2s | `rabbitmq.publish_timeout_ms` |
| Memory Extract | 1500ms | `memory.extract_timeout_ms` |
| MQ 连接超时 | 5s | 硬编码 |

### 6.4 启动自愈（Auto-Reconnect）

系统启动时不是"要么连上要么 panic"，而是**循环重试直到连上**：

```go
func startChatTaskInitLoop(cfg rabbitMQChatTaskConfig) {
    go func() {
        for {
            client, err := newChatTaskClient(cfg)  // 尝试连接 MQ
            if err == nil {
                if cfg.ConsumerEnabled {
                    if err = client.startConsumer(); err != nil {  // 启动 Consumer
                        _ = client.Close()
                    }
                }
                if err == nil {
                    closeAndSwapChatTaskClient(client)  // 原子替换全局 client
                    return  // 成功，退出循环
                }
            }
            // 失败 → 等待 reconnect_delay（默认 2s）→ 重试
            sleepMemoryReconnect(initCtx, delay)
        }
    }()
}
```

**这意味着**：
- 先启动 Web Server → 再异步连接 MQ
- MQ 没启动时，服务正常接受请求，只是异步任务不可用（降级）
- MQ 启动后自动连接，不需要重启整个服务

### 6.5 优雅关闭（Graceful Shutdown）

**完整关闭流程**（`main.go:waitForShutdown`）：

```go
func waitForShutdown(
    ctx context.Context,
    s *ghttp.Server,
    shuttingDown *atomic.Bool,
    pprofServer *http.Server,
    traceShutdown func(context.Context) error,
    memoryPipelineShutdown func(context.Context) error,
    chatTaskPipelineShutdown func(context.Context) error,
) {
    // Step 0: 注册信号监听
    sigCtx, stop := signal.NotifyContext(context.Background(), os.Interrupt, syscall.SIGTERM)
    defer stop()

    <-sigCtx.Done()  // 阻塞直到收到 SIGTERM / Ctrl+C

    // Step ①: 标记 shuttingDown=true → /readyz 返回 503
    g.Log().Info(ctx, "shutdown signal received")
    shuttingDown.Store(true)
    g.Log().Info(ctx, "server marked unready, waiting for in-flight requests")

    // Step ②: HTTP Server 优雅关闭（最多等 30s）
    if err := s.Shutdown(); err != nil {
        g.Log().Errorf(ctx, "server shutdown failed: %v", err)
    }
    // GoFrame 的 s.Shutdown() 做了三件事：
    //   a. 停止接受新连接
    //   b. 等待当前 in-flight 请求处理完成（最多 GracefulShutdownTimeout=30s）
    //   c. 超时后强制关闭

    // Step ③: pprof 调试端口关闭
    if pprofServer != nil {
        pprofServer.Shutdown(context.Background())
    }

    // Step ④: Memory Pipeline 关闭（cancel → 等 consumeDone → 关连接）
    if memoryPipelineShutdown != nil {
        memoryPipelineShutdown(context.Background())
        // → stopMemoryQueueInitLoop()
        //   → cancel(initCtx)          // 停止 init loop
        //   → closeAndSwapMemoryQueueClient(nil)
        //     → client.Close()
        //       → consumeCancel()      // 停止 consumer loop
        //       → <-consumeDone        // 等待当前消息处理完（最多 1s）
        //       → 关闭 channel + connection
    }

    // Step ⑤: Chat Task Pipeline 关闭
    if chatTaskPipelineShutdown != nil {
        chatTaskPipelineShutdown(context.Background())
    }

    // Step ⑥: 关闭基础设施资源（Redis 连接池等）
    health.CloseResources(ctx)

    // Step ⑦: 关闭 Tracing
    if traceShutdown != nil {
        traceShutdown(context.Background())
    }
}
```

**Client.Close() 内部细节**——等待 Consumer 处理完当前消息：

```go
func (c *rabbitMQChatTaskClient) Close(ctx context.Context) error {
    // ① 标记 closed，阻止新的发布和消费
    c.stateMu.Lock()
    c.closed = true
    consumeCancel := c.consumeCancel
    consumeDone := c.consumeDone
    c.stateMu.Unlock()

    // ② 取消 consumer 的 context → consumer loop 退出
    if consumeCancel != nil {
        consumeCancel()
    }

    // ③ 等待 consumer goroutine 完成（最多 1s）
    //    consume loop 在退出前会完成当前正在处理的 delivery
    if consumeDone != nil {
        select {
        case <-consumeDone:           // 正常完成
        case <-ctx.Done():            // 外部超时
        case <-time.After(time.Second): // 兜底超时
        }
    }

    // ④ 关闭 AMQP Channel 和 Connection
    c.swapAMQPState(nil, nil, nil)  // 原子替换为 nil
    closeMemoryChannel(oldConsumeCh)
    closeMemoryChannel(oldPublishCh)
    closeMemoryConnection(oldConn)
    return nil
}
```

**优雅关闭的完整时序图**：

```
时间线 →
──────────────────────────────────────────────────────────────────────────────

K8s/Scheduler             我们的服务                    RabbitMQ

  │                          │                            │
  │  发送 SIGTERM            │                            │
  │ ────────────────────────►│                            │
  │                          │                            │
  │                    ① shuttingDown.Store(true)         │
  │                    ② /readyz → 503                    │
  │                          │                            │
  │  停止路由流量            │                            │
  │ ◄─────────────────────── │                            │
  │                          │                            │
  │                    ③ s.Shutdown()                    │
  │                     ├─ 停止接受新 HTTP 连接            │
  │                     ├─ 等待 in-flight 请求完成        │
  │                     │  (正在处理 PersistOutcome       │
  │                     │   可能还在 publish MQ 消息)     │
  │                     └─ 最多等 30s                     │
  │                          │                            │
  │                          │  此时 MQ publish 正常       │
  │                          │ ────────────────────────►  │
  │                          │                            │
  │                    ④ memoryPipelineShutdown()        │
  │                     ├─ stop init loop                 │
  │                     ├─ cancel consumer context        │
  │                     ├─ 等 consumeDone                 │
  │                     │  (当前 delivery 处理完才退出)   │
  │                     └─ close connection               │
  │                          │                            │
  │                          │  Consumer ACK 最后一条消息  │
  │                          │ ────────────────────────►  │
  │                          │                            │
  │                    ⑤ chatTaskPipelineShutdown()      │
  │                     └─ 同上                            │
  │                          │                            │
  │                    ⑥ 关闭 Redis / Trace              │
  │                          │                            │
  │                    进程退出                           │
```

**关键设计决策**：

| 决策点 | 原因 |
|--------|------|
| 先关 HTTP，后关 MQ | HTTP 处理中可能还在 publish MQ 消息（如记忆抽取），过早关 MQ 会导致 publish 失败 |
| consume done 只等 1s | 单个消息处理超时由 execute_timeout/memory_extract_timeout 控制，1s 是 drain 兜底 |
| 关闭连接前先 cancel consumer | 让 consumer loop 感知到关闭信号，优雅退出而非强制断连 |
| `amqp.Persistent` 消息投递模式 | 即使进程崩溃，消息已持久化到 RabbitMQ 磁盘，重启后重新投递 |

## 7. 设计理念

### 7.1 核心原则：解耦请求生命周期和任务生命周期

```
传统同步模型：
  ┌──────────────────────────────┐
  │  用户请求开始                 │
  │    │                         │
  │    ├── 任务执行               │
  │    │     │                   │
  │    │     └── 如果用户断开     │
  │    │          → 任务取消      │
  │    │                         │
  │    └── 返回结果               │
  │  用户请求结束                 │
  └──────────────────────────────┘
  问题：请求和任务是"同生共死"的


MQ 异步模型：
  ┌─────────────────┐        ┌─────────────────┐
  │  用户请求        │        │  任务执行         │
  │  (短生命周期)    │  MQ    │  (长生命周期)     │
  │                 │ ◀────▶ │                 │
  │  submit → 返回   │  解耦   │  consume → 执行  │
  │  关页面 → 结束   │        │  完成 → Redis     │
  └─────────────────┘        └─────────────────┘
  优势：请求和任务"各活各的"，互不影响
```

### 7.2 降级优先（Degradation First）

OpsCaption 的 MQ 架构贯穿降级理念：

| 场景 | 降级策略 |
|------|---------|
| MQ 未连接 | Chat Task API 返回错误，但同步 Chat 可用 |
| MQ 发布失败 | 写入 Redis 后尝试发布；发布失败记录 failed 状态 |
| 记忆抽取 MQ 不可用 | 走本地 goroutine + semaphore 降级 |
| 记忆抽取 LLM 不可用 | LLM Agent → Rule Agent fallback |
| Consumer 执行超时 | context timeout → Redis 写入 failed → 触发重试 |

**核心信条**：局部失败不崩溃，总有降级路径。

## 8. 面试问答

### 问题 1："为什么用消息队列？"

**标准回答**：

> AIOps 分析可能跑几分钟。如果让 HTTP 请求一直等着，用户关页面任务就断了。MQ 解耦了"用户请求"和"任务执行"——用户提交后立刻拿到 task_id，可以关页面；后台 MQ Consumer 异步执行任务，完成后结果写入 Redis。用户回来后用 task_id 查结果。

**延伸亮点**（如果面试官追问）：

> 我们还设计了**两条独立链路**：Chat Task（聊天异步）+ Memory Extraction（记忆抽取）。记忆抽取有**双链路降级**：MQ 正常走 MQ，MQ 不可用走本地 goroutine + semaphore，保证核心记忆能力不丢。

### 问题 2："Chat Task 怎么实现的？具体说说链路"

**标准回答**：

> 用户 POST `/api/v1/chat_task/submit` → Controller 做三件事：① Redis 写入 `status=queued`；② 发布消息到 RabbitMQ Exchange（routingKey: `chat.task.request`）；③ 立刻返回 `task_id`。
>
> Consumer 拿到消息后：去重检查 → Redis 更新 `running` → 执行 Chat Pipeline（ReAct Agent / Plan-Execute-Replan）→ 成功写 `succeeded` + answer，失败且未超重试次数进 retry queue，超限进 DLQ。
>
> 用户用 `GET /api/v1/chat_task?task_id=xxx` 从 Redis 查结果。

### 问题 3："怎么保证可靠性？"

**标准回答**：

> 五层保障：
>
> 1. **At-least-once + 幂等消费**：RabbitMQ 保证消息至少投递一次；消费端用 TTL 集合去重（`completed.Has(taskID)`），防止重复执行。
> 2. **重试 + 死信队列**：失败消息进入 retry queue（TTL 2s 后自动回流主队列），最多重试 3 次。超过后进入 DLQ，人工介入。
> 3. **超时保护**：Chat Task 和 Memory Extraction 都有独立 `context.WithTimeout`，不会无限阻塞。
> 4. **启动自愈**：服务启动时异步连接 MQ，失败按间隔重试。MQ 后启动也能自动连上，不需要重启服务。
> 5. **优雅关闭**：SIGTERM → 标记 unready → drain in-flight HTTP 请求 → 关闭 MQ Consumer（等待当前消息处理完）→ 关闭连接。整个过程中的消息不丢。

### 问题 4："RabbitMQ 的拓扑结构是怎样的？能画一下吗？"

**标准回答**：

> 一个 Exchange（`opscaption.events`，type=direct），下面挂 6 个 Queue，分成两条链路：
>
> Chat Task 链路：主队列 `opscaption.chat.task`（prefetch=4）→ Retry Queue（`x-message-ttl=2s` + `x-dead-letter-exchange` 回流）→ DLQ（终态）
>
> Memory 链路：主队列 `opscaption.memory.extract`（prefetch=8）→ Retry Queue → DLQ
>
> Redis 作为任务状态机，Key 格式 `opscaptionai:chat_task:task:<uuid>`，TTL 24h，存储状态从 queued → running → succeeded/failed 的完整生命周期。

**延伸亮点**：

> 重试不靠代码 sleep，而是利用 RabbitMQ 原生死信机制。Retry Queue 声明时指定 `x-message-ttl` 和 `x-dead-letter-exchange`，消息在 Retry Queue 中存活 2s 后自动 dead-letter 回原 Exchange，路由到主队列重新消费。这样不占用 goroutine 资源，服务重启也不丢失重试状态。

### 问题 5："双链路设计的核心价值是什么？为什么要记忆抽取单独走一路？"

**标准回答**：

> 记忆抽取和 Chat Task 的 SLA 不同。Chat Task 是用户主动提交的分析请求，用户知道自己在等结果；记忆抽取是后台静默执行的，用户无感知。放在同一条链路会有两个问题：
>
> 1. **优先级冲突**：记忆抽取高频触发（每轮对话一次），但每次只花 1-2s。如果和 Chat Task 共享 Consumer，Chat Task 可能被大量记忆抽取消息阻塞。
> 2. **故障隔离**：Chat Task 挂了不应影响记忆能力，反之亦然。
>
> 所以我们设计了独立的 Consumer、独立的 prefetch（Memory=8, Chat=4）、独立的 Retry/DLQ。
>
> 更关键的是**双链路降级**：如果 RabbitMQ 完全挂了，记忆抽取不会"罢工"——`PersistOutcome` 中 `enqueueMemoryExtraction` 返回 `(false, nil)` 后，fallback 到本地 goroutine + semaphore（最多 8 个并发），用 `context.WithoutCancel` 脱离父请求生命周期，保证核心记忆能力不丢。

### 问题 6："优雅关闭的顺序为什么是先 HTTP 后 MQ？"

**标准回答**：

> 核心原则是**从外到内**——先停入口（HTTP），再停内部（MQ），确保在途数据不丢。
>
> 具体来说：HTTP 请求处理中可能还在调用 `PersistOutcome` → `enqueueMemoryExtraction` → `publish` MQ 消息。如果先关 MQ，publish 会失败，触发不必要的降级路径。先关 HTTP（drain in-flight 30s），确保所有请求完成后再关 MQ，整个过程零消息丢失。
>
> Consumer 关闭时也不是暴力断连：先 `cancel` consumer context 让 consumer loop 退出（当前 delivery 处理完才退出），等待 `consumeDone`（最多 1s），然后 ACK + 关闭 Channel + Connection。消息不会丢，因为用的是 `DeliveryMode: amqp.Persistent`。

## 9. 自测

### 问题 1

Chat Task 的 Consumer 执行失败后，消息是怎么回到主队列被重新消费的？描述完整路径。

<details>
<summary>点击查看答案</summary>

**不是代码 sleep 重试，而是利用 RabbitMQ 死信机制：**

1. Consumer 执行失败，且 `attempt < max_retries(3)`
2. 发布消息到 **Retry Queue**（routingKey: `chat.task.retry`）
3. Retry Queue 的声明参数中包含了：
   - `x-message-ttl: 2000` → 消息在 Retry Queue 中存活 2 秒
   - `x-dead-letter-exchange: opscaption.events` → 过期后发回原 Exchange
   - `x-dead-letter-routing-key: chat.task.request` → 用原 routing key
4. 2 秒后消息"过期"，RabbitMQ 自动将其 dead-letter 回主 Exchange，路由到主队列
5. Consumer 再次消费（attempt+1）

**为什么不用代码 sleep？**
- goroutine 阻塞浪费资源
- 服务重启丢失重试状态
- 死信机制是 RabbitMQ 原生能力，更可靠

</details>

### 问题 2

Memory Extraction 的 MQ 链路和本地 fallback 链路有什么区别？为什么需要两条？

<details>
<summary>点击查看答案</summary>

**两条链路的区别**：

| 维度 | MQ 链路 | 本地 fallback |
|------|--------|---------------|
| 触发条件 | MQ 客户端正常 | MQ 客户端为 nil |
| 执行方式 | RabbitMQ Consumer 消费 | 本地 goroutine + semaphore |
| 并发控制 | prefetch（默认 8） | semaphore（最多 8） |
| 超时控制 | `memory.extract_timeout_ms` | 同配置 |
| Context 隔离 | `context.Background()` | `context.WithoutCancel(parent)` |
| 可靠性 | MQ 持久化，重启不丢 | 进程重启丢失（但短期记忆已同步写入） |

**为什么需要两条？**

记忆抽取虽然只是"锦上添花"（不影响当前对话），但长期记忆是系统的核心能力。如果 MQ 挂了就完全不抽取记忆，用户几轮对话后就"失忆"了。双链路保证：
- MQ 正常 → 最可靠的异步链路
- MQ 异常 → 降级到本地 goroutine，不丢记忆能力

这是**降级优先（Degradation First）**设计哲学的体现。

</details>

### 问题 3

优雅关闭的顺序为什么是"先标记 unready → 再 drain HTTP → 最后关 MQ"？如果反过来会怎样？

<details>
<summary>点击查看答案</summary>

**正确顺序**：
```
SIGTERM → ① 标记 unready(/readyz→503) → ② drain HTTP 请求 → ③ 关 MQ Consumer → ④ 关 MQ 连接
```

**每一步的原因**：

- **先标记 unready**：让负载均衡/K8s 停止路由新流量到此实例，避免关闭过程中收到新请求
- **再 drain HTTP**：等待当前正在处理的 HTTP 请求完成（最多 30s），不粗暴断开
- **最后关 MQ**：HTTP 请求处理过程中可能还在发布 MQ 消息（如记忆抽取），过早关闭 MQ 会导致消息发布失败

**如果反过来（先关 MQ 再 drain HTTP）**：

```
SIGTERM → 关 MQ → drain HTTP
                  ↑
                  问题：HTTP 处理中的 PersistOutcome 尝试发布 MQ 消息
                  → MQ 已关闭 → 发布失败 → 降级到本地 goroutine
                  → 虽然不丢数据，但增加不必要的降级路径
```

**如果只关 HTTP 不关 MQ**：

Consumer 还在运行 → 正在处理的消息可能被强制中断 → 消息 requeue → 可能被其他实例消费（或丢失，取决于配置）

**一句话**：关闭顺序的本质是**从外到内**——先停入口（HTTP），再停内部（MQ），确保在途数据不丢。

</details>

### 问题 4

RabbitMQ 的完整拓扑中，Retry Queue 是怎么实现"延迟重试"的？画出关键参数。

<details>
<summary>点击查看答案</summary>

**关键代码**（`declareChatTaskTopology` 中的 Retry Queue 声明）：

```go
retryArgs := amqp.Table{
    "x-dead-letter-exchange":    cfg.Exchange,        // opscaption.events
    "x-dead-letter-routing-key": cfg.RoutingKey,      // chat.task.request
    "x-message-ttl":             int32(cfg.RetryDelay.Milliseconds()), // 2000ms
}
ch.QueueDeclare(cfg.RetryQueue, true, false, false, false, retryArgs)
ch.QueueBind(cfg.RetryQueue, cfg.RetryRoutingKey, cfg.Exchange, false, nil)
```

**工作流程**：

1. Consumer 处理失败 → publish 消息到 Retry Queue（routingKey: `chat.task.retry`）
2. 消息进入 Retry Queue，因为 Queue 声明了 `x-message-ttl=2000`，消息在 2s 后"过期"
3. 消息过期时，RabbitMQ 检查 Queue 的 `x-dead-letter-exchange` 参数 → 自动将消息 dead-letter 回 `opscaption.events`
4. 同时使用 `x-dead-letter-routing-key: chat.task.request` → 消息被路由回主队列 `opscaption.chat.task`
5. Consumer 再次消费（attempt 已 +1）

**为什么不直接用 `x-delayed-message` 插件？**
- 避免额外 RabbitMQ 插件依赖，降低部署复杂度
- TTL + DLX 是 RabbitMQ 标准能力，任何版本都支持
- 2s 延迟对 AIOps 场景足够

</details>

### 问题 5

为什么 Consumer 的 `Close()` 要等 `consumeDone`？如果不等会怎样？

<details>
<summary>点击查看答案</summary>

**当前实现**：

```go
func (c *rabbitMQChatTaskClient) Close(ctx context.Context) error {
    consumeCancel()  // 通知 consumer loop 退出
    select {
    case <-consumeDone:            // 等待 goroutine 正常退出
    case <-ctx.Done():
    case <-time.After(time.Second): // 兜底超时
    }
    // 然后才关闭 AMQP Channel 和 Connection
}
```

**如果不等 `consumeDone`**：

1. 直接关闭 AMQP Connection → consumer loop 中的 `<-deliveries` channel 被关闭
2. 当前正在执行的 `processEvent()` 可能刚处理到一半
3. AMQP Channel 关闭后无法 ACK → 消息会 **requeue**（因为 `autoAck=false`）
4. 消息被其他实例重复消费 → **可能造成重复执行**

**等 `consumeDone` 的作用**：确保当前 delivery 处理完（成功 ACK 或进入 retry/DLQ）后才释放 AMQP 资源，**零消息丢失**。
