# 第 14 章：防幻觉体系 — "让 AI 不说假话"

> **本章目标**：理解 AIOps 场景下幻觉的致命危害，掌握十四层防幻觉防线的设计与实现，能向面试官清晰解释"工具调用闭环"和"输出校验链路"的核心思想。

---

## 1. 白话理解：为什么 AIOps 的幻觉比普通 ChatBot 更危险？

### 1.1 一句话解释

**AIOps 的幻觉 = 普通 ChatBot 的幻觉 × 10 倍危害**。

普通 ChatBot 编造一个菜谱，用户最多做出来不好吃。AIOps 编造一个告警，运维人员可能凌晨 3 点起来处理一个不存在的故障。更可怕的是，AIOps 场景下工具返回的数据就是 Ground Truth——Prometheus 的指标是真实的、日志系统的内容是真实的。我们可以用工具结果来校验 LLM 输出，这是 AIOps 天然的防幻觉优势。

### 1.2 一个类比：你开了一家有 AI 诊断医生的医院

```
普通 AI 助手（通用 ChatBot）的幻觉：
  ❌ "红烧肉需要放 500 克糖" → 用户做出来太甜，但不会死人

AI 诊断医生（AIOps）的幻觉：
  ❌ "患者血压 180/120"（实际是 120/80）→ 护士立刻抢救，患者被折腾
  ❌ "患者心跳骤停"（实际正常）→ 全院紧急动员，浪费资源
  ❌ "患者血糖正常"（实际低血糖）→ 延误治疗，可能致死

OpsCaption 的防幻觉设计就是给 AI 诊断医生配了：
  ✅ 必须先做检查才能下诊断（工具调用闭环）
  ✅ 诊断必须引用检查报告数据（强制引用）
  ✅ 诊断结果由系统自动比对检查报告（输出校验）
  ✅ 发现不一致立刻报警（Contract 阻断）
```

### 1.3 幻觉类型分类

| 类型 | 示例 | 危害 |
|------|------|------|
| **编造数据** | "payment_service 的 P99 延迟是 2.3s"（实际未查询） | 运维人员基于虚假指标做决策 |
| **错误归因** | "告警来自 order_service"（实际来自 payment_service） | 排查方向完全错误 |
| **过时信息** | "当前有 5 个 firing 告警"（实际是 10 分钟前的数据） | 误判系统状态 |
| **无中生有** | "存在一个 connection pool exhausted 告警"（实际不存在） | 引发不必要的应急响应 |
| **过度自信** | "根因是数据库连接池满"（实际只是推测） | 运维人员当作确定结论处理 |
| **遗漏关键信息** | 工具返回了 3 个告警，只提了 1 个 | 遗漏真正的问题 |

## 2. 防线全景：十四层防线

```

  ▼

  │   └─ [硬] 工具失败 → 格式化字符串

  │
  ├─ [软] ③ System Prompt 铁律

  ▼

  │

  │   ├─ 密钥泄露
  │   ├─ 内部 IP

  │

  │   ├─ 指标来源校验（无来源 → warning）
  │   ├─ 未调用工具检测（运维问题无工具 → hallucination risk）
  │   └─ 置信度词检测（推测/可能 → info 日志）
  │
  ├─ [硬] ⑥ Contract 校验

  │
  ├─ [硬] ⑧ LLM 遗漏检测
  │   └─ 对比工具结果 vs 输出，找出遗漏的重要信息
  │
  ├─ [硬] ⑨ LLM 准确性校验
  │   └─ 检查输出中的声明是否与工具数据一致
  │
  ▼
用户看到的回答 + 可信度标签
```

## 3. 核心机制：工具调用闭环

### 3.1 为什么这是最强的防线？

**一句话：在系统层面阻断"没有数据就编造"的路径。**

```
传统方式：
  用户问 "payment_service 延迟怎么样？"
  → LLM 直接回答 "payment_service 延迟升高，建议排查"
  → ❌ 没有调用工具，数据是编造的

工具调用闭环：
  用户问 "payment_service 延迟怎么样？"
  → 系统检查：运维问题，必须调用工具
  → 调用 query_metrics 工具
  → 工具返回 "payment_service P99: 2300ms, 错误率: 5.2%"
  → LLM 基于真实数据回答
  → ✅ 数据来源可追溯
```

### 3.2 工具失败时怎么办？

**关键设计：返回格式化字符串，而非 Go error。**

```go
// 工具失败时返回格式化字符串
"[工具调用失败] query_metrics: connection timeout
请告知用户该工具返回了错误，不要用通用建议替代实际数据。"
```

**为什么不返回 Go error？**
- Go error 会触发 eino 框架的重试机制
- 格式化字符串让 LLM 明确知道失败，并决定如何处理
- LLM 看到"[工具调用失败]"后，必须告知用户，不能编造数据

### 3.3 Prompt 铁律

```markdown
## 工具使用铁律
- 你拥有查询 Prometheus 告警、搜索日志、检索内部知识库等工具。
- 运维相关问题必须先调用工具获取实际数据，再基于数据回答。
- 绝对不要说"我无法查询"、"我没有能力访问"。
- 回答要基于工具返回的真实数据，不要编造指标、日志或告警信息。

## 输出引用规则
- 每个结论必须关联一个工具返回的具体数据
- 格式：[来源: 工具名] 数据内容 → 结论
- 如果没有工具数据支持，必须标注"推测"或"待确认"

## 工具失败处理
- 工具返回错误时，必须告知用户具体的错误信息
- 不要用通用建议替代实际数据
```

## 4. 输出校验链路

### 4.1 三层校验架构

```
LLM 输出
  │
  ├─ 第一层：规则校验（快速，无延迟）
  │   ├─ 指标来源校验：正则提取数值指标，对比工具结果
  │   ├─ 未调用工具检测：运维问题无工具调用 → 幻觉风险
  │   └─ Contract 校验：5 条规则检查输出合规性
  │
  ├─ 第二层：结构校验（快速，无延迟）
  │   └─ Schema Gate：3 条规则检查输出质量
  │
  └─ 第三层：LLM 校验（慢，异步执行）
      ├─ 遗漏检测：对比工具结果 vs 输出，找出遗漏信息
      └─ 准确性校验：检查输出声明 vs 工具数据
```

### 4.2 Contract 校验（5 条规则）

| 规则 | 等级 | 说明 |
|------|------|------|
| must_use_tools | VIOLATION | 运维问题必须有工具调用 |
| must_report_failure | VIOLATION | 工具失败必须在输出中提及 |
| should_reference_data | WARN | 有工具结果时应引用数据 |
| no_fabricated_alerts | VIOLATION | 无工具数据时不得编造告警 |
| should_hedge | WARN | 无数据支持时应使用置信度标注 |

```go
// Contract 校验示例
result := events.ValidateContractWithConfig(
    hallucinationCfg,
    completeResponse,      // LLM 输出
    toolResults,           // 工具返回的结果
    failedTools,           // 失败的工具列表
    hasToolCalls,          // 是否有工具调用
)

if !result.Passed {
    // VIOLATION 级别：通知前端响应可能不可靠
    client.SendToClient("contract_violation", result.Summary())
}
```

### 4.3 Schema Gate（3 条规则）

| 字段 | 必须 | 说明 |
|------|------|------|
| has_answer | ✅ | 输出必须包含实质性回答（>10 字） |
| no_contradiction | ❌ | 输出不应自相矛盾（如"正常"+"异常"） |
| actionable | ❌ | 描述问题时应提供可操作建议 |

**Contract vs Schema Gate 的区别：**
- Contract 校验关注**事实准确性**——输出中的数据是否来自工具、是否有编造
- Schema Gate 关注**输出质量**——回答是否有实质内容、是否自相矛盾、是否有可操作建议
- 两者互补：Contract 防止"说假话"，Schema Gate 防止"说废话"

### 4.4 LLM 高级校验

复用 GLM-4.5-AIR 模型，新增两项 NLP 校验：

**遗漏信息检测：**
```
工具返回: payment_service 错误率 5.2%, order_service P99 3200ms
AI 回答: payment_service 需要关注
→ [遗漏] order_service 的 P99 延迟 3200ms 未被提及
```

**准确性校验：**
```
工具返回: P99 延迟 3200ms
AI 回答: P99 延迟是 2300ms
→ [问题] AI 称 P99 延迟为 2300ms，但工具数据显示 3200ms
```

## 5. 配置化设计

### 5.1 config.yaml 配置

```yaml
hallucination:
  enabled: true                          # 总开关
  contract_enabled: true                 # Contract 校验开关
  contract_block_on_violation: false     # Contract 阻断模式（默认关闭）
  schema_gate_enabled: true              # Schema Gate 开关
  output_validation_enabled: true        # 输出校验开关
  no_tool_call_detection_enabled: true   # 未调用工具检测开关
  ops_keywords:                          # 运维关键词列表
    - "告警"
    - "alert"
    - "prometheus"
    - "日志"
    - "log"
    # ... 更多关键词
  hedge_words:                           # 置信度标注词
    - "推测"
    - "可能"
    - "不确定"
    - "待确认"
  llm_validation:                        # LLM 校验配置
    enabled: false                       # 默认关闭（避免每次请求都调 LLM）
    model: "glm_chat_model_fast"
    timeout_ms: 5000
    max_tokens: 512
    omission_detection: true
    accuracy_check: true
```

### 5.2 为什么配置化？

**AGENTS.md 规范：配置项不硬编码，走 config.yaml。**

- 运维关键词可能随业务变化
- 置信度标注词可能需要调整
- LLM 校验可能需要根据误报率调整
- 阻断模式可能需要在确认误报率后开启

## 9. 面试回答

### Q1：你的 AIOps 系统怎么防幻觉？

> 我们有十四层防线，按从输入到输出的顺序：
>
> 第一层是**注入检测**，用正则拦截 prompt injection，这是硬拦截。
>
> 第二层是**工具调用闭环**。运维问题必须先调用工具获取数据，工具失败时返回格式化字符串（`[工具调用失败]`），让 LLM 明确知道失败。这是最强的防线——在系统层面阻断"没有数据就编造"的路径。
>
> 第三层是**Prompt 铁律**。明确告诉模型"你有工具，用它"、"不要编造指标"、"不确定时标注推测"。
>
> 第四层是 **Contract 校验**。5 条规则检查输出合规性：必须调工具、必须报告失败、必须引用数据、不得编造告警、无数据时标注推测。
>
> 第五层是 **Schema Gate**。结构化校验输出质量：必须有实质内容、不应自相矛盾、问题描述应有建议。
>
> 第六层是**输出指标校验**。正则提取输出中的数值指标，对比工具结果中是否有来源。
>
> 第七层是 **LLM 遗漏检测**。用 GLM 模型对比工具结果和输出，找出遗漏的重要信息。
>
> 第八层是 **LLM 准确性校验**。用 GLM 模型检查输出中的声明是否与工具数据一致。
>
> 后面还有 Memory 4 层验证、Context 预算管理、输出安全过滤、证据结构化等防线。

### Q2：工具调用闭环为什么是最强的防线？

> 因为它在系统层面阻断了"没有数据就编造"的路径。
>
> 传统的防幻觉靠 Prompt——告诉模型"不要编造"，但这是软约束，模型想违反就违反。
>
> 工具调用闭环是硬约束：运维问题必须先调用工具，工具失败时格式化字符串让 LLM 明确知道。LLM 看到"[工具调用失败]"后，必须告知用户，不能编造数据。
>
> 这就像交通规则和护栏的关系。Prompt 是交通规则（告诉你不能超速），工具闭环是护栏（你超速了会撞上护栏，不会冲下悬崖）。

### Q3：Contract 校验和 Schema Gate 有什么区别？

> Contract 校验关注**事实准确性**——输出中的数据是否来自工具、是否有编造。
>
> Schema Gate 关注**输出质量**——回答是否有实质内容、是否自相矛盾、是否有可操作建议。
>
> 两者互补：Contract 防止"说假话"，Schema Gate 防止"说废话"。

### Q4：这些校验会不会影响响应速度？

> 所有校验都在响应完成后异步执行，不阻塞用户看到回答。
>
> 校验结果只记录日志（warning/info），不修改输出。这是"观测不干预"的设计——先收集数据，确认误报率后再考虑是否阻断。
>
> 如果未来误报率低到可以接受，可以把 Contract 的 VIOLATION 级别升级为阻断——在输出前拦截，返回"数据不足，无法给出可靠回答"。

### Q5：LLM 校验的性能影响大吗？

> LLM 校验默认关闭，只在需要时开启。
>
> 开启后，每次请求会额外调用一次 GLM-4.5-AIR 模型，耗时约 1-2 秒。但这是异步执行的，不阻塞用户看到回答。
>
> 我们设置了 5 秒超时，如果 LLM 响应超时，会跳过校验，不影响正常流程。

## 10. 关键设计决策

| 决策 | 原因 |
|------|------|
| 工具失败返回格式化字符串而非 Go error | Go error 会触发 eino 重试，格式化字符串让 LLM 决定如何处理 |
| 校验在响应完成后异步执行 | 不阻塞用户看到回答 |
| 校验结果只记录日志 | "观测不干预"，先收集数据确认误报率 |
| 所有配置走 config.yaml | AGENTS.md 规范：配置项不硬编码 |
| LLM 校验默认关闭 | 避免每次请求都调 LLM，增加成本 |
| Contract 阻断模式默认关闭 | 需要确认误报率后再开启 |
