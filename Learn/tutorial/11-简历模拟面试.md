## 第二部分：实习经历深挖

### 智元创新 — GPU 自动化治理

### Q2："你做的 GPU 自动化治理系统——说清楚：空跑任务怎么识别？识别出来怎么处置？"

**🔑 你答：**

> 问题背景：Genie Studio 是多租户的具身智能开发平台，GPU 集群接了多个资源池。有的用户提交了训练任务但代码 bug 卡死——GPU 占用但利用率是 0%；有的是跑完了但没释放——占着 GPU 不下线。人工排查成本很高。
>
> 识别分两层：
> 1. **规则层**：GPU 利用率连续 10 分钟 < 10% + 显存占用 > 80%（说明代码挂了但没释放显存）→ 判为空跑。纯 CPU 任务占着 GPU 节点 → 判为错配。
> 2. **工作流层**：我把整个治理流程编排成工作流——识别 → 分级（通知/强杀）→ 回收 → 验证。不同资源池、不同任务类型走不同的处置策略。
>
> 处置不是一刀切杀——先通知用户（飞书消息："你的任务 GPU 利用率已低于 10% 持续 15 分钟"），给 5 分钟宽限期。超时无响应再强制回收。回收也不是直接销毁——先 checkpoint 保存当前状态，再释放 GPU。

**⚔️ 追问 2.1：** "你说分级——什么情况下直接强杀，什么情况只通知？"

> **答：** 判断逻辑基于两个维度：
> - 任务类型：训练任务可以通知+宽限期（丢了进度损失大）；推理任务强杀影响小（无状态，重拉就行）
> - 资源池压力：如果资源池 wait queue > 5 个任务在排队，宽松度降低——宽限期从 15 分钟缩短到 5 分钟。因为有人在等着用 GPU，不能让空跑任务占着。

**⚔️ 追问 2.2：** "工作流编排用的什么？怎么处理工作流失败？"

> **答：** 用的是平台自有的工作流引擎（类似 Argo Workflows 的思路）。每个步骤有独立超时和重试策略。关键是识别步骤和处置步骤的耦合——识别失败（Prometheus 挂了拿不到 GPU 指标）不应该阻塞已经识别出的任务的处置。我做了**部分失败容忍**：哪个 step 失败就跳过那个 step 的任务，其他的继续走。

**⚔️ 追问 2.3：** "你做这个系统最大的坑是什么？"

> **答：** 最初我把利用率阈值设成一个全局常量——GPU 利用率 < 5% 就判空跑。结果发现不同模型的训练特性差异巨大：大模型分布式训练有通信间隙，利用率天然波动；小模型可能长时间 100%。全局阈值导致大模型训练被误杀。
>
> 修正方案：引入**任务类型感知的动态阈值**——根据任务提交时声明的 model_type 和 framework，自动调整阈值和观察窗口。后来还加了历史基线的概念——如果一个任务历史上利用率稳定在 80%，突然降到 40%，比数字本身更能说明异常。

### 觅蜂 — RLinf 云边通信优化

### Q3："你发现 TCP 乱序 18%——怎么发现的？当时的排查过程是什么样的？"

**🔑 你答：**

> 问题背景：RLinf 是做具身智能后训练的框架，需要云数据中心和边缘节点协同训练。我们发现训练吞吐量异常低——预期 1 Gbps，实际只有 423 Mbps。
>
> 排查过程：
> 1. **先看监控**：用 Grafana 看了网络吞吐，确认瓶颈在云→边的下行方向。
> 2. **tcpdump 抓包**：在边缘节点上 `tcpdump -i eth0 -w dump.pcap host <云 IP>`，抓了 5 分钟的包。
> 3. **用 Wireshark 分析**：Statistics → IO Graph 看吞吐曲线，发现波动异常大。然后看 TCP 分析 → Expert Info，发现大量的 "Out-of-Order" 和 "Retransmission" 标记。
> 4. **量化问题**：`tshark -r dump.pcap -Y "tcp.analysis.out_of_order" | wc -l` 除以总包数，算出 18% 的乱序率。正常值是 < 2%。
> 5. **定位根因**：跟运维确认——云到边的链路经过一个 SD-WAN 设备，该设备在多路径负载均衡时没有保证 TCP 流的包顺序，导致同一个 TCP 连接的包走了不同路径、到达顺序乱了。

**⚔️ 追问 3.1：** "为什么选 BBR 而不是继续调 Cubic 的参数？BBR 在高乱序环境下好在哪？"

> **答：** Cubic 的核心假设是 "丢包 = 拥塞"。但在高乱序网络里，很多包并没有丢——只是到达顺序错了，触发了 TCP 的 fast retransmit。Cubic 看到 "丢包" 就猛砍 cwnd（乘性减），但实际上网络带宽是空闲的——它误判了。
>
> BBR 不看丢包，看的是 **RTT（往返延迟）和 Bottleneck Bandwidth（瓶颈带宽）**。乱序不会造成 RTT 变大——包只是到了顺序不对，不是到得慢。所以 BBR 不会因为乱序而错误地降速。这就是为什么切换到 BBR 后，吞吐从 423 Mbps 翻到 907 Mbps——BBR 填满了实际可用的带宽，没有被假 "丢包" 骗。
>
> 调 Cubic 参数（比如调大 ssthresh）也能缓解，但治标不治本——只要底层链路还在乱序，Cubic 的丢包驱动逻辑就会周期性误判。换 BBR 是根因修复。

**⚔️ 追问 3.2：** "吞吐从 423 到 907 Mbps 提升了一倍多——但你有没有验证 907 Mbps 是不是已经接近链路极限了？"

> **答：** 验证了。用 iperf3 打了纯 TCP 吞吐测试——极限约 940 Mbps。907 Mbps 已经达到了极限的 96%，说明 BBR 基本榨干了链路。剩下那 3-4% 的 gap 可能是协议头和 ACK 的开销，这是 TCP 本身的物理极限。

**⚔️ 追问 3.3：** "XOR 差分 + 字节位面重排 + ZSTD——讲一下为什么这个组合能达到 70% 压缩率？普通 ZSTD 做不到吗？"

> **答：** 浮点数序列有特殊的数据特征——相邻的浮点数（比如传感器相邻帧的关节角度）数值接近，IEEE 754 编码在位模式上差异很小。但普通 ZSTD 是按字节流压缩的——它对 "相邻字节的相似性" 敏感，但对 "跨 float32 边界的位相似性" 不敏感。
>
> 三步管线：
> 1. **XOR 差分**：把 `[3.141, 3.142, 3.140]` 变成 `[3.141, 0.001, 0.002]`——后面绝大部分位变成 0，ZSTD 最喜欢连续为零的字节。
> 2. **字节位面重排**：把所有 float32 的第 0 个 byte 放在一起、第 1 个 byte 放在一起... 这样 ZSTD 看到的是一大片相似字节（高位字节几乎全是符号位+指数位，很相似），而不是交错的不同字节。
> 3. **ZSTD**：在前面两步的预处理后，ZSTD 能高效压缩。
>
> 纯 ZSTD 大约 40-50% 压缩率。加上 XOR + 位面重排后到 70%。代价是编解码多了几步，但都是线性时间的位操作，延迟很小。

## 第三部分：OpsCaptionAI 项目深挖

### Q4："你说 Top-10 召回率 78%——这个 78% 是 Recall@10 还是 NDCG@10？评测集怎么划分的？"

**🔑 你答：**

> 是 **Recall@10**——10 个候选文档里至少包含一个 ground truth 相关文档的比例。我们的评测集是 AIOps Challenge 2025 的 50+ 个故障案例，每个案例标注了正确答案引用的文档 ID。
>
> 划分上严格分开：按案例做 **build/holdout split**（70/30），不是按文档随机切——因为同一个系统（如 checkoutservice）的文档在 build 和 holdout 里同时出现会导致 LLM "背答案"，评测失真。

**⚔️ 追问 4.1：** "78% 你觉得满意吗？如果让你做到 90%，你会做什么？"

> **答：** 78% 是基线，不算优秀。做到 90% 我会从三个方向发力：
> 1. **Chunking 策略**：当前的分块偏 Markdown 标题切分，对 JSON/YAML 格式的配置文档不友好。改成语义分块（用 embedding 算相似度断点），能让更相关的片段被检索到。
> 2. **Query Expansion**：HyDE（Hypothetical Document Embeddings）——不是直接用用户 query 检索，而是先让 LLM 生成一个假设的答案文档，用这个假设文档做向量检索。运维场景特别适合——用户的 query 很短，但相关文档很长。
> 3. **Fine-tuned Embedding**：当前用通用的 Doubao Embedding。如果针对 AIOps 领域的 query-document 对做对比学习微调，Recall 应该能再提一截。
>
> 但这三个改进的成本也是递增的——HyDE 每次检索要多一次 LLM 调用，微调需要标注数据。先做 Chunking 优化，低成本高收益。

### Q5："你的 Token Budget 控制——如果四种来源（history/memory/docs/tools）同时都很重要，你怎么取舍？"

**🔑 你答：**

> ContextEngine 给每种来源分配了预算上限（config 里配置的），但它不做 "等比分"——有优先级：
>
> 1. **System Prompt（不计入 budget）** ——这是 Agent 的 "人格"，不能砍
> 2. **当前用户 query** ——最高优先级，不压缩
> 3. **RAG 文档（docs）** ——按 Rerank 分数截断，只取 top-N 篇且每篇截取 relevant_excerpt
> 4. **Tool outputs** ——最新的优先，超限部分做摘要而非丢弃
> 5. **History** ——按轮次截断，最近 N 轮完整保留，更早的做 LLM 摘要压缩
> 6. **Long-term memory** ——只选最相关的 K 条（根据 embedding 相似度）
>
> 冲突时：docs > history > memory。因为文档是 "证据"，历史和记忆是 "上下文"——面试同理，证据的优先级高于聊天记录。

**⚔️ 追问 5.1：** "你这个优先级是硬编码还是可配置的？如果不同场景需要不同的优先级呢？"

> **答：** 当前是 **ContextProfile** 驱动的。Profile 是配置文件里的一组策略——比如 "diagnosis" profile 给 docs 分配更多预算，"chat" profile 给 history 更多预算。Agent 启动时按场景选 Profile，优先级跟着变。
>
> 但预算具体数字目前还是人调的。理想状态是自动调参——根据任务的 "可回答度" 做反馈，如果 docs 不够才解析失败，下次自动扩大 docs 预算。这个还没做。

### Q6："你用了两种 Agent 模式（ReAct + Plan-Execute-Replan），路由选择还是关键词匹配——为什么不直接用 LLM 做路由判断？"

**🔑 你答：**

> 诚实地说——关键词匹配是当前版本的权宜之计，不是最佳方案。我知道它的局限——"分析一下这个日志" 触发 Plan-Execute 太重，"为什么最近一周支付系统不稳定" 按关键词会漏判。
>
> 更好的方案是 **LLM Router** ——请求进来后做一次极轻 LLM 调用（P50 延迟 < 500ms），判断问题复杂度。复杂度判据是结构化 Prompt 驱动的：涉及几个数据源？需不需要跨时间对比？需不需要根因推理？三个维度交叉判断，比关键词准确得多。
>
> 但 LLM Router 也有成本——每次请求多一次 LLM 调用。我在等 Go 编译的本地小模型（如 Ollama + Qwen 1.5B）成熟——用本地小模型做路由，没有 API 成本，延迟也低。

**⚔️ 追问 6.1：** "你说本地小模型——但运维场景的语义理解 Qwen 1.5B 能做吗？"

> **答：** 做完整推理不行。但做复杂度判断够——它只需要回答 "1-5 的复杂度打分"，不是生成诊断报告。Prompt 是高度结构化的选择题，不是开放生成。小模型在选择题上准确率和 7B 差距不大。核心风险是漏判复杂度（把复杂问题判成简单）——我的兜底是 ReAct 超时有 25 步上限，即使走错了模式也不会无限循环。

### Q7："你的 RAG 有四种检索模式——说一下四种模式各自适用什么场景？"

**🔑 你答：**

> 1. **Dense（纯向量检索）**：适合语义相似但关键词不重叠的查询——用户说 "服务挂了"，文档写 "pod OOMKilled"。向量能跨过术语鸿沟。
> 2. **Sparse（BM25 关键词检索）**：适合精确匹配——用户问 "MySQL max_connections 默认值是多少"，文档里正好有这句话。BM25 比向量准。
> 3. **Hybrid（Dense + Sparse 融合）**：通用场景默认模式。用 RRF（Reciprocal Rank Fusion）融合两路结果——不是加权，是按排名取倒数，自动平衡两路的贡献。
4. **Keyword（纯规则匹配）**：适合有明确实体 ID 的查询——"告警 ALRT-2025-0427-001 的详情"。不需要语义检索，用 ID 直接查更快更准。（**注：当前版本尚未实现独立的 Keyword 检索模式，作为后续优化方向**）

> 当前版本默认走 Hybrid 模式。未来方向是增加 Keyword 模式和让 LLM Router 自动选择检索模式。

### Q7.5："你做了 Agent Contract 系统和知识索引管线——讲讲这两个新能力的价值"

**🔑 你答：**

> **Agent Contract 系统**是 Harness MVP 的核心。问题是：Agent 的行为边界没有"验收标准"。metrics agent 可能把指标告警当成日志证据，reporter 可能把推断写成确定根因。Contract 系统给每个 Agent（triage/metrics/logs/knowledge/reporter）定义正式合约：Role/Responsibilities/Inputs/Outputs/Must/MustNot/EvidencePolicy。运行时通过 `EnforceContract()` 做 Schema Gate 校验——Status 为 degraded 时必须给 DegradationReason，Status 为 succeeded 时 Summary 不能空。不通过的输出自动降级，Confidence ×0.5。
>
> **知识索引管线**解决了一个运维常见问题：知识库文档怎么入向量库？之前是离线脚本跑的，现在有了知识索引 Agent——文件上传 → 解析（Markdown/JSON/CSV）→ 分块（按结构边界）→ 向量化 → 写入 Milvus。用户直接通过前端上传文件，后端自动走完整条管线，不需要运维工程师写 Python 脚本。

**⚔️ 追问 7.5.1："Contract 的 EvidencePolicy 为什么不放在 Prompt 里，而是单独做了一个结构化系统？"**

> **答：** 两个原因。**第一，可验证性**——Prompt 是"建议"，Contract 是"校验"。reporter 的 Prompt 里写"不要把知识库建议包装成已发生事实"，LLM 可能遵守，也可能不遵守。但 Contract 的 EnforceContract 是代码层面的强制检查——它在 reporter 返回结果后，检查是否引用了不可靠的证据来源。这是确定性的，不依赖 LLM 的"理解"。
>
> **第二，版本管理**——Contract 有版本号（Version: 2026-04-21），每次修改可以追溯。Prompt 的修改很难做 diff 和 blame。当团队扩展到多人后，Contract 的版本化管理比 Prompt 管理更工程化。

## 第四部分：专业技能深挖

### Q8："你写了'熟练 Golang，理解 GMP 调度模型'——说一下 GMP 的 M 是什么时候创建、什么时候回收的？"

**🔑 你答：**

> M 代表 Machine（OS 线程）。创建时机：当有空闲的 P 但没 M 来绑定它的时候——通常是 G 的数量增长导致需要更多 OS 线程来并行执行。Go runtime 的 `sysmon` 监控线程会检查。
>
> 回收时机：M 在两种情况下会进入休眠并被回收。一是 M 绑定的 P 的本地队列空了，M 去全局队列 steal work 也失败——M 进入 spinning 状态，如果持续空转一段时间（没有新 G 进来），M 会休眠，OS 线程被回收。二是 M 执行了阻塞系统调用（如文件 IO）——Go runtime 会把 M 和 P 解绑，P 移交另一个 M，当前 M 带着阻塞中的 G 等待系统调用返回，返回后将 G 放回队列，M 休眠。
>
> 和传统线程池的最大区别：Go 的 M 数量是动态的，且上限是 `GOMAXPROCS` 的某种倍数（默认 10000，但受 OS 限制）。不是固定大小的线程池。

**⚔️ 追问 8.1：** "如果 M 在执行系统调用时阻塞，绑在它上面的 P 怎么办？"

> **答：** Go runtime 的 `handoffp` 机制——当 M 即将进入阻塞系统调用时，runtime 会把 M 上的 P 剥离，交给另一个空闲 M（或新建 M）。这样 P 的队列里的其他 G 不会因为一个阻塞而被饿死。这就是 Go 的 "work stealing" 和 "handoff" 协同：M 是执行资源的池，P 是调度上下文——两者解耦，阻塞不拖累同 P 的其他 G。

**⚔️ 追问 8.2：** "你的项目里用到了 goroutine——最多同时跑了多少个？有 goroutine 泄漏的排查经验吗？"

> **答：** OpsCaption 的 goroutine 来源：HTTP Handler（GoFrame 每个请求一个）、MQ Consumer（常驻 1-2 个）、LLM 流式 channel reader、记忆异步抽取。正常运行时约 15-25 个。
>
> 排查泄漏的经验来自 GPU 治理项目——Prometheus 采集指标时忘记关 HTTP response body，每次采集泄漏一个 goroutine。发现方式是 `pprof.Lookup("goroutine").WriteTo()` + `go tool pprof` ——看到 goroutine 数量持续增长，stack trace 指向 `net/http.(*persistConn).readLoop`。修复就是加 `defer resp.Body.Close()`。
>
> 现在养成了习惯——所有长期运行的 Go 服务都嵌入了 pprof endpoint。

### Q9："你写了'熟悉 RabbitMQ 和 Kafka'——说一下两者的消息可靠性模型有什么本质区别？"

**🔑 你答：**

> 本质区别在于 **"消息所有权"的语义**：
>
> **RabbitMQ** 是 **消息级别的确认（ACK）**。Consumer 拉取一条消息 → 处理 → 显式 ACK → Broker 才删除。如果 Consumer 挂了没 ACK，消息自动重新入队给另一个 Consumer。配合 TTL + DLX（死信交换），天然支持 "失败 → 延迟 → 重试" 的语义。缺点是吞吐量受 ACK 开销限制，大约每秒 1-5 万条。
>
> **Kafka** 是 **偏移量级别的确认（Offset Commit）**。Consumer 按 Partition 顺序消费，处理到 offset N 后提交 N 给 Broker。但即使 Consumer 挂了，消息不会重入队——另一个 Consumer 从上次 commit 的位置继续。这意味着如果 Consumer 在处理消息后、commit 前崩溃，消息会被重复处理。Kafka 保证的是 "at-least-once" 且要求消费幂等。
>
> 所以：RabbitMQ 适合**任务队列**（每条消息处理一次、失败重试、不丢消息）。Kafka 适合**事件流**（高吞吐、顺序消费、允许重复处理）。我的项目两个都用了——RabbitMQ 做异步任务队列，Kafka 对接日志流。

**⚔️ 追问 9.1：** "RabbitMQ 的 ACK 模式下，如果 Consumer 处理成功了但网络断了 ACK 没发出去——消息是丢了还是重复了？"

> **答：** 重复了。RabbitMQ 的保证是 "at-least-once"——如果 Broker 没收到 ACK（不管是 Consumer 真挂了还是网络断），它会重新投递消息给其他 Consumer。所以 Consumer 必须做**幂等处理**。
>
> 在 OpsCaption 里，幂等靠两样：Redis 记录消息 ID 的处理状态（SETNX message_id:processed）；任务级别的状态机（queued→running→succeeded）——如果消息重投但状态机已经是 succeeded，Consumer 直接 ACK 并跳过。

### Q10："你做了浮点数压缩管线——如果让你设计一个通用的 AI 训练数据压缩方案，你会怎么设计？"

**🔑 你答：**

> 分层设计，按数据类型选策略：
>
> 1. **Float32 序列（传感器、关节角度）**：XOR 差分 + 位面重排 + ZSTD——这是我已验证的管线。70% 压缩率，无损。
> 2. **图像数据（RGB 帧）**：不上无损——无损对像素太低效。用 JPEG 有损编码（quality=85），在肉眼无感知的前提下 90%+ 压缩率。关键是选 quality 参数要跑 PSNR/SSIM 基准测试，不能拍脑袋。
> 3. **文本/日志**：ZSTD 即可——日志天然有大量重复模板（"[INFO] connection from 10.0.x.x"），ZSTD 的字典模式能直接吃掉。不需要预处理。
> 4. **Embedding 向量**：这是特殊的数据——float16 量化（从 float32→float16）基本不掉召回，压缩率直接 50%。如果需要更高，用 PQ（Product Quantization）做有损压缩。
>
> 核心原则两条：① 按数据类型选编码策略，不要一刀切；② 有损压缩必须跑评测——PSNR/Recall @ K 不掉才算通过。

### Q11："你说'熟练使用 Cursor、Codex 等 AI 开发工具'——能不能具体举一个 AI 辅助帮你解决了一个你一个人搞不定的问题的例子？"

> 这个建议让我少改了大量代码——原本的方案要在 assembler 内部到处插日志，decorator 方案只需要一个 wrapper。这让我意识到 AI 不只是省打字时间，它有时能给一个你没想到过的架构模式。

**⚔️ 追问 11.1：** "有没有 AI 工具帮了倒忙的情况？"
