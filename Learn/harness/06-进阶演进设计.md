# 进阶演进设计 — 第三优先级

> 当前统一口径（2026-05）
> - 本文记录的是历史 Harness / 多智能体演进想法。
> - 当前主实现依然以 `Chat ReAct` 与 `AIOps Runtime + Plan-Execute-Replan` 为准，文中出现的 `specialist / triage / reporter` 不应再被当成当前聊天主链路。

> 目标：系统从"稳定可控"进入"自进化、规模化"阶段

---

## 总览

P2 阶段解决的是系统成熟后的高阶问题：

| 能力 | 解决的问题 | 触发条件 |
|---|---|---|
| 自动化 Correction Pipeline | 每次犯错都要人手动写规则 | AGENTS.md 规则数 > 30 |
| 成本优化 | Token 消耗随 Agent 数量线性增长 | 月 LLM 成本 > 预算线 |
| 分布式 A2A | 单体 runtime 无法独立扩缩容 | specialist 需要独立部署 |
| 组织级 Governance | 多团队协作时的规范一致性 | 项目 > 3 人维护 |

---

## 1. 自动化 Correction Pipeline

### 1.1 目标

当前流程：
```
Agent 犯错 → 人发现 → 人分析 → 人写规则 → 人更新 AGENTS.md
```

半自动化后：
```
Agent 犯错 → 系统检测 → 自动提取上下文 → 生成规则草稿 → 人审核确认 → 自动并入 AGENTS.md
```

### 1.2 触发条件

当 `TaskResult.Status == failed` 或 `TaskResult.Status == degraded` 时，且满足以下条件之一：
- 同一 agent 同一 error code 在 24 小时内出现 ≥ 2 次
- supervisor 重试后仍然失败
- Tool 返回 unexpected error（非网络/超时类）

### 1.3 产出格式

```yaml
# .hermes/corrections/2026-05-01-logs-agent-mcp-timeout.md
---
error_code: "MCP_TIMEOUT"
agent: "logs"
tool: "query_log"
occurrences: 3
first_seen: "2026-04-29T10:15:00Z"
last_seen: "2026-04-29T14:22:00Z"
suggested_rule_type: "must_not"
---

## 现象
Logs Agent 调用 MCP 日志工具时频繁超时（3 次/24h）

## 根因分析
MCP 服务端在高峰时段响应超过 30s，Agent 无超时感知

## 建议规则（草稿）
- 日志工具调用设置 15s 超时
- 超时后不重试，直接返回 degraded
- 超时信息在 reporter 中透明展示

## 建议补充的测试
- TestLogsAgent_MCPTimeout: mock MCP 超时 → 预期 degraded + reason 含 "MCP timeout"
```

### 1.4 实现路径

```
internal/ai/service/correction_candidate.go  ← 失败检测 + 上下文提取
internal/ai/cmd/correction_cmd/main.go       ← CLI 工具生成规则草稿
.hermes/corrections/                          ← 规则草稿目录
```

### 1.5 为什么放在 P2

当前 AGENTS.md 规则数偏少（20+ 条），人工维护成本可接受。等规则数 > 30 且新增频率稳定后，自动化才有 ROI。

## 2. 成本优化

### 2.1 当前成本模型

```
一次 AIOps 请求:
  triage:     GLM fast (轻)     ~500 tokens
  planner:    GLM (重)          ~2000 tokens
  specialists: GLM/GLM fast × 3 ~3000 tokens
  reporter:   GLM (重)          ~1500 tokens
  rerank:     GLM fast          ~1000 tokens
  query rewrite: GLM fast       ~500 tokens
  ─────────────────────────────────────────
  总计:                          ~8500 tokens/request
```

### 2.2 优化策略

| 策略 | 预期节省 | 实现难度 |
|---|---|---|
| **Tool 输出缓存**：相同 tool+input 在 1h 内复用 | 30% | 低（client 端加 TTL map） |
| **Context budget 差异化**：planner 给更多，triage 给更少 | 15% | 低（改 config） |
| **Triage 降级为规则匹配**：高频意图不走 LLM | 10% | 中（规则表已经存在） |
| **Reporter 摘要裁剪**：长 evidence 只传 snippet，完整内容放 artifact | 20% | 中（已有 artifact 机制） |
| **Rerank 结果缓存**：相同 query+doc 组合 30min 内复用 | 10% | 低 |

### 2.3 实现路径

```yaml
# manifest/config/config.yaml
harness:
  cost:
    tool_cache_ttl_sec: 3600     # tool 输出缓存
    rerank_cache_ttl_sec: 1800   # rerank 结果缓存
    triage_rule_first: false     # true: 优先规则匹配，miss 再走 LLM
```

```
internal/ai/cache/tool_cache.go    ← tool 输出缓存（map + TTL）
internal/ai/cache/rerank_cache.go  ← rerank 结果缓存
```

### 2.4 为什么放在 P2

当前项目处于功能建设阶段，用户量不大，成本不是瓶颈。优先把功能做对，再做成本优化。

## 3. 分布式 A2A（Agent-to-Agent）

### 3.1 当前架构

```
┌──────────────────────────────────────────┐
│              单体进程                      │
│  Supervisor → Triage → Specialists        │
│                    → Metrics Agent        │
│                    → Logs Agent           │
│                    → Knowledge Agent      │
│                    → Reporter             │
│  全部在同一 Go 进程中                       │
└──────────────────────────────────────────┘
```

### 3.2 目标架构

```
┌──────────┐   gRPC    ┌─────────────────┐
│Orchestrator│ ←─────→ │ Metrics Service │
│ (Supervisor)│         └─────────────────┘
│            │  gRPC    ┌─────────────────┐
│            │ ←─────→ │ Logs Service    │
│            │         └─────────────────┘
│            │  gRPC    ┌─────────────────┐
│            │ ←─────→ │Knowledge Service│
└──────────┘         └─────────────────┘
       │
       │ NATS/Redis Streams
       ▼
┌──────────┐
│  Event   │  ← TaskEvent 异步通知
│  Bus     │
└──────────┘
```

### 3.3 何时需要

| 信号 | 说明 |
|---|---|
| Metrics Agent 需要独立扩展 | 查询 Prometheus 集群时内存/CPU 需求远超其他 Agent |
| Logs Agent 需要不同运行时 | 日志分析需要 Python 生态（pandas/numpy）而主进程是 Go |
| 不同 Agent 有不同部署节奏 | Knowledge Agent 频繁更新 RAG pipeline，不想影响其他 Agent |
| 组织拆分 | 不同团队维护不同 Agent |

### 3.4 实现路径

```
internal/ai/transport/           ← 新增：A2A 传输层
  grpc_server.go                 ← gRPC 服务端
  grpc_client.go                 ← gRPC 客户端
  types.proto                    ← protobuf 定义（基于现有 protocol/types.go）
internal/ai/transport/nats/      ← 事件总线
  publisher.go
  subscriber.go
```

### 3.5 为什么放在 P2

当前单体 runtime 完全够用。拆分微服务会显著增加运维复杂度（服务发现、负载均衡、分布式追踪），ROI 在当前阶段为负。P2 作为远期架构方向保留。

## 4. 组织级 Harness Governance

### 4.1 目标

当项目有多人维护时，建立统一的 Harness 规范：

| 规范 | 内容 |
|---|---|
| Agent 命名规范 | `<domain>_<role>` 例如 `metrics_analyst`、`logs_investigator` |
| Tool 所有权 | 每个 tool 有明确 owner，修改需要 review |
| Schema 版本化 | TaskResult/TaskEnvelope 大版本号，breaking change 有 migration 计划 |
| Review Checklist | 新增 Agent → 必须补 contract + replay + docs |
| 配置变更流程 | `manifest/config/config.yaml` 修改需要 PR + 至少 1 人 review |

### 4.2 实现路径

```
docs/policies/
  agent-naming.md           ← Agent 命名规范
  tool-ownership.md         ← 工具所有权清单
  schema-versioning.md      ← Schema 版本策略
  review-checklist.md       ← PR review 检查项
```

### 4.3 为什么放在 P2

当前项目维护者数量少（≤ 2 人），非正式沟通成本低。团队扩大到 3+ 人后再建立正式 governance。

---

## 5. 优先级排序

```
1. 成本优化（ROI 最高，实现最简单）
2. Correction Pipeline（Agent 犯错频率上升后启动）
3. 组织级 Governance（团队 > 3 人后启动）
4. 分布式 A2A（独立扩展需求出现后启动）
```

---

## 6. 启动条件总结

| 能力 | 启动条件 |
|---|---|
| Correction Pipeline | AGENTS.md 规则 > 30 且月新增 > 3 |
| 成本优化 | 月 LLM token 消耗 > 目标预算 |
| Distributed A2A | specialist 需要独立扩展/部署 |
| Governance | 维护者 > 3 人 |
