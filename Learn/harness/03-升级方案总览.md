# Harness Engineering 升级方案总览

> 当前统一口径（2026-05）
> - 本文对应的是历史 Harness 升级规划，不再描述当前聊天主链路。
> - 当前实现请以 `Chat ReAct` 与 `AIOps Runtime + Plan-Execute-Replan` 为准。

## 1. 升级目标

基于 [02-当前状态审计](./02-当前状态审计.md) 的结论，本次升级聚焦三个目标：

1. **补上最短板**：验证层从 ⚠️ 提升到 ✅ — 让系统输出从"看起来对"变成"可验证地对"
2. **加固运维面**：统一配置、建立文档体系、扩充 replay 用例 — 让系统从"能跑"到"好维护"
3. **建立进化能力**：半自动化纠偏闭环 — 让系统从"被动修复"到"主动进化"

---

## 2. 优先级矩阵

```
影响力
 高 │  contract tests  │  replay 用例    │  auto-correction
    │  schema gate     │  har布配置统一  │  cost optimization
    │                  │  docs 目录      │  distributed A2A
 中 │                  │                │
    │                  │                │
 低 │                  │                │
    └──────────────────┴───────────────┴─────────────→ 复杂度
          🔴 第一优先级      🟡 第二优先级    🟢 第三优先级
```

---

## 3. 三级优先级详解

### 🔴 第一优先级：验证闭环（预计 5 天）

**为什么排第一**：验证层是当前最薄弱的环节。没有验证，其他 Harness 层都建立在"信任模型输出"的基础上。

| ID | 任务 | 产出 | 预计工时 |
|---|---|---|---|
| V-1 | 工具 contract tests | `tools/contract_test.go` | 1.5 天 |
| V-2 | TaskResult schema gate | `protocol/validate.go` | 1 天 |
| V-3 | Contract enforce 运行时 | `contracts/enforce.go` | 1 天 |
| V-4 | 集成到 CI | workflow 项 | 0.5 天 |
| V-5 | 文档 | 本文档体系 | 1 天 |

**验收标准**：
- `go test ./internal/ai/tools/...` 包含 3+ 个 contract test
- `go test ./internal/ai/agent/contracts/...` 包含 enforce test
- CI 中 `go test ./...` 自动执行上述测试
- 任一 specialist 返回不合法 TaskResult 时 runtime 能检测并降级

**风险**：低。新增测试，不改生产代码逻辑。

---

### 🟡 第二优先级：运维加固（预计 5 天）

**为什么排第二**：当前系统缺乏统一的运维配置入口，文档散落，replay 覆盖不足。

| ID | 任务 | 产出 | 预计工时 |
|---|---|---|---|
| O-1 | har布统一配置 | `config.yaml` 新增 `harness` 段 | 0.5 天 |
| O-2 | retry/max_iterations 接入 | 所有 Agent 读取配置 | 1.5 天 |
| O-3 | docs 目录建立 | `docs/architecture|agents|tools|runbooks` | 1 天 |
| O-4 | replay 用例扩充 | 5+ 条 supervisor replay | 1.5 天 |
| O-5 | 集成验证 | CI + 文档索引更新 | 0.5 天 |

**验收标准**：
- `manifest/config/config.yaml` 包含 `harness` 配置段
- `max_iterations` / `task_timeout_ms` / `retry_budget` 在所有 Agent 中生效
- `docs/` 目录存在且 AGENTS.md 索引了它
- 5+ 条 replay case 全部通过

**风险**：低。配置增加和 replay 用例是纯增量，不改现有逻辑。

---

### 🟢 第三优先级：进阶演进（可后续排期）

| ID | 任务 | 说明 |
|---|---|---|
| A-1 | 自动化 Correction Pipeline | 失败→半自动规则草稿提取 |
| A-2 | 成本优化 | tool output 缓存、context budget 差异化 |
| A-3 | 分布式 A2A | specialist 独立部署 |
| A-4 | 组织级 Governance | agent naming / schema versioning / review checklist |

## 5. 不改什么

本次升级严格遵循 **先增量、不改核心逻辑** 的原则，以下不动：

- ❌ 不改 `supervisor/triage/specialists/reporter` 的核心编排逻辑
- ❌ 不改 LLM 调用链
- ❌ 不改 RAG 检索和 rerank 链路
- ❌ 不改 Memory Service 实现
- ❌ 不改 Approval Gate 机制
- ❌ 不引入新的外部依赖

改动范围严格限制在：
- 新增测试文件（`_test.go`）
- 新增少量工具函数（`validate.go`, `enforce.go`）
- 配置文件新增字段（`config.yaml`）
- 新增目录和文档（`docs/`）
- 新增 replay 测试用例

## 6. 风险与边界

### 风险

| 风险 | 概率 | 缓解 |
|---|---|---|
| contract tests 过于脆弱，轻微改动就失败 | 中 | 验证关键字段即可，不对完整输出做严格匹配 |
| replay 用例依赖外部服务（Prometheus/MCP） | 高 | mock 外部依赖，只验证编排逻辑 |
| har布配置引入后旧路径不兼容 | 低 | 新增字段都有默认值，不设则不生效 |

### 边界

- 本次不做 P2 的自动化 Correction Pipeline
- 本次不做 P2 的分布式 A2A
- 本次不做成本优化（那是模型选择层的事）
- 本次不改 `.golangci.yml`（P1 范围外的独立任务）

---

## 7. 验收清单

升级完成后逐项检查：

### 验证闭环
- [ ] `go test ./internal/ai/tools/...` 通过，包含 contract tests
- [ ] `go test ./internal/ai/agent/contracts/...` 通过，包含 enforce tests
- [ ] 修改 specialist 输出格式 → contract test 失败 → 阻止不合规输出
- [ ] CI 流水线自动执行验证

### 运维加固
- [ ] `harness` 配置段可读、有默认值
- [ ] Agent 读取 `max_iterations` / `task_timeout_ms` / `retry_budget`
- [ ] `docs/` 目录结构完整
- [ ] 5+ 条 replay case 通过

### 整体
- [ ] `go build ./...` 通过
- [ ] `go test ./...` 通过（新增测试不破坏已有测试）
- [ ] AGENTS.md 已更新文档索引
