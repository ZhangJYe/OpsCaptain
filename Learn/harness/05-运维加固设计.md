# 运维加固设计 — 第二优先级

> 目标：让 Harness 系统从"能跑"到"好配置、好查文档、好回归测试"

---

## 1. 问题现状

1. **配置散落**：`timeout`、`max_iterations` 在各处 hardcode 或各自读取，没有一个统一的 Harness 配置入口
2. **文档不完整**：`docs/` 目录不存在，新人/Agent 没有结构化的项目文档入口
3. **Replay 覆盖不足**：只有 1 条 supervisor replay test，覆盖不到降级、超时、空结果等关键路径

---

## 2. 设计原则

- **统一入口**：所有 Harness 相关配置收敛到 `manifest/config/config.yaml` 的 `harness` 段
- **向后兼容**：每个配置项都有合理默认值，不设也能跑
- **文档驱动**：文档从 AGENTS.md 提炼，保持一致性
- **Replay 可重放**：所有新增 replay case 必须可独立执行，不依赖外部服务

## 4. 详细设计

### 4.1 Harness 统一配置

`manifest/config/config.yaml` 新增：

```yaml
harness:
  max_iterations: 10          # Agent 最大迭代次数
  task_timeout_ms: 300000     # 单任务超时 (5min)
  retry_budget: 3             # 失败重试次数
  fail_fast: false            # true: 一个 specialist 失败则终止全部
  context_budget_tokens: 8192 # 单 agent 上下文 token 上限
  validation:                 # 验证层配置（参见 04-验证闭环设计）
    enabled: true
    contract_enforce: true
    strict_mode: false
```

**默认值策略**：

| 配置项 | 默认值 | 说明 |
|---|---|---|
| `max_iterations` | 10 | Agent 循环/工具调用上限 |
| `task_timeout_ms` | 300000 | 5 分钟，覆盖大部分 AIOps 场景 |
| `retry_budget` | 3 | 单 specialist 最多重试 3 次 |
| `fail_fast` | false | 默认允许其他 specialist 继续执行 |
| `context_budget_tokens` | 8192 | 8K tokens，够放历史+文档+工具输出 |
| `validation.enabled` | true | 默认开启 schema gate |
| `validation.contract_enforce` | true | 默认开启 contract enforce |
| `validation.strict_mode` | false | 默认降级而非拒绝 |

**读取方式**：通过 `g.Cfg().Get(ctx, "harness.max_iterations")` / `g.Cfg().MustGet(ctx, ...)` 。

Runtime 启动时加载一次，注入到各个 Agent 的 task envelope 的 `Constraints` 字段。

### 4.3 Replay 用例扩充

**新增文件**：`internal/ai/agent/supervisor/replay_cases_test.go`

**设计原则**：
- 每个 case 是一个独立测试函数
- Mock 所有外部依赖（Prometheus/MCP/Milvus）
- 只验证编排逻辑（哪个 specialist 被调用、结果如何聚合）
- 支持 `t.Parallel()` 并行执行

**5 个新增 case**：

```
Case 1: 正常全链路
  输入: "payment service 响应变慢"
  预期: triage → [metrics, logs, knowledge] 并行 → reporter 聚合 → status=succeeded

Case 2: MCP 日志工具不可用
  输入: "checkoutservice 报错"
  预设: logs tool 返回 degraded
  预期: logs specialist → degraded, 其他继续 → reporter 标记 "log evidence unavailable"

Case 3: 文档命中为空
  输入: "unknown_error_code_999"
  预设: knowledge retriever 返回空
  预期: knowledge specialist → degraded (no docs found), 不编造内容

Case 4: 多证据冲突
  输入: "adservice CPU 高但无报错日志"
  预设: metrics → CPU 告警, logs → 无异常, knowledge → SOP 指向 CPU 排查
  预期: reporter 说明 "metrics 发现异常，logs 未发现，建议按 SOP 排查"

Case 5: Specialist 超时
  输入: "全集群健康检查"
  预设: metrics agent 模拟超时
  预期: metrics → degraded (timeout), 其他完成, reporter 说明 metrics 超时
```

**测试辅助**（已在项目中存在的 infrastructure）：
- `runtime_test.go` 已有 mock 能力可复用
- `internal/ai/agent/specialists/*/agent_test.go` 可作为 mock 参考
- `eval/` 下的 baseline 体系可提供样本数据

## 5. 验收标准

### 配置
```bash
# 验证配置可正确读取
go test ./internal/ai/runtime/ -run TestHarnessConfig -v
```

### 文档
- `docs/` 目录包含所有 5 个 agent 文档 + 2 个工具文档 + 1 个 runbook
- AGENTS.md 索引了 docs/ 目录
- 每个文档内容从已有代码/contracts 提取，不凭空创造

### Replay
```bash
# 5+ 条 replay case 全部通过
go test ./internal/ai/agent/supervisor/ -run TestReplay -v -count=1
```

---

## 6. 不做什么

- ❌ 不重写现有任何 Agent 实现
- ❌ 不修改 Runtime 的核心调度逻辑
- ❌ 不在 config.yaml 中删除或重命名现有配置项
- ❌ 不新增外部依赖（replay 测试用标准库 mock）
- ❌ 不修改 CI 配置（go test ./... 自然覆盖新测试）
