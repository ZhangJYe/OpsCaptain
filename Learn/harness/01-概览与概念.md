# Harness Engineering 概览与概念

## 1. 一句话定义

> Harness Engineering = 给 AI Agent 系统套一层工程化"外壳"，用确定性约束去管理非确定性的模型行为。

它不是某个 SDK、平台或框架，而是一种**工程方法**。

类比：
- Agent 系统是发动机——产生驱动力（模型推理、工具调用、任务执行）
- Harness 是底盘、刹车、仪表盘和安全带——确保发动机不会把车开进沟里

---

## 2. 它解决什么问题

Harness Engineering 不解决"如何让模型输出更聪明"，它解决的是：

| 问题 | 没有 Harness | 有 Harness |
|---|---|---|
| Agent 调用错误工具 | 不知道、无法阻止 | 工具白名单拒绝 + 记录 |
| 上下文膨胀导致幻觉 | 查不出来为什么 | 按 agent 角色控制 budget，可追踪每条注入 |
| Multi-Agent 协作失败 | 不知道哪个环节出错 | trace_id 全链路追踪，精确定位 |
| 输出不可靠 | 只能人工判断 | schema 校验 + replay 测试自动验证 |
| 同一个错误反复犯 | 没人记得上次怎么修的 | 失败→补规则→补测试 的反馈循环 |

---

## 3. 核心六层模型

```
┌──────────────────────────────────────────────────┐
│                  Harness 六层模型                    │
├────────────┬─────────────────────────────────────┤
│  协议层     │ TaskEnvelope / TaskResult / Event    │
│  Protocol   │ 统一消息结构、生命周期、序列化           │
├────────────┼─────────────────────────────────────┤
│  边界层     │ tool registry / per-agent capability │
│  Boundary   │ Must / MustNot / Input / Output       │
├────────────┼─────────────────────────────────────┤
│  上下文层    │ ContextAssembler / Budget / Resolver │
│  Context    │ 按角色装配、按预算裁剪、按引用传递       │
├────────────┼─────────────────────────────────────┤
│  验证层     │ schema gate / contract test / replay │
│  Verification│ 输入输出契约、回归测试、结构化校验       │
├────────────┼─────────────────────────────────────┤
│  观测层     │ trace_id / task_id / session_id       │
│  Observability│ 全链路追踪、性能度量、错误定位          │
├────────────┼─────────────────────────────────────┤
│  纠偏层     │ AGENTS.md / correction pipeline       │
│  Correction │ 失败→规则→测试→文档 的闭环              │
└────────────┴─────────────────────────────────────┘
```

### 3.1 协议层 (Protocol)

**职责**：定义 Agent 之间、Agent 与 Runtime 之间的消息格式。

关键结构：
- `TaskEnvelope`：任务信封，包含 task_id、parent_task_id、session_id、trace_id、assignee、deadline
- `TaskResult`：任务结果，包含 status（succeeded/failed/degraded）、summary、evidence、degradation_reason
- `TaskEvent`：生命周期事件，task_started、tool_called、task_completed 等
- `ArtifactRef`：中间产物引用，不传大文本，只传 ID+URI

**为什么重要**：没有统一协议就没有统一编排。每个 Agent 说不同"方言"，Supervisor 无法调度。

### 3.2 边界层 (Boundary)

**职责**：限制每个 Agent 的能力范围。

关键机制：
- 每个 Agent 声明自己的 `Inputs`（能接收什么）和 `Outputs`（能产出什么）
- 每个 Agent 声明 `Must`（必须遵守）和 `MustNot`（禁止行为）
- Tool 按 Agent 白名单分配，Agent 不能调用不属于自己的工具

**为什么重要**：
- Metrics Agent 不应该尝试查日志
- Logs Agent 不应该尝试改数据库
- 边界清晰 → 权限可控 → 审计可追溯

### 3.3 上下文层 (Context)

**职责**：不把所有信息塞给每个 Agent，而是按角色、按预算、按引用传递。

关键机制：
- `ContextAssembler`：根据 Agent 角色和任务类型，选择性地装配 history/memory/docs/tool outputs
- `ContextBudget`：限制每个 Agent 的上下文 token 上限
- `ContextTrace`：记录每次装配了什么、为什么装配

**为什么重要**：上下文是 Agent 系统的第一生产要素，也是第一失控点。无节制的上下文注入 = 幻觉放大器。

### 3.4 验证层 (Verification)

**职责**：不靠"看起来对"，靠"可验证地对"。

关键机制：
- **Contract Tests**：每个工具的输入输出契约测试（给定输入→期望输出格式）
- **Schema Gate**：运行时检查 TaskResult 是否满足 Agent 声明的 Outputs 契约
- **Replay Tests**：用历史真实 case 回放完整链路，对比结果

**为什么重要**：这是当前 OpsCaption 项目最薄弱的一层。没有验证的 Agent = 每次改动都在赌。

### 3.5 观测层 (Observability)

**职责**：出问题时知道"哪一个环节出了什么错"。

关键字段：
- `session_id` — 用户会话
- `trace_id` — 一次完整请求链路
- `task_id` — 一个具体任务
- `parent_task_id` — 父子任务关系
- `agent_name` / `tool_name` / `latency_ms` / `status`

**为什么重要**：Multi-Agent 系统中，一次用户请求可能产生 10+ 个子任务。没有 trace，故障定位 = 大海捞针。

### 3.6 纠偏层 (Correction)

**职责**：失败不是终点，是系统进化的起点。

核心循环：
```
Agent 犯错 → 人类分析 → 更新 AGENTS.md 规则 → 补测试 → Agent 下次不犯
```

**为什么重要**：这是 Harness Engineering 的灵魂。没有这个循环，其他五层只是"防御"，纠偏层是"进化"。

## 4. Harness Engineering vs Multi-Agent

这两个概念常被混淆，但它们解决的问题完全不同：

| | Multi-Agent | Harness Engineering |
|---|---|---|
| 核心问题 | 怎么拆角色、怎么协作 | 怎么让协作可控、可调试、可验证 |
| 关注点 | 任务分工 | 系统约束 |
| 产出 | Supervisor/Triage/Specialists | 协议/边界/验证/观测/纠偏 |
| 目标 | 扩展性 | 生产性 |

关系：**正交且互补**。

- 没有 Harness 的 Multi-Agent = 一群没有交通规则的司机
- 只有 Harness 没有 Agent 拆分 = 空有交通规则但没车在跑

---

## 5. 为什么最近 Harness 在 Agent 圈越来越热

因为大家集体从 Demo 阶段进入了生产阶段：

1. Demo 很容易，生产很难
2. 单 Agent + 2 个工具 → 可控；Multi-Agent + 20 个工具 → 复杂度爆炸
3. 模型能力越强，越需要约束（能力越大，搞砸的规模越大）
4. 企业场景要求可审计、可回溯、可灰度，这些不是 prompt 能解决的

---

## 6. Harness 的投入产出比判断

### 适合引入 Harness 的信号

- ✅ 已经不是纯聊天，涉及工具调用
- ✅ 有 RAG、知识库检索
- ✅ 有多轮上下文，context 在增长
- ✅ 正在或即将引入 Multi-Agent
- ✅ 对输出正确性有严格要求（运维场景）
- ✅ 已经踩过"同一个错反复犯"的坑

### 不适合引入 Harness 的信号

- ❌ 只是简单的单轮问答
- ❌ 没有工具调用
- ❌ 不需要多角色协作
- ❌ 输出没有正确性要求

**OpsCaption 当前状态**：全部 ✅，属于 Harness 高价值场景。

---

## 7. Harness 的实施原则

1. **先业务后框架**：先在 AI Ops 链路落地，不要全面铺开
2. **先单体内再分布式**：进程内 runtime 足够，先不拆微服务
3. **先约束再自治**：先让 Agent 受控，再追求更高自主性
4. **先验证再扩容**：没有验证的 Agent 扩张 = 放大风险
5. **简单场景保持简单**：普通对话链路不重构成重型 Multi-Agent

---

## 8. 一句话总结

> Multi-Agent 负责"分工"，Harness Engineering 负责"控场"。没有分工，系统没扩展性。没有控场，系统没生产性。
