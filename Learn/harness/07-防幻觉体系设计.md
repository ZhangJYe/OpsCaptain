# AIOps 防幻觉体系设计

> OpsCaption 项目如何在高准确性场景下防止大模型产生幻觉

---

## 1. 为什么 AIOps 对防幻觉要求极高

AIOps 场景的特殊性：

| 维度 | 普通聊天 | AIOps |
|---|---|---|
| 错误代价 | 用户体验差 | 误判告警→错误操作→生产事故 |
| 信息真实性 | AI 可以发挥 | 必须基于实时系统状态 |
| 可审计性 | 不需要 | 必须可追溯到具体日志/告警/文档 |
| 不确定时的行为 | 可以猜 | 必须明确说"不知道" |

OpsCaption 的核心原则：

> **宁可说"我不知道"（degraded），也不说"可能是"（幻觉）。**

---

## 2. 六层纵深防御架构

```
用户
 │
 ├─ 第一层：Prompt 指令层 ─────── 直接禁止编造
 ├─ 第二层：Contract 契约层 ───── 每个 Agent 有 MustNot 禁令
 ├─ 第三层：证据驱动层 ─────────── 所有结论必须有 EvidenceItem
 ├─ 第四层：降级而非编造 ──────── 没证据→degraded，不硬编
 ├─ 第五层：上下文预算 ─────────── 防止上下文过载诱发幻觉
 └─ 第六层：运维纠偏层 ─────────── 出过一次的幻觉不会再出
```

---

## 3. 第一层：Prompt 指令层

**文件**：`internal/ai/agent/chat_pipeline/prompt.go:177`

**策略**：在系统 prompt 中直接禁止编造，且**点名具体类目**。

```
不要编造文档内容、接口行为、配置项、日志结果或外部事实
```

这不是泛泛的"请准确回答"，而是根据运维场景中最高频的 5 个幻觉来源做的针对性约束。同时还要求：

```
如果信息不足，明确指出缺少哪些关键事实、配置、日志、报错或上下文
```

**设计原理**：对于特定领域的幻觉问题，模糊的"不要说谎"不如精确的"不要在以下五个方面编造"有效。模型对具体指令的遵守度远高于泛化指令。

---

## 4. 第二层：Contract 契约层

**文件**：`internal/ai/agent/contracts/contracts.go`

**策略**：每个 Agent 有独立的 MustNot 禁令，防止角色越界导致幻觉。

| Agent | 关键 MustNot | 防的幻觉类型 |
|---|---|---|
| **Triage** | 不要读取工具或外部文档 | 路由阶段自行查证据 |
| **Metrics** | 不要把指标告警推断成日志证据 | 跨域推断 |
| **Logs** | 不要把 raw output 伪装成已验证结论 | 未验证事实 |
| **Knowledge** | 不要在无文档命中时编造 SOP 内容 | 空检索→胡编 |
| **Reporter** | 不要新增 specialist 没有提供的新事实 | 聚合阶段无中生有 |

**最关键的是 Reporter 的 MustNot**：

```
不要新增 specialist 没有提供的新事实
不要把不确定推断写成确定根因
```

这意味着即便前面所有 Agent 都没拦住幻觉，Reporter 作为最后一道门也不会把幻觉传播给用户——它只能聚合已有 evidence，不能创造新内容。

**设计原理**：Multi-Agent 系统中，幻觉防控不能只在入口做，必须在每个角色的能力和禁止项中显式约束。这就是 contract 体系的价值——不是一个大 prompt 管所有，而是每个角色有自己的行为边界。

---

## 5. 第三层：证据驱动层

**文件**：`internal/ai/protocol/types.go:74`

**策略**：所有结论必须有结构化的 `EvidenceItem`，包含来源追踪。

```go
type EvidenceItem struct {
    SourceType string  // prometheus_alert / mcp_log / internal_doc
    SourceID   string  // 具体告警名/日志文件/文档路径
    Title      string  // 证据标题
    Snippet    string  // 证据片段
    Score      float64 // 检索置信度
    URI        string  // 可追溯链接
}
```

**证据来源必须来自真实工具调用**：

| Specialist | 工具 | Evidence SourceType |
|---|---|---|
| Metrics | `query_prometheus_alerts` | `prometheus_alert` |
| Logs | MCP 日志工具 | `mcp_log` |
| Knowledge | Milvus RAG | `internal_doc` |

**证据链完整闭环**：

```
用户问题 → Triage 路由 → Specialist 调用工具 → 工具返回真实数据
    → Specialist 封装为 EvidenceItem → Reporter 聚合 evidence
    → 用户看到有据可查的结论
```

没有工具调用 → 没有 EvidenceItem → 没有可呈现的结论。

**设计原理**：这是防幻觉的"硬约束"层。不是靠 prompt 劝说模型说实话，而是架构层面要求输出必须有来源。SourceType + SourceID 的设计使得每个事实都可以追溯到"哪个工具的哪次返回"。

## 6. 第四层：降级而非编造

**文件**：`internal/ai/protocol/types.go` / `internal/ai/service/degradation.go` / `internal/ai/runtime/runtime.go`

**策略**：当证据不足时，系统选择降级（degraded）而不是编造。

### 6.1 三态结果模型

```go
const (
    ResultStatusSucceeded ResultStatus = "succeeded"  // 有证据，正常输出
    ResultStatusFailed    ResultStatus = "failed"     // 明确失败
    ResultStatusDegraded  ResultStatus = "degraded"   // 降级：证据不足但不编造
)
```

没有模糊的 "partial" 或 "uncertain"。`Degraded` 是明确的工程决策："我有能力但条件不满足"。

### 6.2 DegradationReason 强制透明

`runtime.go:414` 有自动补全机制：

```go
if result.Status == ResultStatusDegraded && result.DegradationReason == "" {
    result.DegradationReason = deriveDegradationReason(result)
}
```

任何降级都必须携带原因。用户看到的不是"我不知道"，而是：
- "日志证据缺失，分析仅基于告警和知识库"
- "Prometheus 查询超时，指标数据不可用"
- "未找到相关文档，建议人工排查"

### 6.3 全局 Kill Switch

`degradation.go` 提供两级紧急开关：

| 级别 | 方式 | 生效速度 |
|---|---|---|
| Config | `degradation.kill_switch: true` | 下次部署 |
| Redis | `SET opscaptionai:degradation:kill_switch true` | 即时 |

当发现模型版本或工具产生幻觉时，可以**秒级关闭所有 Agent 推理**，所有请求统一返回 degraded 消息。这是生产环境防幻觉的最后防线。

### 6.4 Specialist 级别的降级

每个 Specialist 在以下情况主动降级：

| 场景 | Agent 行为 |
|---|---|
| Tool 返回错误 | `status=degraded, reason="tool error: ..."` |
| Tool 返回空结果 | `status=degraded, reason="no data found"` |
| Tool 超时 | `status=degraded, reason="timeout after 15s"` |
| 权限不足 | `status=degraded, reason="permission denied"` |

**设计原理**：LLM 的一个核心缺陷是——当它不知道答案时，它倾向于"猜一个看起来合理的"。降级机制通过在系统层面提供"不确定"的合法出口，从根本上消除了"被迫编造"的场景。让 LLM 知道"说不知道也是正确答案"。

## 7. 第五层：上下文预算

**文件**：`internal/ai/contextengine/`

**策略**：精细的 token 预算控制，防止上下文过载诱发幻觉。

### 7.1 为什么上下文过载导致幻觉

当一个 Agent 的上下文塞入：
- 20 条相似文档 + 15 轮历史 + 大量工具输出 + 系统 prompt

模型会产生"注意力稀释"——把文档 A 的配置项安到文档 B 的服务上，把历史的错误信息当成当前的实时数据。

### 7.2 预算控制模型

```go
type ContextBudget struct {
    MaxTotalTokens int  // 总上限
    SystemTokens   int  // 系统 prompt
    HistoryTokens  int  // 对话历史
    MemoryTokens   int  // 长期记忆
    DocumentTokens int  // RAG 文档
    ToolTokens     int  // 工具输出
    ReservedTokens int  // 预留给模型回复
}
```

每种上下文来源有独立上限，超出的部分不会被注入，并标记 `DroppedReason`。

### 7.3 Memory 置信度过滤

```go
MinMemoryConfidence float64  // 低于此值的记忆不注入
```

不靠谱的记忆（低置信度）直接过滤掉，不给模型制造混淆的机会。

### 7.4 按 Agent 角色裁剪

不同 Agent 有不同的上下文需求：

| Agent | History | Memory | Docs | Tool Output |
|---|---|---|---|---|
| Triage | 不需要 | 不需要 | 不需要 | 不需要 |
| Specialist | 需要 | 需要 | 需要 | 不需要 |
| Reporter | 需要 | 不需要 | 不需要 | 需要 |

**设计原理**：Triage 只需要原始 query 做路由判断，给它注入 memory 反而会污染路由决策（AGENTS.md §24.1 已作为规则固化）。按角色裁剪上下文 = 减少无关信息 = 减少幻觉诱因。

## 8. 第六层：运维纠偏层

**文件**：`AGENTS.md`

**策略**：每次出现幻觉 → 分析根因 → 沉淀为规则 → Agent 下次不再犯。

### 8.1 来自真实事件的防幻觉规则

AGENTS.md 中的规则不是凭空设计的，每条都对应一个真实踩过的坑：

```
- 不要把 groundtruth 标签直接当实时证据喂给模型
- 不要把历史复盘标签当实时日志证据  
- 不要把 raw output 伪装成已结构化验证的结论
- 不要在无文档命中时编造 SOP 内容
- 不要把不确定推断写成确定根因
- 不要把历史标签和实时证据混在同一层，必须分层
```

### 8.2 纠偏循环

```
Agent 产生幻觉 → 人类发现 → 分析幻觉类型
    → 更新 AGENTS.md / contracts MustNot
    → 补充 replay test（确认修复有效）
    → Agent 加载新规则 → 同类幻觉不再出现
```

**设计原理**：防幻觉不是一次性设计完的，是一个持续进化的过程。纠偏层的价值在于将每次事故转化为系统免疫力——同样的幻觉不会犯两次。

## 9. 防幻觉策略分层总结

```
┌─────────────────────────────────────────────────────────────┐

├─────┬──────────────────┬────────────┬────────────────────────┤
│ 层  │ 机制             │ 约束类型   │ 代表文件                │
├─────┼──────────────────┼────────────┼────────────────────────┤
│  1  │ Prompt 指令      │ 软约束     │ prompt.go              │
│  2  │ Contract MustNot │ 软约束+文档│ contracts.go           │
│  3  │ EvidenceItem     │ 硬约束     │ protocol/types.go      │
│  4  │ Degraded 降级    │ 硬约束     │ degradation.go         │
│  5  │ Context Budget   │ 硬约束     │ contextengine/         │
│  6  │ AGENTS.md 规则    │ 反馈循环   │ AGENTS.md              │
├─────┼──────────────────┼────────────┼────────────────────────┤
│ ★   │ Schema Gate      │ 硬约束     │ (04-验证闭环设计中)      │

└─────┴──────────────────┴────────────┴────────────────────────┘

★ = 本次升级计划补上的能力
```

---

## 10. 一句话总结

> **OpsCaption 的防幻觉策略是：用 EvidenceItem 确保每句话都有据可查，用 Degraded 给模型"不知道"的合法出口，用 Contract MustNot 给每个角色画清边界，用 Context Budget 防止信息过载诱发幻觉，用 AGENTS.md 确保同样的幻觉不犯第二次。**
