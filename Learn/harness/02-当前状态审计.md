## 二、边界层 (Boundary)

### 2.1 Agent Contract 注册表

**状态**：✅ 已完成

**代码路径**：`internal/ai/agent/contracts/contracts.go`（309 行）

**注册的 Agent**：triage, metrics, logs, knowledge, reporter

**每个 Contract 包含**：
- Role（一句话定位）
- Responsibilities（职责范围）
- Inputs / Outputs（接口契约）
- Must（必须遵守的行为）
- MustNot（禁止行为）
- EvidencePolicy（证据策略）

**优点**：
- 契约式设计，每个 Agent 有明确边界
- `AttachMetadata()` 函数将 contract 信息注入 TaskResult
- `Render()` 函数可将 contract 渲染为 Agent prompt

**不足**：
- Contract 目前是**文档性质**，没有运行时 enforce
- 没有工具级的 contract（tool → 允许的 agent 映射不存在）

### 2.2 Tool 白名单机制

**状态**：⚠️ 部分完成

Agent 层面有约束（contracts 定义了 Must/MustNot），但工具层面没有显式的 "tool → allowed agents" 映射表。

当前实际情况：
- Metrics Agent 内 hardcode 了 `query_metrics_alerts`
- Logs Agent 内 hardcode 了日志查询工具
- Knowledge Agent 内 hardcode 了 `query_internal_docs`

问题：如果 Agent prompt 被注入诱导调用了不该调的工具，没有第二层防护。

## 四、验证层 (Verification) — 🔴 最薄弱层

### 4.1 Contract Tests（工具契约测试）

**状态**：❌ 未开始

搜索 `contract.*test`、`_test.go` 文件：contracts 包本身无测试文件。

核心工具（`query_log`, `query_prometheus_alerts`, `query_internal_docs`）没有独立的契约测试。

风险：修改工具实现后，无法自动验证是否破坏了 Agent 对工具的期望。

### 4.2 Schema Gate（运行时结果校验）

**状态**：❌ 未开始

`TaskResult` 有结构定义，但没有运行时校验逻辑。

Contracts 定义了每个 Agent 的 Outputs（如 triage 应输出 intent/domains/priority），但代码中不检查实际 TaskResult 是否满足这些契约。

### 4.3 Replay Tests

**状态**：⚠️ 部分完成

- Supervisor replay test：`internal/ai/agent/supervisor/supervisor_replay_test.go` — 1 条
- RAG eval baseline：`internal/ai/rag/eval/` — 有 AIOps baseline 评测体系

**不足**：
- 只有 1 条 supervisor replay，覆盖不到完整链路
- 缺少：MCP 不可用、文档为空、多证据冲突、超时等场景

## 八、关键发现总结

### 做得好的（保持）

1. **协议层设计成熟**：TaskEnvelope/TaskResult/TaskEvent 结构完整，甚至有 `deadline_at` 字段
2. **Contract 体系有价值**：Must/MustNot 约束清晰，但需从文档变为运行时 enforce
3. **观测层覆盖全面**：trace_id 在所有关键链路中传递
4. **AGENTS.md 有实战价值**：每条规则对应真实失败案例

### 需要补的（本次升级重点）

1. **🔴 验证层几乎空白**：没有 contract test，没有 schema gate
2. **🟡 运维面不完整**：缺少统一配置、repo docs、replay 用例不足
3. **🟢 纠偏闭环半手动**：失败→规则仍是纯人工流程
