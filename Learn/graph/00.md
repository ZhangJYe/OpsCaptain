# AIOps 故障案例知识图谱与 Graph-Enhanced RAG 设计（内部 RCA 产品化）

## 1. 文档定位

本文档面向 `D:\Agent\OpsCaption\aiopschallenge2025` 数据集，目标不是比赛提分，也不是商业化交付，而是为 **内部 RCA 产品化** 设计一套可落地、可审计、可演进的知识图谱与 RAG 协同方案。

这里的“产品化”指的是：

- 让系统能够服务日常故障排查
- 让历史案例、原始证据、结构化关系进入统一检索链路
- 让回答可追溯到证据，而不是只给一个像答案的答案
- 让后续评测、回放、优化基于同一套事实层推进

这里的“非商业”指的是：

- 该数据集可用于内部研究、学习、验证和产品原型
- 但要保留来源与许可边界，避免把数据集内容直接包装成商业收费能力

一句话概括：

> 这份设计的目标，是把这批 AIOps 挑战赛数据改造成内部 RCA 的“历史故障案例知识底座”，而不是把它当成一批普通文档塞进向量库。

---

## 2. 目标与非目标

### 2.1 目标

1. 建立一套可复用的故障案例事实层
2. 建立一套面向 RCA 的轻量知识图谱，而不是重型通用图谱
3. 让图谱服务 RAG：辅助召回、过滤、排序、解释
4. 让系统输出能区分“原始证据”“派生判断”“历史结论”
5. 让后续优化有明确数据边界、构图边界和评估边界

### 2.2 非目标

1. 不做百科式运维知识图谱
2. 不把所有日志、指标、trace 原始行直接向量化
3. 不在第一阶段引入复杂多跳 Agent 图推理框架
4. 不要求第一阶段必须依赖 Neo4j 之类的重型图库才能运行
5. 不把 challenge 的评测标签直接伪装成“实时观测证据”

---

## 3. 核心判断

### 3.1 这批数据适合做图谱，但不是“原样入库”

这批数据天然具备结构化案例属性：

- 有明确 case 标识
- 有明确时间窗口
- 有明确故障对象
- 有日志、指标、trace 三类观测
- 有历史标签与结论

因此它适合抽象成：

- 实体
- 关系
- 时间窗口
- 证据链
- 历史案例标签

但它 **不适合** 直接按文档 RAG 的思路原样塞进向量库。原因很直接：

- 原始 parquet 太大、太噪
- 时序信号不适合按固定长度切块
- 单行日志或单个 span 的语义检索价值有限
- 直接向量化会把噪声和重复观测一起带进召回链路

正确路线是：

1. 先抽出事实层
2. 再构图
3. 再生成 serving 文档
4. 再与向量检索协同

### 3.2 内部 RCA 产品化可以使用历史标签，但必须分层使用

这点和“比赛评测”不同。

如果目标是内部 RCA 产品，`groundtruth.jsonl` 里的历史标签可以作为“已解决案例”的一部分进入知识库。因为在内部产品里，历史故障结论本来就是知识资产。

但必须严格区分三类信息：

- `原始证据`：日志、指标、trace、时间窗口、对象
- `结构化事实`：从原始证据归一化后的 case / observation / service-call / topology facts
- `历史结论`：fault_type、fault_category、fault_description、key_metrics 等标签与结案信息

这三层不能混成一个字段，更不能在回答时把“历史标签”伪装成“当前实时证据”。

### 3.3 图谱应该增强 RAG，而不是替代 RAG

第一阶段不要做重型 GraphRAG。更稳妥的做法是 **Graph-Enhanced RAG**：

- 向量检索负责召回语义相关案例和摘要
- 图谱负责补充结构关系、过滤范围、相似性信号和解释路径
- 原始证据查询负责给最终回答提供可追溯依据

## 4. 数据使用边界

### 4.1 数据源分组

建议把数据分成四组：

1. `输入层`
- `input.json`
- 用于确定 case 时间窗与检索入口

2. `原始证据层`
- `extracted/YYYY-MM-DD/log-parquet`
- `extracted/YYYY-MM-DD/metric-parquet`
- `extracted/YYYY-MM-DD/trace-parquet`

3. `历史标签层`
- `groundtruth.jsonl`
- 用于沉淀已知案例标签与结案描述

4. `派生产物层`
- 归一化事实
- 图谱节点与边
- serving docs
- eval artifacts

### 4.2 groundtruth 的使用原则

`groundtruth.jsonl` 可以进入内部 RCA 产品，但必须按下面规则使用：

1. `fault_type`、`fault_category`、`service`、`instance_type`、`instance` 可作为历史案例标签使用
2. `fault_description` 可作为“历史结论摘要”使用，但不能当成原始证据
3. `key_metrics` 可作为“历史标签关键词/特征词”使用，但不能替代原始指标事实
4. `key_observations` 可以作为高价值线索来源，但进入 serving 链路前应转为结构化 observation facts
5. 若做离线评测，必须明确哪些评测使用了历史标签，哪些评测只允许使用原始证据

### 4.3 许可边界

该数据集说明中包含 `CC BY-NC 4.0` 约束。对于当前目标：

- 内部研究、内部产品原型、非商业验证：可行
- 面向外部收费产品：需要重新确认许可边界

因此这份设计可以支撑内部 RCA 产品化，但不应默认等价于可商业分发的数据资产。

## 5. 总体架构

建议采用六层架构，而不是把“数据清洗、构图、Serving、评测”混成一层。

### L0 原始证据层

输入：

- `input.json`
- `groundtruth.jsonl`
- 原始 parquet

职责：

- 保存原始事实
- 不做推断
- 作为最终溯源源头

### L1 归一化事实层

建议产物：

- `source_inventory.json`
- `canonical_case_facts.jsonl`
- `observation_records.jsonl`
- `service_call_facts.jsonl`
- `entity_inventory.jsonl`

职责：

- 用统一 schema 表达 case、observation、service-call、实体清单
- 保留来源与时间边界
- 为图谱和 RAG 共享输入

### L1.5 历史标签层

建议产物：

- `case_labels.jsonl`
- `fault_type_catalog.jsonl`
- `resolution_patterns.jsonl`

职责：

- 表达历史结论、标签、已知模式
- 与原始证据分层存储
- 明确哪些字段属于 resolved knowledge，哪些字段属于 runtime evidence

### L2 图谱层

建议产物：

- `graph_nodes.jsonl`
- `graph_edges.jsonl`
- `graph_schema.md`

职责：

- 建立 RCA 所需的实体与关系
- 支撑关系检索、图过滤、相似案例扩展、解释路径生成

### L3 Serving 层

建议产物：

- `case_serving_doc.jsonl`
- `evidence_serving_doc.jsonl`
- `fault_pattern_serving_doc.jsonl`
- `retrieval_features.jsonl`

职责：

- 为向量库提供高质量摘要文档
- 为混合检索提供图特征、标签特征、实体特征
- 为回答组装阶段提供可引用片段

### L4 Eval 与回放层

建议产物：

- `build_split.json`
- `eval_split.json`
- `retrieval_eval.jsonl`
- `rca_eval.jsonl`

职责：

- 固定构图边界和评测边界
- 防止“全量构图再评估自己”的自证循环

## 6. 事实层设计

### 6.1 canonical_case_facts.jsonl

每条 case 只记录稳定事实，不混入太多解释性字段。建议字段：

- `case_id`
- `dataset`
- `start_time`
- `end_time`
- `duration_sec`
- `instance_type`
- `instance`
- `service`
- `source`
- `destination`
- `evidence_refs`
- `label_refs`

这里要注意两点：

1. `source/destination` 仅表示该 case 下的网络故障注入方向或观测方向，不等价于全局稳定服务依赖
2. `fault_type`、`fault_description` 等结论类字段不要直接放进 facts 主体，而是通过 `label_refs` 指向 L1.5

示例：

```json
{
  "case_id": "case-001",
  "dataset": "aiopschallenge2025",
  "start_time": "2025-06-06T09:31:00Z",
  "end_time": "2025-06-06T09:36:00Z",
  "duration_sec": 300,
  "instance_type": "service",
  "instance": "checkoutservice",
  "service": "checkoutservice",
  "source": "frontend",
  "destination": "checkoutservice",
  "evidence_refs": ["ev-log-01", "ev-metric-07", "ev-trace-09"],
  "label_refs": ["label-case-001"]
}
```

### 6.2 observation_records.jsonl

这是第一版最关键的模型，不要只做“metric/log/trace 三选一”的扁平抽象，否则后面会形成大面积 hub 节点。

建议每条 observation 至少包含以下维度：

- `observation_id`
- `case_id`
- `signal_type`：metric / log / trace
- `subtype`
- `name`
- `service`
- `instance_type`
- `instance`
- `pod`
- `node`
- `trace_operation`
- `direction`：increase / decrease / error / timeout / restart / exception / loss / delay
- `summary_stats`
- `keywords`
- `time_window`
- `source_ref`
- `extractor_version`
- `confidence`

这样做的原因是：

- `rrt` 不是一个完整观察，`checkoutservice rrt increase` 才是更接近 RCA 的观察单元
- `context canceled` 不是完整观察，`paymentservice + error log + context canceled` 才有区分度
- `GET /cart` 某个 operation 上的 trace 异常，和另一个 operation 的 trace 异常，不应该落到同一个粗粒度节点

示例：

```json
{
  "observation_id": "obs-0001",
  "case_id": "case-001",
  "signal_type": "metric",
  "subtype": "latency",
  "name": "rrt",
  "service": "checkoutservice",
  "instance_type": "service",
  "instance": "checkoutservice",
  "pod": null,
  "node": null,
  "trace_operation": null,
  "direction": "increase",
  "summary_stats": {
    "baseline_p95": 180,
    "incident_p95": 950,
    "delta_ratio": 5.28
  },
  "keywords": ["rrt", "latency", "slow"],
  "time_window": {
    "start": "2025-06-06T09:31:00Z",
    "end": "2025-06-06T09:36:00Z"
  },
  "source_ref": "metric://2025-06-06/checkoutservice/rrt",
  "extractor_version": "metric-anomaly-v1",
  "confidence": 0.93
}
```

### 6.3 service_call_facts.jsonl

这是为了表达“稳定调用关系”和“case 内传播路径”的差异。

建议拆成两类事实：

1. `service_call_fact`
- 来自 trace 聚合
- 表示稳定或高置信的服务调用关系
- 可用于生成 `SERVICE_CALLS_SERVICE`

2. `case_network_path_fact`
- 来自某个 case 的 `source/destination`
- 表示该故障注入路径或传播路径
- 只在 case 上下文内成立

这两者不能混用。

## 7. 图谱设计

### 7.1 节点类型

第一版建议只保留对 RCA 有直接价值的节点：

- `Case`
- `Service`
- `Instance`
- `Pod`
- `Node`
- `FaultType`
- `FaultCategory`
- `Observation`
- `TraceOperation`
- `LogTemplate`
- `MetricSignal`
- `ResolutionPattern`

说明：

- `Observation` 是 case 内可定位的异常观察单元
- `MetricSignal` / `LogTemplate` / `TraceOperation` 更像字典或模式节点
- `ResolutionPattern` 用于承接历史结论与排障建议，不直接冒充证据

### 7.2 边类型

建议第一版只保留下面这些关系：

- `CASE_AFFECTS_SERVICE`
- `CASE_AFFECTS_INSTANCE`
- `CASE_HAS_LABEL_FAULT_TYPE`
- `CASE_HAS_LABEL_FAULT_CATEGORY`
- `CASE_HAS_OBSERVATION`
- `OBSERVATION_ON_SERVICE`
- `OBSERVATION_ON_INSTANCE`
- `OBSERVATION_MATCHES_METRIC_SIGNAL`
- `OBSERVATION_MATCHES_LOG_TEMPLATE`
- `OBSERVATION_MATCHES_TRACE_OPERATION`
- `CASE_HAS_RESOLUTION_PATTERN`
- `SERVICE_CALLS_SERVICE`
- `CASE_NETWORK_PATH_FROM`
- `CASE_NETWORK_PATH_TO`
- `CASE_SIMILAR_TO_CASE`
- `FAULT_TYPE_ASSOCIATED_WITH_SIGNAL`

### 7.3 关键边的语义约束

#### `SERVICE_CALLS_SERVICE`

必须来自 trace 聚合事实，而不是 `groundtruth.source/destination`。

可接受来源：

- trace span 聚合
- service-level call graph 提取
- 多次 case 观测中稳定出现的上游下游关系

不可接受来源：

- 单次网络故障中的 `source/destination` 字段直接上升为全局拓扑边

#### `CASE_NETWORK_PATH_FROM` / `CASE_NETWORK_PATH_TO`

这两条边只描述 case 局部上下文中的网络路径事实。

它的语义是：

- 在这个 case 中，故障注入或异常传播主要发生在 `source -> destination`
- 它不等价于系统稳定依赖，不可替代 `SERVICE_CALLS_SERVICE`

#### `CASE_SIMILAR_TO_CASE`

不要直接靠标签相同就连边，建议由多种信号组成：

- 故障类型相同或相近
- 影响服务相同或同域
- observation 特征相似
- trace 路径相似
- 指标模式相似

并把相似度来源写入 edge metadata。

## 8. 历史标签层设计

### 8.1 为什么要单独做 L1.5

如果把 `fault_type`、`fault_description`、`key_metrics` 直接塞进 `canonical_case_facts` 或 `Observation`，后面会出现两个问题：

1. 回答时分不清“这是当前查到的证据”，还是“历史上已经知道的结论”
2. 评审时分不清“模型会推理”，还是“系统把答案召回出来了”

因此建议单独做 `case_labels.jsonl`。

### 8.2 case_labels.jsonl 建议字段

- `label_id`
- `case_id`
- `fault_type`
- `fault_category`
- `historical_resolution_summary`
- `key_metrics_terms`
- `source`: groundtruth
- `confidence`: 1.0

这里的 `historical_resolution_summary` 可以来自 `fault_description`，但对外展示时应明确标记为：

- 历史结案描述
- 已知案例标签
- 非实时原始证据

### 8.3 产品使用方式

在内部 RCA 产品里，历史标签层可以用于：

- 相似案例推荐
- 历史结论摘要
- RCA 候选排序
- 根因候选解释

但最终回答应保持结构：

1. 当前证据
2. 图谱关联
3. 相似历史案例
4. 历史结论与处置建议

不要把第 4 步倒灌成第 1 步。

## 9. 溯源设计

“可追溯”不能只做到 `source_file` 级别，至少要能回答以下问题：

1. 这条 observation 来自哪个 case
2. 来自哪类原始数据
3. 对应哪个时间窗口
4. 抽取规则版本是什么
5. 原始定位条件是什么
6. 是原始事实还是派生关系
7. 如果是派生关系，是由哪些输入聚合出来的

因此每条 observation / edge / serving doc 至少要包含：

- `source_type`
- `source_file`
- `case_id`
- `time_window`
- `locator`
- `extractor_version`
- `derivation_type`: raw / normalized / derived / label
- `upstream_refs`

例如一条 `FAULT_TYPE_ASSOCIATED_WITH_SIGNAL` 边，应记录：

- 来自哪些 case
- 统计口径是什么
- 最低支持样本数是多少
- 是否包含历史标签信息

否则这条边就只是“看起来合理”，而不是“可以审计”。

## 10. Graph-Enhanced RAG 方案

### 10.1 不要直接把 parquet 入向量库

向量库建议只放两类内容：

1. `case_serving_doc`
- 描述某个 case 的背景、主要异常、影响对象、代表性 observation、历史标签摘要

2. `evidence_serving_doc`
- 描述某个 observation 或某组证据的结构化摘要

原始 parquet 继续保留在冷层，按需查询，不直接做主检索语料。

### 10.2 推荐检索链路

建议检索链路如下：

1. 用户输入故障现象或问题
2. 做实体识别与意图识别
3. 从图谱中过滤候选服务、实体、fault type、相关 observation 模式
4. 用这些结构特征扩展向量检索查询
5. 召回 case docs + evidence docs
6. 按图相似度、时间重合度、服务重合度、标签相似度做 rerank
7. 最终组装答案时明确区分：
   - 当前证据
   - 图谱关系
   - 相似历史案例
   - 历史结论/建议

### 10.3 回答结构建议

内部 RCA 产品的回答建议固定为四段：

1. `当前异常证据`
2. `影响范围与关系链`
3. `相似历史案例`
4. `可能根因与建议动作`

这样更符合运维排障流程，也能避免模型直接给“拍脑袋答案”。

## 11. 评测设计

虽然目标不是比赛，但评测边界仍然要有，否则后面优化无法判断真假增益。

### 11.1 必须冻结构图边界

至少拆成两类 case：

- `build split`
- `eval split`

规则是：

- build split 用于构图、统计 fault-pattern、构造 case similarity
- eval split 只用于查询与验证，不参与构图统计

### 11.2 必须区分两类评测

1. `证据驱动评测`
- 只允许使用原始证据层与归一化事实层
- 用于验证系统是否真的能从证据中找到线索

2. `产品化评测`
- 允许使用历史标签层
- 用于验证“历史案例库 + 图谱 + RAG”对内部 RCA 是否有帮助

这两类评测都要做，否则你无法知道系统的真实能力边界。

## 12. 推荐实施顺序

### Phase 1：事实层与标签层拆分

交付：

- `source_inventory.json`
- `canonical_case_facts.jsonl`
- `observation_records.jsonl`
- `service_call_facts.jsonl`
- `case_labels.jsonl`

目标：

- 先把事实、标签、证据分开
- 为后续构图建立统一输入

### Phase 2：轻量 RCA 图谱

交付：

- `graph_schema.md`
- `graph_nodes.jsonl`
- `graph_edges.jsonl`

目标：

- 只做服务内部 RCA 必需的节点与边
- 不追求花哨图谱规模

### Phase 3：Serving 文档与混合检索

交付：

- `case_serving_doc.jsonl`
- `evidence_serving_doc.jsonl`
- `retrieval_features.jsonl`

目标：

- 让图谱真正进入线上检索与问答链路

### Phase 4：回放与评测

交付：

- `build_split.json`
- `eval_split.json`
- `retrieval_eval.jsonl`
- `rca_eval.jsonl`

目标：

- 评估图谱是否真的提升内部 RCA
- 判断是事实层不足、构图不足，还是 rerank 不足

1. 不要把这批 parquet 直接放进当前 Milvus 文档库
2. 先在 `aiopschallenge2025` 边上做独立 ETL 目录，生成统一事实层产物
3. 把图谱增强后的 serving docs 再接到现有 RAG 查询入口
4. 在回答模板里明确标注“历史案例结论”与“当前证据”
5. 内部 RCA 可以使用历史标签，但 UI 和 prompt 都必须防止把它伪装成实时观测

最关键的一点是：

> 先把“案例事实层”和“历史结论层”拆开，再谈图谱和 RAG。否则后面无论怎么检索，输出都会混淆证据与结论。

---

## 14. 最终结论

对于“内部 RCA 产品化、非商业使用”这个目标，这批数据 **值得** 加入你的知识库体系，而且 **适合** 做知识图谱增强的 RAG。

但正确做法不是：

- 原始 parquet 直接入向量库
- `groundtruth` 直接当实时证据
- `source/destination` 直接当稳定调用拓扑

正确做法是：

1. 建立归一化事实层
2. 单独建立历史标签层
3. 构建面向 RCA 的轻量案例图谱
4. 用图谱增强检索与解释
5. 用冻结 split 做产品化评测

一句话总结：

> 这套设计最适合演进成“内部 RCA 故障案例图谱 + Graph-Enhanced RAG”，而不是一个泛化的图数据库项目，也不是一个只会背历史答案的检索系统。
