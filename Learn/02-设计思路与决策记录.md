# 设计思路与决策记录

这份文档不是隐藏式推理记录，而是面向工程学习的“决策日志”。  
你以后做类似项目时，可以按这个框架自己判断。

## 一、为什么 CI 先只做这些

`ci.yml` 目前做了三件事：

1. 跑 `go test ./...`
2. 构建后端镜像
3. 构建前端镜像

这样设计的原因是：

- Go 测试能挡住最明显的功能回归。
- 镜像构建能挡住 Dockerfile、静态资源、复制路径的问题。
- 这三步足够快，适合作为主分支和 PR 的第一层门禁。

我没有在第一版里把 CI 做得很重，比如：

- 不强加复杂安全扫描
- 不接入繁琐的多环境流水线
- 不引入过多第三方 GitHub Action

因为对当前项目来说，最需要的是“先稳定把版本发出去”。

## 二、为什么 CD 选择 ACR + ECS

这是成本、复杂度和可控性的平衡。

### 1. 选择 ACR

因为部署目标在阿里云，镜像放 ACR 有三个直接好处：

- 镜像拉取链路更短
- 权限边界更清晰
- 后续如果迁移到 ACK，也不用重做镜像仓库部分

### 2. 选择 ECS

因为 ECS 是最容易掌控全链路的形态。

- 你能直接 SSH 上去看日志
- 你能直接看容器、卷、端口、健康检查
- 出问题时排障成本最低

对于学习和第一版生产都很合适。

## 三、为什么前端也做成容器

前端虽然是静态文件，但我仍然把它做成了 Nginx 容器，原因有三个：

1. 这样 GitHub Actions 可以把前端和后端统一地版本化。
2. Nginx 可以直接代理 `/api` 和流式接口，部署简单。
3. 同域访问可以绕开大多数 CORS 复杂度。

如果以后你想把前端迁到 OSS + CDN，也可以。  
这次我保留了 `config.js` 运行时配置能力，就是为了给后续拆分留出口。

## 四、为什么一定要加 `.dockerignore`

这是很多同学第一次做 CI/CD 时最容易忽略的点。

如果没有 `.dockerignore`，常见后果是：

- 把本地 `.env` 打进镜像
- 把编译产物和缓存打进镜像
- 把本地 Milvus 卷数据打进镜像上下文
- 构建非常慢，镜像非常大

所以 `.dockerignore` 不是“优化项”，而是发布安全项。

## 五、为什么单独准备 `config.prod.yaml`

因为开发默认配置和生产配置不是一回事。

开发阶段你更关心：

- 能不能跑起来
- 默认值够不够方便

生产阶段你更关心：

- 参数是否可审计
- 是否能通过环境变量注入
- 是否能和部署脚本解耦

把生产配置独立出来后，有两个好处：

1. 你不会为了上线去反复改仓库默认配置。
2. 你能清楚地区分“应用代码”和“环境差异”。

## 六、你接下来可以怎么继续升级

当这套方案稳定后，你可以按顺序升级：

1. 把 ECS 上的单机 Docker Compose 升级为多环境 Compose。
2. 把上传文件从本地卷迁移到 OSS 或 NAS。
3. 把运行时状态进一步外置，降低单机依赖。
4. 再考虑迁移到 ACK 或 SAE。

这个顺序很重要。  
不要在状态问题没梳理清楚之前，就急着上更复杂的平台。

## 七、GitHub Secrets 建议清单

你需要在 GitHub 仓库里至少配置这些 Secrets：

- `ACR_REGISTRY`
- `ACR_NAMESPACE`
- `ACR_USERNAME`
- `ACR_PASSWORD`
- `ECS_HOST`
- `ECS_USER`
- `ECS_SSH_PORT`
- `ECS_SSH_PRIVATE_KEY`
- `ECS_APP_DIR`
- `PROD_ENV_FILE`

其中 `PROD_ENV_FILE` 建议直接保存成完整的生产环境变量文件内容。  
这样部署时最省事，也最容易做版本说明和回滚。
