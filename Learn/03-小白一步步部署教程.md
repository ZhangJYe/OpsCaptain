# 小白一步步部署教程

这篇是给第一次做部署的人看的。  
你不需要先懂 GitHub Actions、Docker、ACR、ECS 的全部原理，只要先按步骤做。

## 一、先知道我们到底在做什么

你可以把这次上线理解成一条流水线：

1. 代码放到 GitHub。
2. GitHub 自动帮你构建程序。
3. GitHub 把程序镜像推到阿里云镜像仓库 ACR。
4. 阿里云 ECS 服务器把新镜像拉下来。
5. 服务器把程序启动起来。

你现在不用怕这些名词，先记住它们的角色：

- GitHub：放代码。
- GitHub Actions：自动执行命令。
- Docker 镜像：程序的打包结果。
- 阿里云 ACR：存放 Docker 镜像。
- 阿里云 ECS：真正运行你程序的服务器。

## 二、你要准备什么

开始前，你需要有：

1. 一个 GitHub 账号。
2. 一个阿里云账号。
3. 这个项目已经在你的电脑里。
4. 这个项目已经推到一个 GitHub 仓库。

如果第 4 步还没做，你先完成它。

## 三、第一步：把项目推到 GitHub

如果你的项目还没有在 GitHub 上，先做这个。

### 1. 在 GitHub 新建仓库

去 GitHub 网站：

- 点击右上角 `New repository`
- 仓库名随便，比如 `OpsCaptain`
- 创建后，GitHub 会给你一个仓库地址

例如：

```bash
https://github.com/你的用户名/OpsCaptain.git
```

### 2. 在本地把代码推上去

在项目目录执行：

```powershell
git remote add origin 你的仓库地址
git branch -M main
git add .
git commit -m "init deployment setup"
git push -u origin main
```

如果你之前已经连过 GitHub，就不用重复 `git remote add origin`。

## 四、第二步：在阿里云创建镜像仓库 ACR

这一步的目的只有一个：给 Docker 镜像找个“存放的地方”。

### 1. 打开阿里云容器镜像服务 ACR

登录阿里云控制台后：

1. 搜索 `容器镜像服务 ACR`
2. 进入实例列表
3. 创建个人版或企业版实例

如果你只是先学会流程，个人版通常够用。

### 2. 创建命名空间

进入 ACR 后，创建一个命名空间，例如：

```text
opscaptain
```

### 3. 记下这几个信息

后面 GitHub 要用：

- ACR 登录地址，例如 `registry.cn-hangzhou.aliyuncs.com`
- 命名空间，例如 `opscaptain`
- ACR 用户名
- ACR 密码或访问凭证

## 五、第三步：创建 ECS 服务器

这台机器就是你最后运行项目的地方。

### 1. 在阿里云创建 ECS

推荐一开始用 Linux 服务器。

建议：

- 系统：Ubuntu 22.04 或 Alibaba Cloud Linux
- 配置：先用 2 核 4G 试运行
- 带宽：按你预算来

### 2. 安全组开放端口

你至少要放行：

- `22`：SSH 登录服务器
- `80`：网站访问

如果以后要 HTTPS，还要放行：

- `443`

### 3. 记下这些信息

你后面 GitHub 要用：

- ECS 公网 IP
- ECS 登录用户名，例如 `root` 或 `ubuntu`
- ECS SSH 端口，通常是 `22`
- 你的 SSH 私钥

## 六、第四步：先在 ECS 上安装 Docker

登录服务器后，安装 Docker 和 Docker Compose。

如果你的系统是 Ubuntu，可以先试：

```bash
sudo apt update
sudo apt install -y docker.io docker-compose-plugin
sudo systemctl enable docker
sudo systemctl start docker
```

然后检查：

```bash
docker --version
docker compose version
```

只要这两个命令能输出版本号，就说明安装成功。

## 七、第五步：理解 GitHub Actions 要怎么帮你

我们已经在仓库里放了两个工作流文件：

- [ci.yml](D:/Agent/OnCallAI/.github/workflows/ci.yml)
- [cd.yml](D:/Agent/OnCallAI/.github/workflows/cd.yml)

你可以这样理解：

- `ci.yml`：检查代码有没有坏。
- `cd.yml`：真的去发布上线。

所以不要把它们想得太神秘，它们本质上就是“GitHub 帮你自动执行命令”。

## 八、第六步：去 GitHub 配置 Secrets

这是最关键的一步。  
如果这里没配好，工作流会失败。

打开你的 GitHub 仓库：

1. 进入 `Settings`
2. 点击 `Secrets and variables`
3. 点击 `Actions`
4. 一个一个新增下面这些 Secret

### 你需要添加的 Secrets

#### ACR 相关

- `ACR_REGISTRY`
- `ACR_NAMESPACE`
- `ACR_USERNAME`
- `ACR_PASSWORD`

#### ECS 相关

- `ECS_HOST`
- `ECS_USER`
- `ECS_SSH_PORT`
- `ECS_SSH_PRIVATE_KEY`
- `ECS_APP_DIR`

#### 生产环境变量

- `PROD_ENV_FILE`

## 九、第七步：`PROD_ENV_FILE` 到底填什么

这是很多新手最容易卡住的点。

你可以把 `PROD_ENV_FILE` 理解成：  
“线上服务器启动程序时需要的那一整份 `.env` 内容”。

仓库里已经给了示例：

- [env.production.example](D:/Agent/OnCallAI/deploy/env.production.example)

你要做的是：

1. 打开这个示例文件
2. 把里面的占位值改成你自己的真实值
3. 把完整内容复制进去，保存为 GitHub Secret `PROD_ENV_FILE`

例如它里面会有这些内容：

```env
GLM_API_KEY=你的模型密钥
DS_THINK_BASE_URL=https://open.bigmodel.cn/api/paas/v4
DS_THINK_MODEL=GLM-4.5-AIR
SILICONFLOW_API_KEY=你的向量模型密钥
MILVUS_ADDRESS=你的Milvus地址:19530
MYSQL_DSN=你的MySQL连接串
AUTH_ENABLED=true
AUTH_JWT_SECRET=一个足够长的密钥
CORS_ALLOWED_ORIGIN=https://你的域名
```

注意：这里不是写文件路径，而是把“文件内容本身”存进 GitHub Secret。

## 十、第八步：第一次手动触发发布

前面都准备好后，你可以第一次手动触发发布。

### 1. 先触发 CI

把代码 push 到 `main` 后：

- GitHub 会自动运行 `ci.yml`
- 它会测试后端并检查镜像能不能构建

如果 CI 失败，就先不要继续发版。

### 2. 再触发 CD

进入 GitHub 仓库：

1. 点击 `Actions`
2. 找到 `cd`
3. 点击 `Run workflow`

然后 GitHub 会自动做这些事：

1. 构建后端镜像
2. 构建前端镜像
3. 推到阿里云 ACR
4. SSH 登录 ECS
5. 上传部署文件
6. 执行部署脚本

## 十一、第九步：怎么判断是不是部署成功了

部署成功后，你可以去 ECS 上检查：

```bash
cd 你的部署目录
docker compose --env-file .env.production -f docker-compose.prod.yml ps
```

还可以看日志：

```bash
docker compose --env-file .env.production -f docker-compose.prod.yml logs -f
```

如果一切正常，你访问：

```text
http://你的ECS公网IP
```

应该能看到前端页面。

## 十二、第十步：这几个文件分别是干什么的

你只要先记住用途，不用一下全背下来。

### 1. [ci.yml](D:/Agent/OnCallAI/.github/workflows/ci.yml)

作用：代码检查。

### 2. [cd.yml](D:/Agent/OnCallAI/.github/workflows/cd.yml)

作用：自动发布。

### 3. [docker-compose.prod.yml](D:/Agent/OnCallAI/deploy/docker-compose.prod.yml)

作用：告诉服务器要启动哪些容器。

### 4. [config.prod.yaml](D:/Agent/OnCallAI/deploy/config.prod.yaml)

作用：告诉后端服务用哪些生产配置。

### 5. [remote-deploy.sh](D:/Agent/OnCallAI/deploy/remote-deploy.sh)

作用：服务器收到新版本后，真正执行拉镜像和重启容器。

### 6. [env.production.example](D:/Agent/OnCallAI/deploy/env.production.example)

作用：给你一个生产环境变量模板。

## 十三、如果你现在就想开始，最小执行顺序

你现在先只做这 6 件事：

1. 把项目推到 GitHub。
2. 在阿里云开一个 ACR。
3. 在阿里云开一台 ECS。
4. 在 ECS 上安装 Docker。
5. 在 GitHub 配好 Secrets。
6. 手动运行 `cd` 工作流。

不要一开始就想着：

- HTTPS
- 域名
- 负载均衡
- Kubernetes
- 多副本高可用

那些是第二阶段，不是你现在第一步最需要的。

## 十四、我建议你的学习方式

不要试图一次看懂所有文件。  
正确方法是按顺序学：

1. 先学“发布链路长什么样”
2. 再学“GitHub Secrets 是什么”
3. 再学“ECS 上怎么检查容器”
4. 最后学“为什么这样设计”
