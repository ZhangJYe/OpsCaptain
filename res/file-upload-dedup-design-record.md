# 文件上传去重与更新设计记录

日期：2026-04-24

## 背景

原实现每次上传都会生成新的随机文件名，再把这个随机路径作为索引来源。

这会带来两个直接问题：

1. 同一个文件重复上传时，本地会堆出多份副本。
2. Milvus 删除旧索引时按随机 `_source` 匹配，命不中历史版本，知识库里会残留重复内容。

## 这次改动的目标

这次不追求做完整 ingestion 平台，只先把最核心的用户问题修掉：

1. 同名同内容文件重复上传时，不重复落盘、不重复建索引。
2. 同名但内容变化时，当成一次更新处理。
3. 上传产生的本地副本只保留当前版本。
4. 词法检索索引不要残留旧版本内容。

## 设计思路

### 1. 文件存储和文档身份分开

上传后的物理文件仍然允许用随机文件名落盘，避免文件名冲突。

但逻辑上的“这是谁”不再靠随机落盘名判断，而是引入稳定的 `source_key`：

- `source_key = upload://{sanitize(filename)}`

这意味着当前版本的最小实现里：

- 同一个清洗后文件名，视为同一个逻辑文档
- 同名新内容，视为更新
- 同名同内容，视为重复上传

### 2. 用 sidecar 记录上传元数据

每个上传文件旁边生成一个 `.metadata.json`，记录：

- `source_kind`
- `source_key`
- `_source`
- `original_filename`
- `stored_filename`
- `content_hash`
- `uploaded_at`
- `version`
- `file_size`
- `mime_type`

其中 `_source` 直接写成稳定的 `source_key`，这样 loader 读文件时会把文档源标识覆盖成稳定值，索引层就能按同一个逻辑文档做替换。

### 3. 上传判定规则

上传新文件时：

1. 先保存到隔离目录。
2. 计算内容 `sha256`。
3. 读取同 `source_key` 的历史上传记录。
4. 如果发现 `content_hash` 相同且本地文件仍存在：
   - 判定为重复上传
   - 删除隔离目录中的新文件
   - 直接返回已有记录
5. 如果内容不同：
   - 生成新版本号
   - 保存新文件与 sidecar
   - 触发索引更新
   - 索引成功后删除旧上传副本

## BM25 处理

原实现对共享 BM25 索引是“只追加”，即使 Milvus 中旧向量删掉了，BM25 里仍可能保留旧块。

这次一起做了两件事：

1. BM25 同一 `doc id` 再次加入时改为替换，不再盲目追加。
2. 每次索引后按当前 `file_dir` 全量重建共享 BM25，避免旧副本残留在内存索引里。

## 当前边界

这版是面向当前接口条件的最小可行方案，仍有边界：

1. 现在没有“项目内逻辑路径”这个输入，所以只能按清洗后的文件名识别逻辑文档。
2. 如果业务上存在“不同目录下同名文件都应该并存”的需求，后续需要前端或 API 补充更稳定的逻辑路径字段。
3. Milvus 仍然是“先删旧索引，再写新索引”的替换模式；这次主要解决的是重复命中不了旧版本的问题，还没有做到严格事务化切换。

## 后续建议

如果后面要继续演进，建议顺序如下：

1. 给上传接口补 `logical_path` 或 `document_key`
2. 把 `source_key` 从“文件名级”升级到“业务路径级”
3. 再补完整版本管理和更稳的索引切换

这样能先用最小成本解决重复文件问题，再逐步演进到更完整的文档生命周期管理。
