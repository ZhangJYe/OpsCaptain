# MySQL InnoDB 死锁与锁等待排查

## 适用场景

- 接口 RT 飙高
- 数据库事务超时
- 批量更新卡死
- 业务日志里频繁出现 deadlock / lock wait timeout

## 先区分两个概念

### 死锁

两个或多个事务形成循环等待，数据库必须主动打断其中一个事务。

### 锁等待

不是循环依赖，只是某个事务持有锁太久，其他事务在排队等。

## InnoDB 死锁怎么查

官方文档明确给出两个关键入口：

### 1. 看最近一次死锁

```sql
SHOW ENGINE INNODB STATUS;
```

这可以看到最近一次 InnoDB 用户事务死锁的详细信息。

### 2. 如果死锁频繁，打开全量死锁日志

```sql
SET GLOBAL innodb_print_all_deadlocks = ON;
```

之后可以到 `mysqld` error log 里看每一次死锁。

## 死锁排查重点

你要回答的不只是“谁跟谁锁住了”，而是：

1. 哪两类 SQL 在冲突
2. 它们访问资源的顺序是否不一致
3. 是否有大事务或长事务
4. 是否因为缺索引导致锁范围过大
5. 是否把多个不必要操作包进了同一事务

## 常见根因

- 两条事务以不同顺序更新同一组资源
- 批量更新范围过大
- 二级索引不命中，导致扫描和加锁范围扩大
- 应用把网络调用、重逻辑放在事务里，导致持锁时间过长

## 处理策略

### 1. 应用层必须能重试

死锁不是“数据库坏了”，而是并发控制中的正常失败模式之一。

所以业务方要有：

- 明确的事务重试策略
- 幂等设计
- 重试退避

### 2. 缩短事务

不要在事务里：

- 调远程服务
- 做复杂计算
- 等待外部 IO

### 3. 固定访问顺序

如果多个事务都需要访问同一组资源，尽量固定顺序，避免事务 A 先锁订单再锁账户，而事务 B 先锁账户再锁订单。

### 4. 补索引

很多所谓“并发问题”，本质是 SQL 没走对索引，导致锁范围比预期大得多。

## 面试怎么讲

如果面试官问“数据库死锁你怎么处理”，你可以回答：

1. 先用 `SHOW ENGINE INNODB STATUS` 看最近一次死锁
2. 如果频繁出现，打开 `innodb_print_all_deadlocks`
3. 再回到应用侧检查事务顺序、事务长度和索引命中
4. 最后确认业务是否支持幂等重试
