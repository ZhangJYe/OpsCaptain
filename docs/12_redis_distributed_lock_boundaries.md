# Redis 分布式锁边界与正确使用场景

## 适用场景

- 面试被问“Redis 分布式锁怎么做”
- 你想把 Redis 锁相关知识放进 RAG
- 需要区分“能用”和“真正可靠”之间的边界

## 先记住 3 个结论

1. 单实例 Redis 锁不是万能的
2. 主从切换方案不天然满足互斥安全性
3. 分布式锁要先讲清楚安全目标，再谈实现

## 官方给出的三个目标

Redis 官方文档把分布式锁最小保障总结为三点：

- Safety: Mutual exclusion，同一时刻只有一个客户端持有锁
- Liveness A: Deadlock free，即使持锁客户端崩溃，锁最终也能再次被获取
- Liveness B: Fault tolerance，只要多数节点可用，客户端仍可加锁和解锁

## 为什么“主从复制 + 故障切换”不够

官方文档直接指出一个关键问题：

- Redis 主从复制是异步的
- 主节点刚写入锁，还没同步到从节点就挂了
- 从节点提升为新主后，另一个客户端又拿到了同一把锁

这就破坏了互斥性。

## 单实例正确姿势

官方给出的基础写法是：

```redis
SET resource_name my_random_value NX PX 30000
```

关键点：

- `NX`：只有 key 不存在时才加锁
- `PX`：带过期时间，避免死锁
- `my_random_value`：必须是唯一值，不能只写固定字符串

解锁时也不能直接 `DEL`，而应该只删除“值仍然等于自己持有随机值”的那把锁。

## Redlock 的思路

官方给出的 Redlock 思路是：

- 使用多个相互独立的 Redis master
- 客户端并行尝试获取多数节点的锁
- 只有在多数成功且总耗时低于有效期时，才算加锁成功

## 对工程实践的真实建议

### 可以用 Redis 锁的场景

- 定时任务去重
- 弱一致性的并发保护
- 可以容忍极小概率重复执行的任务

### 不要只靠 Redis 锁的场景

- 金额扣减
- 强一致订单状态推进
- 不能重复执行的关键事务

这些场景往往还需要：

- 数据库事务约束
- 幂等键
- 唯一索引
- 状态机校验

## 面试怎么讲

如果面试官问“Redis 分布式锁怎么实现”，比较稳的回答是：

> 最基础的单实例做法是 `SET key value NX PX ttl`，并用唯一随机值安全释放。  
> 但如果要求严格互斥，主从切换方案因为异步复制会有安全漏洞。  
> 所以要根据业务对一致性的要求，决定是用单实例锁、Redlock，还是退回数据库约束和幂等设计。
